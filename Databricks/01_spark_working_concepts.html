Prologue

Why Does Spark Exist?

Before learning Spark, understand the pain it was built to solve. You can't appreciate the solution without understanding the problem.

The World Before Spark — Hadoop MapReduce

When companies had terabytes of data to process, they used Hadoop MapReduce. It worked, but it had two painful problems that made engineers' lives hard.

❌ Pain Point 1 — Disk I/O Hell
  • MapReduce writes every intermediate result to disk (the hard drive)
  • After each Map step → write to disk. After Shuffle → write to disk. After Reduce → write to disk.
  • Disk reads/writes are 100–1000× slower than memory (RAM)
  • A job that could take minutes runs in hours
  • Real example: Machine learning needs 100 iterations over the same data. MapReduce reads from disk 100 times.
VS
✅ How Spark Fixes It
  • Spark stores intermediate results in RAM (worker node memory)
  • Data flows through operations without touching disk (unless data is too large)
  • Same 100-iteration ML job now reads from RAM 100 times — 100× faster
  • Only final output and shuffle data touch disk
❌ Pain Point 2 — Rigid One-Shuffle-Per-Job
  • MapReduce has exactly one map + one reduce phase per job
  • You can only group by one key per job
  • Example: Find total amount per country for completed orders only. This needs: filter → join (by customer_id) → aggregate (by country)
  • MapReduce needs two separate jobs: Job 1 (join by customer_id) → write to disk → Job 2 (aggregate by country)
  • Every new key = a new job = more disk I/O
VS
✅ How Spark Fixes It
  • Spark uses a DAG (Directed Acyclic Graph) model
  • One application can have multiple stages with different grouping keys
  • Same example in Spark: one application, 2 stages — no disk writes between stages
  • Spark handles the entire pipeline: filter → join → aggregate — all in one run
🏠 Real-World Analogy
Think of MapReduce like a restaurant where the waiter has to write the order on paper, hand it to the chef, wait for it, write the partial result on paper again, walk to another chef, hand it over again... for every single step. Spark is a restaurant where the chef just shouts across the kitchen — the order flows directly from station to station without paper shuffling.
"Spark is 100× faster than Hadoop MapReduce for iterative computations because it keeps data in memory, and 10× faster for disk-based operations." — Apache Spark documentation

Chapter One

Spark Application,
Shell & SparkContext

What exactly is a "Spark Application"? What types are there? And what is the SparkContext that sits at its heart?

Three Things That Look Similar but Mean Different Things

Beginners often confuse these three terms. Let's define each clearly before going further.

Spark Application
Any program that uses the Spark framework to process data. It is your code — a PySpark script, a Databricks notebook, or an Airflow task. One Spark Application = one SparkContext. It starts when you create a SparkSession and ends when you call spark.stop().
Spark Shell
An interactive REPL (Read-Eval-Print-Loop) that lets you type Spark code line-by-line and see results immediately. Launch it with pyspark (Python) or spark-shell (Scala). The shell automatically creates a SparkContext called sc and a SparkSession called spark for you.
SparkContext
The original entry point to Spark (Spark 1.x). It represents the connection between your program and the Spark cluster. Today, you create a SparkSession which internally creates and manages the SparkContext for you. There can only be ONE SparkContext per JVM.
Two Types of Spark Applications
1a. Spark Notebook Applications

These are interactive, cell-by-cell Spark applications. You write a few lines, run them, see the result, then write more. Perfect for data exploration, debugging, and experimentation.

  • Examples: Databricks Notebooks, Jupyter Notebooks with PySpark kernel, Zeppelin Notebooks
  • How it works: Each notebook cell runs as part of the same Spark application (same SparkSession and SparkContext) — until you stop the cluster
  • Who uses it: Data engineers exploring data, data scientists building models, analysts writing ad-hoc queries
  • Key trait: Results show up in the notebook immediately — display(df) in Databricks shows a rich interactive table
🟣 Databricks Specific
In Databricks, when you attach a notebook to a cluster, Databricks automatically creates the SparkSession for you. The variable spark is pre-available in every cell. You never write SparkSession.builder.getOrCreate() — Databricks handles it. The cluster = your Spark application's lifetime.
  databricks_notebook.py — Notebook App
# In Databricks, 'spark' is already ready — no setup needed
# Cell 1: Read some data
df = spark.read.csv("/mnt/bronze/transactions.csv", header=True)
display(df)   # Databricks shows interactive table

# Cell 2: Explore (same SparkSession, same application)
df.printSchema()
print(f"Total rows: {df.count()}")

# Cell 3: Transform (still same application)
df_clean = df.filter(df.status == "COMPLETED")
display(df_clean)
1b. Spark Batch Applications

These are standalone scripts submitted to a Spark cluster to run without any human interaction. They run start-to-finish, produce output (files, database writes), then terminate.

  • Examples: Daily ETL jobs, nightly fraud detection runs, end-of-month reconciliation jobs
  • How it works: You submit the script using spark-submit — the cluster receives it, runs it, and exits
  • Who uses it: Scheduled production pipelines, orchestrated by Azure Data Factory (ADF) or Databricks Workflows
  • Key trait: Fully automated — no human watching. Results go to files, databases, or Delta Lake
  batch_job.py — Batch Application
from pyspark.sql import SparkSession
from pyspark.sql import functions as F

# Batch apps create their own SparkSession
spark = SparkSession.builder \
    .appName("DailyFraudDetection_2024-01-15") \
    .getOrCreate()

# All logic runs without any human interaction
df = spark.read.parquet("abfss://bronze@bankadls.dfs.core.windows.net/txns/")
df_fraud = df.filter(F.col("amount") > 50000)
df_fraud.write.mode("append").format("delta") \
    .save("abfss://gold@bankadls.dfs.core.windows.net/fraud/")

# Always stop the session in batch jobs to free cluster resources
spark.stop()

# Submit this script with:
# spark-submit --master yarn --deploy-mode cluster batch_job.py

SparkContext — What It Actually Does
📡

Analogy: SparkContext = Your Phone's SIM Card

A SIM card connects your phone to the network. Without it, your phone can't make calls. SparkContext is what connects your application to the Spark cluster. Without it, your code can't talk to executors. Just like you can only have one active SIM in a phone, you can only have one active SparkContext per JVM.

SIM Card → SparkContext Phone → Your Application (JVM) Mobile Network → Spark Cluster

SparkContext's main responsibilities:

  • Cluster Connection: Maintains the connection between your driver program and the Spark cluster
  • Resource Request: Asks the Cluster Manager to allocate executors for your application
  • Task Distribution: Sends tasks (code) to executors for execution
  • RDD Creation: Creates RDDs (low-level Spark data structures) from files, lists, etc.
  • Broadcast & Accumulator: Sends read-only shared data to all executors (broadcast variables) and collects distributed counters (accumulators)
📘 Historical Note — SparkSession vs SparkContext
In Spark 1.x: You created SparkContext directly. Then SQLContext for DataFrames. Then HiveContext for Hive. Three separate objects.

In Spark 2.0+: SparkSession replaced all of them. It internally creates a SparkContext but hides it. You use SparkSession for everything. If you need the SparkContext, access it via spark.sparkContext.

Rule: In modern PySpark, always create a SparkSession. Never create SparkContext directly.
  sparkcontext_access.py
from pyspark.sql import SparkSession

# ✅ Modern way — SparkSession manages SparkContext internally
spark = SparkSession.builder \
    .appName("MyApp") \
    .master("local[*]") \
    .getOrCreate()

# Access SparkContext through SparkSession if needed
sc = spark.sparkContext

print(sc.appName)          # "MyApp"
print(sc.master)           # "local[*]"
print(sc.defaultParallelism)  # Number of CPU cores available
print(spark.version)       # Spark version, e.g. "3.5.0"

# One application = One SparkContext (hard rule)
# getOrCreate() returns existing session if already running
spark2 = SparkSession.builder.getOrCreate()
print(spark2 is spark)  # True — same object!

Chapter Two

The Spark Cluster —
The Big Picture

What exactly is a cluster? How are machines laid out? Who talks to who? Let's zoom all the way out first before zooming in.

What is a Cluster?

A cluster is simply a collection of computers (called nodes) connected by a network, working together as if they were one powerful machine. These machines can be in:

  • Public Cloud: Azure VMs, AWS EC2 instances, Google Compute Engine
  • Private Data Center: Physical servers in a company's own building
  • Hybrid: Some on-premise, some in the cloud

A Spark Cluster = a cluster where the Spark software is installed on every node, making the machines capable of running Spark jobs.

🏢 Analogy — Cluster = Office Building
Think of a Spark Cluster like an office building. The building has many rooms (nodes). Each room has workers (executor threads) doing tasks. There's a boss's office (Master node) that manages the building. There's an HR department (Cluster Manager) that assigns how many desks each team gets. Your work order (Driver Program) comes in through the front door and gets distributed to the right rooms.

Master Node
The machine that hosts the Cluster Manager. It is the control center — it knows about all worker nodes, their available resources, and which application is using what. It does NOT process your data directly.
Worker Node
The machines that do the actual data processing. Each worker node runs a Spark Worker JVM which launches Executors. Worker nodes have CPU, memory, and local disk for processing data.
Driver Node
The machine running the Driver Process (your application). In client mode: this is your laptop/submission machine (outside the cluster). In cluster mode: this is one of the cluster nodes. This is NOT the Master Node.
⚠️ Common Confusion — Master Node ≠ Driver Node
Master Node = the machine running the Cluster Manager (resource allocation software).
Driver Node = the machine running your application code (SparkSession, DAG Scheduler).

In Spark Standalone mode, they CAN be the same physical machine. But logically, they are always two separate roles. In client mode, the driver runs on your laptop — which has nothing to do with the master node in the cluster. Your notes confirm this: "Driver JVM where it is running — that node is NOT the Master Node."

Chapter Three

Spark Master
& Spark Worker

Zoom in on the two most important nodes in the cluster. What exactly does each one do — down to the JVM level?

Spark Master — The Resource Gatekeeper
👔

Analogy: Spark Master = HR Department of a Company

When a new project (your Spark job) arrives, the HR department (Spark Master) looks at which employees (worker nodes) have free capacity. It assigns the right workers to the project and tells them how much of their time (CPU + memory) to dedicate. HR doesn't do the actual work — it just manages who does what.

HR Dept → Spark Master / Cluster Manager Project Brief → Your Spark Application Employees → Worker Nodes Work Hours → CPU Cores + Memory

What the Spark Master (in Standalone mode) actually does:

  • Knows all Worker nodes: Maintains a registry of every worker in the cluster, their available CPU cores and memory
  • Receives application requests: When a Driver starts, it contacts the Master: "I need executors with X cores and Y memory"
  • Allocates resources: Decides which workers will host executors for this application
  • Monitors health: Tracks if workers are alive and reassigns resources if a worker dies
  • Does NOT understand your DAG: The Master knows nothing about your Spark logic — it only deals with raw resources (CPU, RAM)
📘 Spark Standalone vs Other Cluster Managers
The "Spark Master" specifically refers to the master in Spark Standalone mode — Spark's own built-in cluster manager. When you use YARN or Kubernetes, there is no "Spark Master" — the Cluster Manager role is handled by YARN's ResourceManager or Kubernetes' API server instead. Chapter 5 covers all Cluster Managers.
Spark Worker — The Launcher, Not the Runner

This is a subtle but important distinction that many beginners miss:

⚠️ Critical Distinction — Worker's ONLY Job
The Spark Worker JVM's sole responsibility is to LAUNCH EXECUTORS on behalf of the Spark Master.

The Worker does NOT process data. It does NOT run your tasks. It is a management process that starts and monitors the Executor process on the same machine.
Inside a Worker Node — What's Actually Running
Worker Node (one physical machine)
🏗️ Spark Worker JVM Management process Talks to Master · Launches Executors
↓ spawns
⚙️ Executor JVM Core 1 → Task A Core 2 → Task B Core 3 → Task C Core 4 → Task D
RAM: 8GB executor memory · Local disk: shuffle write buffer

The Spark Worker's lifecycle for a single application:

1

Receives instruction from Master

The Spark Master tells the Worker: "Application X needs an Executor with 4 cores and 8GB memory — start one on your machine."

2

Starts a new Executor JVM process

The Worker JVM forks a new JVM process on the same machine — this is the Executor. The Worker allocates 4 cores and 8GB RAM to this new process.

3

Executor registers with the Driver

The new Executor JVM contacts the Driver directly: "I'm ready. I have 4 core slots available." From now on, the Driver and Executor communicate directly — the Worker steps back.

4

Worker monitors the Executor

The Worker keeps watching the Executor process. If it crashes, the Worker reports to the Master, which can allocate a replacement on another Worker node.

Standalone Mode Special Case — Master Can Also Be a Worker

In Spark Standalone mode, by default the Master node can also run Worker processes — meaning the machine hosting the cluster manager also does computation. This is common in small development clusters or single-machine setups.

⚠️ Production Recommendation
In production deployments (Azure Databricks, YARN on HDInsight), the Master node is dedicated — it only runs the Cluster Manager, never Executors. Mixing them on resource-heavy clusters causes the Cluster Manager to starve for CPU/memory when the node is under load.

Chapter Four

Spark Executor
& Spark Core

This is where your data actually gets processed. Understand the Executor JVM, what a "Core" really means in Spark, and how parallelism works.

Spark Executor — The Worker Bee

A Spark Executor is a JVM (Java Virtual Machine) process that runs on a Worker node and is responsible for two things only:

  1. Run tasks that the Driver sends it (the actual data processing)
  2. Store/cache data partitions in memory for reuse across stages
🏭

Analogy: Executor = A Factory Floor

An Executor is like a factory floor with multiple workstations (cores). Each workstation (core) works on one item (partition) at a time. The factory manager (Driver) sends batches of items and instructions. The factory floor (Executor) processes them in parallel. The storage warehouse (Executor memory) keeps semi-finished goods for the next stage.

Factory Floor → Executor JVM Workstation → Core / Thread Item to process → Data Partition Work instruction → Task Warehouse → Executor Memory (cache)
What Is a "Spark Core"? — The Most Misunderstood Term
⚠️ Critical Point — A Spark Core is NOT a Physical CPU Core
In Spark, a "core" is a logical execution slot (thread) inside an Executor that can run exactly one task at a time.

It is a JVM thread. The operating system then maps JVM threads to physical CPU cores — Spark doesn't control that mapping directly.
Executor Internals — Cores as Task Slots
Executor JVM (8GB RAM, 4 Cores)
Core / Thread 1 Running Task 1 on Partition #1
Core / Thread 2 Running Task 2 on Partition #2
Core / Thread 3 Running Task 3 on Partition #3
Core / Thread 4 ⏳ Waiting No task yet
📦 Executor Memory (8GB total)
├── Execution memory: ~4.8GB (for running tasks)
├── Storage memory: ~3.2GB (for caching partitions)
└── Cached partitions: Partition #1 results (if cached)
Tasks, Partitions, and Parallelism — The Holy Trinity

These three concepts are tightly connected. Understanding the relationship between them is key to understanding Spark performance.

Partition
A logical chunk of your data. When Spark reads a 1TB file, it splits it into thousands of partitions (e.g., 8,000 × 128MB). Each partition is a unit of data. Partitions live in Executor memory during processing.
Task
A unit of work = one operation on one partition. If you have 8,000 partitions in a stage, Spark creates 8,000 tasks. Each task runs on exactly one Core (thread) of one Executor.
Parallelism
How many tasks can run at the same time. Determined by: min(number of partitions, total available cores). More partitions + more cores = more parallelism = faster job.
✅ Parallelism Formula — Memorize This
Parallelism = min(number of partitions, total available cores)

Example: 12 worker nodes × 8 cores = 96 total core slots.
If your data has 8,000 partitions → 8,000 tasks created, but only 96 run at the same time. The other 7,904 wait in the Task Scheduler queue.

Parallelism is NOT determined by: data size, number of nodes, executor memory, or CPU speed.
  executor_config.py — Configure Executors
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("ProductionJob") \
    # ── Executor Resource Configuration ─────────────────────────
    .config("spark.executor.instances",  "12") \    # 12 executors (1 per worker)
    .config("spark.executor.cores",     "8") \     # 8 cores per executor = 96 total
    .config("spark.executor.memory",   "16g") \   # 16GB RAM per executor
    .config("spark.driver.memory",     "4g") \    # 4GB for the driver
    .getOrCreate()

# Check actual parallelism
sc = spark.sparkContext
print(f"Default parallelism: {sc.defaultParallelism}")  # = total cores = 96

# Check partition count of your DataFrame
df = spark.read.parquet("/mnt/adls/transactions/")
print(f"Partitions: {df.rdd.getNumPartitions()}")

# Actual parallelism for this job = min(96, partition_count)
# If partitions < 96 → some cores idle → performance lost
# Fix: repartition(96) to use all cores
df_repartitioned = df.repartition(96)
Executor Memory — How It's Divided

When you set spark.executor.memory = 8g, Spark doesn't give all 8GB to running your tasks. It divides it carefully:

Executor Memory Layout (8GB Example)
8 GB EXECUTOR JVM HEAP
Reserved Memory: ~300 MB For Spark internal objects, always off-limits
User Memory: ~2.4 GB (30% of remaining) Your Python UDFs, non-Spark data structures
Unified Memory Pool: ~5.3 GB (60% of remaining) Dynamically shared between:
Execution Memory For running tasks (sorts, joins, aggregations)
Storage Memory For cached/persisted DataFrames (df.cache())
✅ Memory Spill — What Happens When Memory Is Full
If execution memory fills up (task is processing a huge partition), Spark spills to disk — writes temporary files to the Worker's local disk. This is called a "spill" and dramatically slows down your job. Watch for "Spill (Memory)" and "Spill (Disk)" columns in the Spark UI's Stage view. If you see large spills, increase executor memory or reduce partition size.

Chapter Five

The Cluster Manager

The invisible hand that decides who gets CPU and memory. Four types of cluster managers — when to use which, and what each one does.

What Is a Cluster Manager?

A Cluster Manager is a software layer that sits between your application and the physical machines. Its prime job is to allocate resources (CPU cores, memory) across multiple applications that want to share the same cluster.

🏨

Analogy: Cluster Manager = Hotel Reception

Multiple guests (applications) want rooms (CPU + memory) in the same hotel (cluster). The hotel reception (Cluster Manager) decides: "Application A gets rooms 101–108, Application B gets rooms 201–204." It ensures no two guests share a room (no resource conflicts) and manages check-in/check-out (allocation and deallocation).

Hotel → Spark Cluster Guest → Spark Application Room → CPU Core + Memory Reception → Cluster Manager
⚠️ What the Cluster Manager Does NOT Do
The Cluster Manager does not understand your Spark DAG. It doesn't know about your transformations, stages, or tasks. It only speaks in terms of: "This application wants N CPU cores and M GB of memory." It allocates or denies based on availability — nothing more.
The Four Cluster Managers Spark Supports
1. Standalone
Spark's own built-in cluster manager. No external software needed — just Spark. Has its own Master daemon and Worker daemons. Easy to set up. Each application gets a fixed allocation of the cluster.

Best for: Development, testing, simple single-tenant clusters.
Limitation: No resource sharing with other frameworks (Hive, MapReduce, etc.).
2. Hadoop YARN
YARN = Yet Another Resource Negotiator. The resource manager that comes with Hadoop. Allows Spark, MapReduce, Hive, HBase to all share the same cluster simultaneously.

Best for: Enterprises already using Hadoop ecosystem. Supports both client and cluster deploy modes.
Used in: Azure HDInsight, on-premise Hadoop clusters.
3. Kubernetes
Container orchestration platform — runs Spark in Docker containers. Each executor is a separate container (pod). Cloud-native and highly scalable.

Best for: Cloud-native architectures, Azure AKS (Azure Kubernetes Service), teams already using K8s.
Advantage: Fine-grained resource isolation, easy scaling.
4. Apache Mesos
A general-purpose cluster manager that predates YARN. Can manage any distributed application, not just Spark.

Status: Less common now. YARN and Kubernetes have largely replaced it for most use cases.
Note from your notes: Spark supports Mesos but it's rarely used in new deployments.

🟣 Databricks on Azure — No Manual Cluster Manager Needed
When you use Azure Databricks, Databricks handles cluster management entirely automatically. You select the number of workers and VM type — Databricks provisions the VMs, installs Spark, configures the cluster manager, and manages scaling (including auto-scaling). You never interact with YARN, Standalone, or Kubernetes configuration directly. This is one of Databricks' biggest value propositions.

Chapter Six

Driver Program
& Driver Process

The brain of your Spark application. Two terms that sound similar but mean different things. Where does it run? What's inside it?

Driver Program vs Driver Process — The Confusion Cleared
📄 Driver Program (Your Code)
  • The code you write — the Python/Scala script or notebook
  • Contains: SparkSession creation, all transformations and actions
  • Contains the application's main() method (or notebook entry point)
  • One driver program = one SparkContext
  • "The recipe you wrote for the chef"
  • Example: fraud_detection.py, your Databricks notebook
=
🖥️ Driver Process (Runtime JVM)
  • The JVM process that actually runs that code
  • A running process in memory on some machine
  • Contains: SparkSession, DAG Scheduler, Task Scheduler, and your code in memory
  • Communicates with Cluster Manager (to get executors) and with Executors (to send tasks)
  • "The chef actively cooking from your recipe"
  • Example: The PID 12345 JVM process running on the cluster
📘 Simple Summary
Driver Program = your code (static — a file on disk)
Driver Process = your code running (dynamic — a live JVM process)

When people say "the driver", they usually mean the Driver Process.

DAG Scheduler — Splits Your Plan into Stages

The DAG Scheduler takes the optimized execution plan and splits it into Stages. A new stage starts whenever a shuffle boundary is encountered (i.e., whenever a wide transformation like groupBy() or join() requires data to move across the network).

  • Determines task dependencies — Stage N+1 cannot start until Stage N's shuffle write is complete
  • Within a stage: all tasks are independent and run in parallel
  • Submits completed stages to the Task Scheduler
Task Scheduler — The Delivery Service

The Task Scheduler takes each stage from the DAG Scheduler and creates tasks (one per partition). It then assigns tasks to available executor cores based on:

  • Data locality: Prefers to run a task on the executor that already has the data in memory
  • Executor availability: Only assigns tasks to cores that are currently free
  • Resource constraints: Respects memory and core limits per executor

Two Things the Driver Always Does
1

Schedule tasks and instruct executors what to execute

The Driver builds the execution plan, splits it into tasks, and sends each task (serialized code) to the appropriate Executor core. The Driver tells the Executor: "Run this function on Partition #42."

2

Collect and consolidate results from executors

When tasks complete, Executors send results back to the Driver. For actions like count(), show(), or collect(), the Driver receives the results, aggregates them, and returns them to your code (or prints them in the notebook).

⚠️ The Driver Does NOT Process Data
The Driver is the orchestrator, not the processor. It plans, schedules, and collects — but it never touches the actual data partitions. All real data processing happens in Executor cores on Worker nodes. If you call collect(), data is sent TO the driver — but only after being processed by Executors.

The Complete End-to-End Flow

Let's trace exactly what happens when a data engineer clicks "Run" on this Databricks notebook cell:

  The code we're tracing
# Engineer clicks Run on this cell in Databricks
df = spark.read.csv("/mnt/bronze/transactions.csv", header=True)
df_filtered = df.filter(df.amount > 10000)
df_grouped  = df_filtered.groupBy("country").agg({"amount": "sum"})
df_grouped.show()   # ← This is the action that triggers everything
1

Driver Program executes in the Driver Process

The notebook cell is your Driver Program. Databricks runs it in the Driver Process (a JVM on the cluster). A SparkSession already exists — this is the entry point to everything Spark.

2

Lazy evaluation — Driver builds an Unresolved Logical Plan

Lines 1–3 (read, filter, groupBy) run instantly — but no data is read. Each line adds a node to the Logical Plan tree in the Driver's memory. It's just a blueprint.

3

Action called → Catalyst Optimizer kicks in

show() is the action. Now Spark finalizes the plan. The Catalyst Optimizer analyzes the full plan: pushes the filter down to the CSV reader (read less data), prunes unused columns. Output: Optimized Logical Plan → Physical Plan → Java bytecode (Tungsten).

4

DAG Scheduler splits the plan into Stages

The groupBy is a wide transformation (shuffle boundary). The DAG Scheduler creates: Stage 1 (read CSV + filter) and Stage 2 (aggregate by country). Stage 2 must wait for Stage 1 to finish its shuffle write.

5

Driver contacts Cluster Manager for Executors

If executors aren't already running (first job), the Driver contacts the Cluster Manager: "Give me 12 executors with 8 cores each." The Cluster Manager tells 12 Worker nodes to launch Executors. Workers fork new Executor JVM processes. Executors register with the Driver.

6

Task Scheduler dispatches Stage 1 tasks

Say the CSV has 8,000 partitions → 8,000 Stage 1 tasks. Task Scheduler assigns first 96 tasks (one per core). Each executor core streams its partition from ADLS, applies filter, writes shuffle data to local disk. As cores free up, next batch assigned — until all 8,000 Stage 1 tasks complete.

7

Shuffle — data redistributed across the network

Stage 2 tasks start. Each Stage 2 task pulls relevant shuffle data from Stage 1 worker local disks. All rows with country = "IN" go to one executor, all "US" rows to another, etc. This is the shuffle — the most expensive operation.

8

Stage 2 tasks — aggregation

200 Stage 2 tasks (default spark.sql.shuffle.partitions=200) each aggregate their country chunk — summing the amounts. Results are small row-level outputs.

9

Results collected by Driver → show() displays them

Executors send their Stage 2 results back to the Driver. The Driver collects all 200 small partition results, merges them, and displays the final table in the notebook. Done.
