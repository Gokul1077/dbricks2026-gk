for Secure & Compliant Ingestion

📋 Scenario: Your company is moving datasets from Azure SQL Database, Azure Blob Storage, and Azure Data Lake Storage Gen2 (ADLS Gen2) into Databricks for analytics and machine learning. The leadership team is concerned about data security, lineage, and compliance.

❓ Question: How would you design a data governance architecture to ensure that data ingestion from Azure to Databricks is secure, auditable, and compliant?
01 / Foundation

01What is Data Governance? (The 30-Second Answer)

Data Governance is a set of policies, processes, and technologies that ensure data is accurate, secure, consistent, and used responsibly across an organisation.

In a banking context: Think of it as the rulebook that ensures customer PII (Personally Identifiable Information), transaction data, and loan records are not leaked, tampered with, or accessed by unauthorised engineers — even while moving between systems.
🔒

Security

Who can read, write, or delete data? Enforced via RBAC, encryption, and network isolation.

🧾

Auditability

Every access and transformation is logged — who touched what, when, and why.

📏

Compliance

Adherence to regulations like GDPR, PCI-DSS, and internal banking data policies.

🔗

Data Lineage

Tracking data's journey: source → transformation → consumption. Essential for root-cause analysis.

04 / Implementation

04Step-by-Step Implementation

1

Secure the Ingestion Layer — Azure Data Factory (ADF)

Use Managed Identity (not stored credentials) for ADF to connect to Azure SQL DB and ADLS Gen2. Store all connection strings in Azure Key Vault and reference them in ADF Linked Services. Enable Private Endpoints so data never travels over public internet.

Managed Identity Key Vault Reference Private Endpoints ADF Integration Runtime
2

Configure Unity Catalog in Databricks

Unity Catalog is the central metadata and access control layer for Databricks. Create a 3-level namespace: catalog.schema.table (e.g., banking_prod.lending.loan_applications). Assign groups via Azure AD. Grant fine-grained privileges at table, column, or row level.

SQL // Unity Catalog -- Create catalog bound to ADLS Gen2 storage CREATE CATALOG banking_prod MANAGED LOCATION 'abfss://data@bankingstorage.dfs.core.windows.net/'; -- Grant access to Credit Risk team only GRANT SELECT ON TABLE banking_prod.lending.loan_applications TO `credit_risk_analysts`; -- Column-level masking for PII (pan_number) ALTER TABLE banking_prod.lending.customers ALTER COLUMN pan_number SET MASK mask_pan_number USING COLUMNS (is_data_steward);
3

Register Sources in Microsoft Purview

Register Azure SQL DB, ADLS Gen2, and Databricks in Microsoft Purview. Run automated scans to discover and classify sensitive columns (PAN, Aadhaar, account numbers) using built-in and custom classification rules. Purview will automatically build a lineage graph when ADF pipelines run.

Microsoft Purview + Databricks Unity Catalog can be connected natively — Purview pulls lineage events from Unity Catalog audit logs automatically via the Purview-Databricks connector.
Purview Data Map Auto-Classification Lineage Graph Sensitivity Labels
4

Enable Audit Logging End-to-End

Turn on Diagnostic Settings on all Azure resources (ADLS, Azure SQL, Key Vault) to send logs to Azure Log Analytics Workspace. In Databricks, enable Unity Catalog Audit Logs which capture every data access, query, and permission change. Set up Azure Monitor Alerts for unusual access patterns (e.g., bulk export of customer data after hours).

KQL // Azure Log Analytics // Detect suspicious bulk reads on customer data DatabricksAuditLogs | where ActionName == "runCommand" and RequestParams contains "customer_accounts" and bin(TimeGenerated, 1h) > ago(1d) | summarize count() by UserName, bin(TimeGenerated, 1h) | where count_ > 1000 // Alert if >1000 queries/hour
5

Implement Medallion Architecture with Governance Tags

Organise Databricks data into Bronze → Silver → Gold layers. Each layer has stricter access controls. Raw PII lives only in Bronze (restricted). Silver contains masked/anonymised data. Gold contains aggregated, compliance-safe data for BI and ML teams.

Bronze: Raw + Restricted Silver: Cleaned + Masked PII Gold: Aggregated + Open to Analysts
05 / Integration POV

05How Azure Services Connect to Databricks

All these governance tools are natively connectable. Here's how the integration works end-to-end:

🔑

Azure AD → Databricks

Use Azure AD (Entra ID) as the identity provider for Databricks. Users/groups sync automatically. Unity Catalog enforces the same AD group memberships as permissions.

🗄️

ADLS Gen2 → Databricks

Mount ADLS Gen2 using Databricks Access Connector (Managed Identity). No service principal keys stored in notebooks — secure by default.

🌐

Purview → Unity Catalog

Purview's native Databricks connector reads Unity Catalog audit logs to auto-generate lineage graphs without any manual configuration.

🔐

Key Vault → Databricks

Use Databricks Secret Scopes backed by Azure Key Vault. Notebooks read secrets via dbutils.secrets.get() — keys never appear in plain text.

🔗 Complete Connection Code — ADLS Gen2 to Databricks (Secure)

Python // Databricks Notebook # Secure ADLS Gen2 mount using Managed Identity (no keys!) configs = { "fs.azure.account.auth.type": "OAuth", "fs.azure.account.oauth.provider.type": "org.apache.hadoop.fs.azurebfs.oauth2.MsiTokenProvider", "fs.azure.account.oauth2.msi.endpoint": "http://169.254.169.254/metadata/identity/oauth2/token" } dbutils.fs.mount( source = "abfss://raw@bankingstorage.dfs.core.windows.net/", mount_point = "/mnt/banking/raw", extra_configs = configs ) # Read secrets from Key Vault backed scope db_password = dbutils.secrets.get(scope="banking-kv", key="sql-db-password")
06 / Interview Prep

06Interview Insights — How to Answer This

🎯 Possible Interview Questions & Structured Answers
What is Unity Catalog and how does it help with governance?
Unity Catalog is Databricks' centralised data governance layer that provides unified access control, auditing, and lineage across all data assets in a workspace. It uses a 3-level namespace (catalog.schema.table), integrates with Azure AD for identity, and supports column-level masking and row-level filters.
How would you prevent a junior analyst from accessing raw customer PII during ingestion?
  • Use Unity Catalog RBAC — grant SELECT only on Silver/Gold schemas, not Bronze
  • Apply column masking on PAN, Aadhaar columns at the Unity Catalog level
  • Use Azure AD group memberships — analysts in a restricted group can only see anonymised views
  • Log all access attempts via Unity Catalog audit logs → routed to Azure Monitor
How do you prove to auditors that customer data was not tampered with? (indirect governance question)
Use Delta Lake's transaction log (time travel) to show data history. Combine with Unity Catalog audit logs exported to Azure Log Analytics. Purview lineage graph shows every transformation from source to consumption — providing end-to-end proof for regulators (RBI, GDPR auditors).
What's the difference between Microsoft Purview and Unity Catalog?
  • Unity Catalog = Databricks-native governance (access control, lineage within Databricks)
  • Microsoft Purview = Enterprise-wide governance across ALL Azure services (SQL, Blob, ADF, Databricks)
  • They are complementary, not competing — Purview provides the org-level view; Unity Catalog enforces at the compute layer
How do you store secrets securely in Databricks notebooks?
Never hardcode credentials. Use Databricks Secret Scopes backed by Azure Key Vault. Notebooks call dbutils.secrets.get(scope, key) — the value is never printed or logged. Combine with Managed Identity so ADF and Databricks authenticate without storing any passwords at all.

📝 How to Structure Your Answer in an Interview (STAR Format)

  • Situation: "We're moving sensitive banking data from 3 Azure sources to Databricks."
  • Task: "We need to ensure security, lineage, and compliance."
  • Action: "I'd design a 5-pillar governance framework: Identity (Azure AD + Unity Catalog), Encryption (Key Vault + CMK), Lineage (Purview + Delta), Classification (Purview auto-scan + column tags), Auditing (Diagnostic logs + Monitor alerts)."
  • Result: "This gives us end-to-end traceability, zero-trust access, and audit-ready logs for RBI/GDPR compliance."
