SparkSession — The Single Entry Point to Spark

WHAT · WHY · WHERE it lives · What it gives you access to

SparkSession is the unified entry point to all of Spark's functionality. It is a Python/JVM object created inside the Driver Process and is the first thing you create before doing anything in Spark.

🍕 Analogy: SparkSession is like the front desk of a hotel. You check in through the front desk and it gives you access to the room (data), the restaurant (SQL engine), the gym (streaming), and the concierge (catalog). You never go directly to these services — everything routes through the front desk.

SparkSession vs SparkContext vs SQLContext

History, evolution, and why SparkSession unified everything

Understanding this history is a very common interview question. Spark's entry point has evolved across versions:

SparkContext Spark 1.x

  • Original entry point (Spark 1.x)
  • Low-level API — works with RDDs only
  • Creates and manages the connection to the cluster
  • One SparkContext per JVM (hard rule)
  • Still accessible via spark.sparkContext

SQLContext Spark 1.x

  • Added to support SQL and DataFrames
  • Wrapped around SparkContext
  • Required separate import and setup
  • Deprecated since Spark 2.0

HiveContext Spark 1.x

  • Extended SQLContext with Hive support
  • Needed for HiveQL, UDFs, SerDes
  • Deprecated since Spark 2.0

SparkSession Spark 2.0+

  • Unified replacement for ALL of the above
  • Automatically creates a SparkContext internally
  • Supports RDD, SQL, DataFrame, Dataset, Hive
  • Use .builder pattern to create
  • This is what you always use today
✅ Interview answer: "In Spark 1.x you needed separate SparkContext, SQLContext, and HiveContext. Spark 2.0 unified all of them into SparkSession. SparkSession internally creates a SparkContext but you access it all through one object. You never create SparkContext directly in modern PySpark."

The Relationship Diagram

One SparkContext per JVM — Still True: Although SparkSession allows multiple sessions (e.g., for multi-tenant isolation on Databricks), each session shares the same underlying SparkContext. There is still only ONE SparkContext active per JVM process. Calling SparkSession.builder.getOrCreate() returns the existing session if one already exists.

Creating a SparkSession — All Patterns

Builder pattern · configs · getOrCreate() · Databricks behavior

Pattern 1: Basic Creation (Local / Dev)

PySpark — Basic 
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("MySparkApp") \           # Name visible in Spark UI
    .master("local[*]") \             # local[*] = use all CPU cores on this machine
    .getOrCreate()                   # Returns existing session if already running

# Verify it was created
print(spark.version)               # e.g., "3.5.0"
print(spark.sparkContext.appName)  # "MySparkApp"
local[*] explained: local = run on this single machine (no cluster). The [*] means use ALL available CPU cores. local[4] would use exactly 4 cores. local[1] = single-threaded (useful for debugging sequential logic).

Pattern 2: Production Configuration (Azure + Databricks)

PySpark — Production / Azure ADLS 
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("BankingFraudPipeline_v2") \
    # ── Performance Configs ─────────────────────────────
    .config("spark.sql.shuffle.partitions", "200") \        # Default 200 — tune for cluster
    .config("spark.sql.files.maxPartitionBytes", "134217728") \ # 128MB per partition
    .config("spark.sql.adaptive.enabled", "true") \           # AQE: auto-tune at runtime
    .config("spark.sql.adaptive.coalescePartitions.enabled", "true") \
    # ── Delta Lake ──────────────────────────────────────
    .config("spark.sql.extensions", "io.delta.sql.DeltaSparkSessionExtension") \
    .config("spark.sql.catalog.spark_catalog", "org.apache.spark.sql.delta.catalog.DeltaCatalog") \
    # ── Azure ADLS Gen2 Auth (Service Principal) ─────────
    .config("fs.azure.account.auth.type.bankadls.dfs.core.windows.net", "OAuth") \
    .config("fs.azure.account.oauth.provider.type.bankadls.dfs.core.windows.net",
           "org.apache.hadoop.fs.azurebfs.oauth2.ClientCredsTokenProvider") \
    .getOrCreate()

# Dynamically change config at runtime (no restart needed for most configs)
spark.conf.set("spark.sql.shuffle.partitions", "400")

Pattern 3: Databricks — SparkSession is Pre-Created

Databricks Notebook 
# ✅ In Databricks, SparkSession is ALREADY created for you
# The variable 'spark' is available in every notebook cell automatically
# You do NOT need SparkSession.builder in Databricks!

# Just use it directly:
df = spark.read.csv("/mnt/adls/transactions.csv", header=True)

# Access the SparkContext (also pre-created as 'sc'):
print(sc.defaultParallelism)     # Number of cores available
print(spark.version)             # Databricks Runtime Spark version

# If you need a custom session in Databricks (rare), use getOrCreate
# It will simply return the EXISTING session — won't create a new one
custom_spark = SparkSession.builder.getOrCreate()
print(custom_spark is spark)  # True — same object!

Stopping a SparkSession

PySpark — Cleanup 
# Stop the SparkSession (and underlying SparkContext)
# Always do this in standalone scripts to free cluster resources
spark.stop()

# ⚠️ Do NOT call spark.stop() in Databricks notebooks!
# It will stop the shared cluster session — affecting all users/jobs.
🚨 Never call spark.stop() in a Databricks notebook. The SparkSession is shared across the cluster. Stopping it terminates all running jobs for other users on that cluster.

What is a Spark DataFrame?

WHAT · WHY it exists · How it's different from Pandas

A Spark DataFrame is a distributed, immutable, lazily-evaluated tabular data structure — like a spreadsheet or SQL table, but spread across potentially thousands of machines in parallel.

✅ What a DataFrame IS

  • Named columns with a defined schema
  • Organized into logical partitions across executors
  • Immutable — transformations create new DataFrames, never modify in place
  • Lazy — transformations don't execute until an action is called
  • Distributed across executor JVMs in the cluster
  • An abstraction on top of RDDs

❌ What a DataFrame is NOT

  • NOT stored on the driver machine (unless you collect())
  • NOT mutable — you can't change a cell in place
  • NOT eagerly computed like Pandas
  • NOT a single-node data structure
  • NOT the same as a Pandas DataFrame (different API, different execution model)

DataFrame vs Pandas DataFrame — Key Differences

Converting between them: Use df.toPandas() to pull a Spark DataFrame to the driver as a Pandas DataFrame (only safe for small datasets!). Use spark.createDataFrame(pandas_df) to push a Pandas DataFrame into Spark.

Creating DataFrames from Python Lists & Dictionaries

spark.createDataFrame() · with and without schema

Method 1: List of Tuples (Most Common in Unit Tests)

PySpark 
from pyspark.sql import SparkSession
from pyspark.sql.types import StructType, StructField, StringType, IntegerType, DoubleType

spark = SparkSession.builder.appName("Demo").getOrCreate()

# ── Data as list of tuples ──────────────────────────────────────────
data = [
    ("TXN001", "Alice",  "IN", 15000.0, "COMPLETED"),
    ("TXN002", "Bob",    "US", 82000.0, "FLAGGED"),
    ("TXN003", "Carol",  "UK", 3200.0,  "COMPLETED"),
    ("TXN004", "David",  "IN", 120000.0,"FLAGGED"),
]

# ── Option A: Let Spark infer column names from the list ────────────
#    Pass column names as a second argument (list of strings)
df = spark.createDataFrame(data, schema=["txn_id", "name", "country", "amount", "status"])
df.show()

# ── Option B: Define explicit schema (safer for production) ─────────
schema = StructType([
    StructField("txn_id",  StringType(),  nullable=False),
    StructField("name",    StringType(),  nullable=True),
    StructField("country", StringType(),  nullable=True),
    StructField("amount",  DoubleType(),  nullable=True),
    StructField("status",  StringType(),  nullable=True),
])
df = spark.createDataFrame(data, schema=schema)
df.printSchema()
df.show()

Method 2: List of Dictionaries (Very Readable)

PySpark — List of Dicts 
# Each dict = one row. Keys become column names automatically.
data_dicts = [
    {"txn_id": "TXN001", "amount": 15000.0, "status": "COMPLETED"},
    {"txn_id": "TXN002", "amount": 82000.0, "status": "FLAGGED"},
    {"txn_id": "TXN003", "amount": 3200.0,  "status": "COMPLETED"},
]

df = spark.createDataFrame(data_dicts)
df.show()
# Column ORDER is dict key order (Python 3.7+: insertion order guaranteed)

Method 3: From Pandas DataFrame

PySpark — From Pandas 
import pandas as pd

pandas_df = pd.DataFrame({
    "txn_id": ["TXN001", "TXN002"],
    "amount": [15000.0, 82000.0],
})

# Convert Pandas → Spark DataFrame
# Spark infers schema from Pandas dtypes automatically
spark_df = spark.createDataFrame(pandas_df)
spark_df.printSchema()

# Convert back: Spark → Pandas (only safe for small data!)
back_to_pandas = spark_df.toPandas()
# ⚠️ toPandas() pulls ALL data to the driver — never use on large datasets
⚠️ createDataFrame() is an Action, not a Transformation! Unlike spark.read (which is lazy), spark.createDataFrame() immediately serializes your Python list and distributes it to executors. For large datasets this is slow — use spark.read to load from files instead.

Method 4: Using Row Objects

PySpark — Row Objects 
from pyspark.sql import Row

# Row is a named tuple — access fields by name or position
data_rows = [
    Row(txn_id="TXN001", amount=15000.0, status="COMPLETED"),
    Row(txn_id="TXN002", amount=82000.0, status="FLAGGED"),
]

df = spark.createDataFrame(data_rows)
df.show()

# Access a Row's field:
row = data_rows[0]
print(row.txn_id)      # "TXN001"
print(row[0])          # "TXN001" (positional)

Creating DataFrames from CSV

spark.read.csv() · All options explained · Schema inference vs explicit

Basic CSV Read

PySpark — CSV Read 
# ── Simplest form (bad practice in production) ──────────────────────
df = spark.read.csv("/path/to/transactions.csv")
# Problem: Spark gives columns default names _c0, _c1, _c2 etc.
# All types default to StringType — no real type inference

# ── With header and basic type inference ────────────────────────────
df = spark.read \
    .option("header", "true") \          # 1st row = column names
    .option("inferSchema", "true") \     # Scan data to guess types (SLOW — reads twice!)
    .csv("/path/to/transactions.csv")

Production-Grade CSV Read (All Options)

PySpark — Production CSV 
from pyspark.sql.types import StructType, StructField, StringType, DoubleType, DateType, IntegerType

# Always define schema explicitly in production — never use inferSchema on large files
txn_schema = StructType([
    StructField("txn_id",    StringType(),  nullable=False),
    StructField("account_id",StringType(),  nullable=False),
    StructField("country",   StringType(),  nullable=True),
    StructField("amount",    DoubleType(),  nullable=True),
    StructField("currency",  StringType(),  nullable=True),
    StructField("status",    StringType(),  nullable=True),
    StructField("txn_date",  StringType(),  nullable=True),
])

df = spark.read \
    .format("csv") \                          # Explicit format
    .schema(txn_schema) \                       # Our schema — no scan needed
    .option("header", "true") \               # First row is header
    .option("sep", ",") \                       # Delimiter (default comma)
    .option("quote", '"') \                    # Quote char for fields with commas
    .option("escape", "\\") \                  # Escape character
    .option("nullValue", "NA") \               # Treat "NA" string as null
    .option("emptyValue", "") \               # Treat empty fields as empty string
    .option("ignoreLeadingWhiteSpace", "true") \
    .option("ignoreTrailingWhiteSpace", "true") \
    .option("mode", "PERMISSIVE") \            # PERMISSIVE=null bad rows, DROPMALFORMED=skip, FAILFAST=error
    .option("columnNameOfCorruptRecord", "_corrupt_record") \ # Track bad rows
    .option("encoding", "UTF-8") \            # Character encoding
    .load("abfss://raw@bankadls.dfs.core.windows.net/transactions/*.csv")
    # ↑ Glob pattern — reads ALL CSV files in the folder

print(f"Rows: {df.count()}, Partitions: {df.rdd.getNumPartitions()}")
df.printSchema()
df.show(5, truncate=False)

CSV Read Modes — Critical Interview Topic

ModeBehaviour on Malformed RowsWhen to Use
PERMISSIVESets fields to null; optionally stores bad row in _corrupt_record columnDefault. Good for debugging — keeps bad rows visible
DROPMALFORMEDSilently drops (ignores) rows that can't be parsedWhen bad rows are expected and you want clean data
FAILFASTImmediately throws an exception on the first bad rowProduction pipelines where data quality must be 100%
inferSchema vs explicit schema — always use explicit in production:
• inferSchema=true reads the entire file twice (once to infer types, once to load) — doubles I/O cost.
• With explicit schema, Spark reads once and trusts you for the types.
• inferSchema can guess wrong (e.g., an integer column with one "N/A" row becomes String).

Nested JSON — Flattening with dot notation and explode()

Nested JSON File 
{"txn_id":"TXN001","amount":15000.0,"customer":{"name":"Alice","tier":"Gold"},"tags":["urgent","review"]}
PySpark — Handling Nested JSON 
from pyspark.sql import functions as F

df = spark.read.json("/path/to/nested_txns.json")

df.printSchema()
# Spark auto-detects nested struct and array:
# root
#  |-- txn_id: string
#  |-- amount: double
#  |-- customer: struct              ← nested object
#  |    |-- name: string
#  |    |-- tier: string
#  |-- tags: array<string>          ← array

# Access nested struct field using dot notation:
df_flat = df.select(
    "txn_id",
    "amount",
    "customer.name",         # Dot notation for struct field
    "customer.tier",
    F.col("customer.name").alias("cust_name"),  # Rename
)

# Explode array — creates one row per array element:
df_exploded = df.select("txn_id", F.explode(F.col("tags")).alias("tag"))
df_exploded.show()

JSON with Explicit Schema (Nested StructType)

PySpark — Nested Schema Definition 
from pyspark.sql.types import StructType, StructField, StringType, DoubleType, ArrayType

customer_schema = StructType([
    StructField("name", StringType(), nullable=True),
    StructField("tier", StringType(), nullable=True),
])

full_schema = StructType([
    StructField("txn_id",   StringType(),           nullable=False),
    StructField("amount",   DoubleType(),           nullable=True),
    StructField("customer", customer_schema,         nullable=True),   # nested struct
    StructField("tags",     ArrayType(StringType()), nullable=True),   # array of strings
])

df = spark.read.schema(full_schema).json("/path/to/nested.json")

Creating DataFrames from Parquet, Delta, and Other Formats

Why Parquet is the production standard · Delta Lake · ORC · Avro

Why Parquet (Not CSV) in Production?

✅ Parquet

  • Columnar storage — reads only needed columns
  • Stores schema inside the file — no inferSchema needed
  • Built-in compression (Snappy/ZSTD) — 10-20x smaller than CSV
  • Supports predicate pushdown — reads only matching row groups
  • Industry standard for big data

⚠️ CSV

  • Row-oriented — reads all columns even if not needed
  • No schema — must infer or define manually
  • No compression by default
  • No predicate pushdown
  • Good for: data exchange, Excel exports, source ingestion

🔵 Delta Lake

  • Parquet + transaction log (_delta_log)
  • ACID transactions, time travel, schema evolution
  • Optimized for Databricks / ADLS Gen2
  • Recommended format for all Databricks pipelines

Reading Parquet

PySpark — Parquet 
# No schema needed — Parquet embeds schema in the file footer
df = spark.read \
    .format("parquet") \
    .load("abfss://silver@bankadls.dfs.core.windows.net/transactions/")

# Shorthand (same result):
df = spark.read.parquet("abfss://silver@bankadls.dfs.core.windows.net/transactions/")

# Reading a partitioned Parquet folder (e.g., partitioned by date):
# /transactions/year=2024/month=01/part-0000.parquet
# /transactions/year=2024/month=02/part-0000.parquet
df = spark.read.parquet("abfss://silver@bankadls.dfs.core.windows.net/transactions/")
# Spark automatically picks up partition columns year, month as DF columns

# ── Partition Pruning (huge performance win): ──────────────────────
df_jan = df.filter((df.year == 2024) & (df.month == 1))
# Spark reads ONLY /year=2024/month=01/ folder — skips everything else!

Reading Delta Lake

PySpark — Delta Lake (Databricks / Azure) 
# Read current version (latest snapshot)
df = spark.read \
    .format("delta") \
    .load("abfss://gold@bankadls.dfs.core.windows.net/fraud_summary/")

# ── Time Travel: Read a previous version ──────────────────────────
df_v3 = spark.read \
    .format("delta") \
    .option("versionAsOf", 3) \       # Read version 3
    .load("abfss://gold@bankadls.dfs.core.windows.net/fraud_summary/")

# ── Time Travel: Read by timestamp ────────────────────────────────
df_ts = spark.read \
    .format("delta") \
    .option("timestampAsOf", "2024-01-15 00:00:00") \
    .load("abfss://gold@bankadls.dfs.core.windows.net/fraud_summary/")

# ── Read as SQL Table (Unity Catalog) ─────────────────────────────
df = spark.read.table("banking_catalog.fraud.summary")
# Equivalent SQL:
df = spark.sql("SELECT * FROM banking_catalog.fraud.summary")

Schema — StructType, StructField & All Data Types

Defining schemas explicitly · Why it matters · All Spark types

A Schema in Spark describes the structure of a DataFrame: the column names, their data types, and whether they can contain null values. It is represented as a StructType object.

Anatomy of a Schema

PySpark — Full Schema Definition 
from pyspark.sql.types import (
    StructType, StructField,
    # String
    StringType,
    # Numeric
    IntegerType, LongType, FloatType, DoubleType, DecimalType,
    ShortType, ByteType,
    # Boolean
    BooleanType,
    # Date & Time
    DateType, TimestampType, TimestampNTZType,
    # Complex Types
    ArrayType, MapType,
    # Binary
    BinaryType,
)

banking_schema = StructType([
    # StructField(name, dataType, nullable)
    #                              ↑ True = column CAN have null values
    StructField("txn_id",      StringType(),          nullable=False), # Primary key — never null
    StructField("account_id",  StringType(),          nullable=False),
    StructField("txn_date",    TimestampType(),       nullable=True),
    StructField("amount",      DecimalType(18, 4),    nullable=True), # 18 total digits, 4 decimal
    StructField("currency",    StringType(),          nullable=True),
    StructField("country",     StringType(),          nullable=True),
    StructField("is_flagged",  BooleanType(),         nullable=True),
    StructField("risk_score",  DoubleType(),          nullable=True),
    StructField("tags",         ArrayType(StringType()),nullable=True), # array column
    StructField("metadata",    MapType(StringType(), StringType()), nullable=True), # map column
    StructField("customer",    StructType([            # nested struct
        StructField("name", StringType(), True),
        StructField("tier", StringType(), True),
    ]), nullable=True),
])

🚨 Use DecimalType for Money! Never use DoubleType or FloatType for monetary amounts. Floating-point arithmetic has precision errors (e.g., 0.1 + 0.2 ≠ 0.3 in IEEE 754). Always use DecimalType(18, 4) for financial data — it gives you exact decimal arithmetic.

spark.read — Full API Reference

The DataFrameReader Chain

Every spark.read call returns a DataFrameReader object. You chain methods on it to configure the read, then call a load method (.load(), .csv(), etc.) to get a DataFrame. This is lazy — no actual reading happens until an action.

PySpark — DataFrameReader Pattern 
# Full generic pattern:
df = spark.read \
    .format("format_name") \   # csv / json / parquet / delta / jdbc / orc / avro
    .schema(my_schema) \        # Optional: provide explicit StructType or DDL string
    .option("key", "value") \  # Format-specific options
    .option("key2", "val2") \  # Chain multiple options
    .options({"k":"v", "k2":"v2"}) \ # Or pass a dict of options
    .load("path/to/data")      # Path — can be a file, folder, glob, list

# Shorthand for common formats (skips .format() and .load()):
df_csv     = spark.read.csv("path")
df_json    = spark.read.json("path")
df_parquet = spark.read.parquet("path")
df_orc     = spark.read.orc("path")
df_table   = spark.read.table("catalog.database.table_name")
df_text    = spark.read.text("path")  # Each line as a row in "value" column

Real-World Banking Production Example

Azure ADLS Gen2 + Databricks · Bronze → Silver → Gold ingestion pipeline

Scenario: Ingest daily transaction CSV files from Azure ADLS Gen2 Bronze layer, validate schema, handle bad records, enrich with customer JSON reference data, and write enriched data to Delta Lake Silver layer.

PySpark — Full Ingestion Pipeline (Databricks Notebook) 
from pyspark.sql import SparkSession, functions as F
from pyspark.sql.types import (
    StructType, StructField, StringType, DecimalType,
    TimestampType, BooleanType, LongType
)

# ═══════════════════════════════════════════════════════════
# STEP 1: SparkSession (pre-created in Databricks — just reference it)
# ═══════════════════════════════════════════════════════════
# In Databricks: 'spark' is already available. Configure runtime props:
spark.conf.set("spark.sql.shuffle.partitions", "200")
spark.conf.set("spark.sql.adaptive.enabled", "true")

# ADLS paths (using ADLS Gen2 abfss protocol)
BRONZE = "abfss://bronze@bankadls.dfs.core.windows.net"
SILVER = "abfss://silver@bankadls.dfs.core.windows.net"

# ═══════════════════════════════════════════════════════════
# STEP 2: Define Explicit Schema (NEVER use inferSchema in production)
# ═══════════════════════════════════════════════════════════
txn_schema = StructType([
    StructField("txn_id",     StringType(),         nullable=False),
    StructField("account_id", StringType(),         nullable=False),
    StructField("cust_id",    StringType(),         nullable=True),
    StructField("amount",     DecimalType(18, 4),   nullable=True),
    StructField("currency",   StringType(),         nullable=True),
    StructField("country",    StringType(),         nullable=True),
    StructField("txn_ts",     TimestampType(),      nullable=True),
    StructField("status",     StringType(),         nullable=True),
    StructField("channel",    StringType(),         nullable=True),
])

cust_schema = StructType([
    StructField("cust_id",   StringType(), nullable=False),
    StructField("cust_name", StringType(), nullable=True),
    StructField("risk_tier", StringType(), nullable=True),   # LOW / MEDIUM / HIGH
    StructField("is_active", BooleanType(),nullable=True),
])

# ═══════════════════════════════════════════════════════════
# STEP 3: Read CSV Transactions from Bronze Layer
# ═══════════════════════════════════════════════════════════
df_raw = spark.read \
    .format("csv") \
    .schema(txn_schema) \
    .option("header", "true") \
    .option("nullValue", "NA") \
    .option("mode", "PERMISSIVE") \
    .option("columnNameOfCorruptRecord", "_corrupt_record") \
    .option("encoding", "UTF-8") \
    .load(f"{BRONZE}/transactions/2024/*.csv")       # All daily files
    .withColumn("source_file", F.input_file_name())    # Track origin file
    .withColumn("ingestion_ts", F.current_timestamp())  # Add watermark

# ── Inspect what we got ──────────────────────────────────────────
print(f"Raw rows: {df_raw.count()}")
print(f"Partitions: {df_raw.rdd.getNumPartitions()}")
df_raw.printSchema()
df_raw.show(5, truncate=False)

# ── Separate bad records ─────────────────────────────────────────
df_bad  = df_raw.filter(F.col("_corrupt_record").isNotNull())
df_good = df_raw.filter(F.col("_corrupt_record").isNull()).drop("_corrupt_record")
print(f"Good rows: {df_good.count()}, Bad rows: {df_bad.count()}")

# ─── Write bad records to quarantine zone ────────────────────────
df_bad.write.mode("append").format("delta") \
    .save(f"{BRONZE}/quarantine/transactions/")

# ═══════════════════════════════════════════════════════════
# STEP 4: Read Customer Reference Data (JSON from Bronze)
# ═══════════════════════════════════════════════════════════
df_cust = spark.read \
    .schema(cust_schema) \
    .option("multiLine", "false") \   # JSON Lines format
    .json(f"{BRONZE}/reference/customers.json")

df_cust.printSchema()
print(f"Customers: {df_cust.count()}")

# ═══════════════════════════════════════════════════════════
# STEP 5: Enrich transactions with customer data
# Broadcast join — customers is small (avoids shuffle stage!)
# ═══════════════════════════════════════════════════════════
df_enriched = df_good.join(
    F.broadcast(df_cust),
    on="cust_id",
    how="left"
).select(
    "txn_id", "account_id", "cust_id", "cust_name",
    "amount", "currency", "country", "txn_ts", "status",
    "risk_tier", "channel", "source_file", "ingestion_ts"
)

df_enriched.explain()  # Confirm BroadcastHashJoin in physical plan

# ═══════════════════════════════════════════════════════════
# STEP 6: Write to Delta Lake Silver Layer
# ═══════════════════════════════════════════════════════════
df_enriched.write \
    .format("delta") \
    .mode("append") \
    .partitionBy("country", F.year(F.col("txn_ts")).alias("year")) \
    .option("mergeSchema", "true") \   # Allow schema evolution
    .save(f"{SILVER}/transactions_enriched/")

# ═══════════════════════════════════════════════════════════
# STEP 7: Quick summary for pipeline log
# ═══════════════════════════════════════════════════════════
df_enriched.describe("amount").show()
print(f"Written {df_enriched.count()} rows to Silver layer")
print(f"Output partitions: {df_enriched.rdd.getNumPartitions()}")

Interview Insights

5 real interview questions with model answers and common mistakes
What is SparkSession? How is it different from SparkContext?
"SparkContext was the original entry point in Spark 1.x — it gave you the low-level RDD API and managed the cluster connection. Spark 1.x also had SQLContext for DataFrames and HiveContext for Hive. Spark 2.0 unified all of these into SparkSession, which internally creates and manages a SparkContext but exposes everything through one clean object. In production PySpark today, you always use SparkSession — you never create SparkContext directly. In Databricks, SparkSession is pre-created and available as the variable spark."
Why should you define schemas explicitly instead of using inferSchema?
"Three reasons. First, performance — inferSchema reads the entire file twice (once to infer types, once to load data), doubling I/O on potentially terabytes of data. Second, correctness — inferSchema can guess wrong; a column with 999 integers and one 'NA' becomes StringType, causing downstream failures. Third, schema drift protection — if the source file changes column order or adds a column, inferSchema silently adjusts while an explicit schema fails loudly, alerting the team. In my banking pipeline, I always define StructType with DecimalType(18,4) for monetary amounts — inferSchema would use DoubleType which has floating-point precision issues for financial calculations."
What's the difference between show() and collect() in Spark?
"Both are actions that trigger DAG execution, but they differ in where data goes. show(n) displays n rows in the console — Spark computes only enough to show those rows and prints them. The data stays distributed. collect() pulls all partitions from every executor back to the Driver JVM. On a 100GB DataFrame, collect() would crash the driver with OutOfMemoryError. In production I never use collect() on real data — I use show() for previewing, count() for validation, and write() to persist results."
You read a CSV file and see all columns are StringType even after setting inferSchema=true. Why?
"Two common causes. First, the header option is missing — without option("header","true"), Spark treats the first row as data, all as strings. Second, bad data in the sample — Spark samples a portion of the file for inference; if that portion has non-numeric values in a column (like 'N/A', 'null', or 'TBD'), it infers StringType for the whole column. The fix is to always define an explicit schema with StructType and use option("nullValue","N/A") to map custom null strings. This is exactly why I never rely on inferSchema in production."
Explain the difference between a Spark DataFrame and an RDD.
"An RDD (Resilient Distributed Dataset) is Spark's original low-level distributed data structure — it's an untyped collection of JVM objects with no schema. A DataFrame is a higher-level abstraction: it's essentially an RDD of Row objects plus a schema (StructType). The key advantage of DataFrames is the Catalyst Optimizer — Spark analyzes the full query plan and applies predicate pushdown, column pruning, and join reordering before execution. RDDs skip all of this. For structured data (CSV, Parquet, SQL), DataFrames are always faster and easier. I only drop down to RDDs for truly unstructured data, like processing raw binary files or writing custom partitioners."

🚨 Common Mistakes Candidates Make

  • Saying spark.read.csv() reads data immediately — it's lazy, no data is read until an action.
  • Forgetting header="true" for CSV — all columns default to _c0, _c1, _c2 and StringType.
  • Using DoubleType for monetary columns — always use DecimalType(18,4) for financial precision.
  • Calling spark.stop() in a Databricks notebook — kills the cluster session for everyone.
  • Using collect() on large DataFrames — pulls all data to the driver, causing OutOfMemoryError.
  • Confusing show() (console preview) with display() (Databricks-specific rich table).
  • Not knowing that printSchema() does NOT trigger a Spark job — it reads metadata from the plan.
  • Saying "I create a SparkContext in my PySpark code" — you always create a SparkSession; SparkContext is managed internally.

Pro Tips

Production best practices from real data engineering pipelines
⚡
Cache strategically, not aggressively. Use df.cache() or df.persist() only when the same DataFrame is used in multiple actions. Caching stores the data in executor memory — if you cache a 100GB DataFrame on a cluster with only 50GB executor memory, Spark spills to disk and you lose the benefit. Always call df.unpersist() when done. In Databricks, use df.cache() for reference tables (customer master, currency codes) that are reused in every join.
🔬
Use explain() before running expensive jobs. Call df.explain(extended=True) on your DataFrame before running a long pipeline. Check that filters appear near the bottom of the physical plan (predicate pushdown working) and joins use BroadcastHashJoin for small tables (not SortMergeJoin which triggers a shuffle). This 30-second check can save you hours of debugging slow jobs.
🗂️
Always add ingestion metadata columns. When reading from CSV or JSON, add F.input_file_name() and F.current_timestamp() immediately after read. This tracks which source file each row came from and when it was ingested — invaluable for debugging data quality issues in production pipelines on ADLS. In Delta Lake, combine this with mergeSchema=true to allow graceful schema evolution as source files add new columns.
🔢
Check partition count after every major read or join. Run df.rdd.getNumPartitions() after reading a file and after shuffles. If partitions are too few (e.g., 4 on a 96-core cluster), most cores sit idle. If too many (e.g., 10,000 tiny partitions), shuffle overhead dominates. Target 2-3 partitions per core as a starting point. Use df.repartition(n) to increase (triggers shuffle) or df.coalesce(n) to decrease (no shuffle — just combines).

Quiz Questions

Scenario-based questions to test your understanding
Q1. A Data Engineer reads a CSV file with this code: df = spark.read.csv("/data/transactions.csv") and then runs df.printSchema(). They see columns named _c0, _c1, _c2 all of StringType. What are the TWO things they need to fix?
  • A) Add .option("inferSchema","true") and .option("header","true")
  • B) Add .schema(mySchema) and .option("header","true")
  • C) Use spark.read.format("csv").load() instead of spark.read.csv()
  • D) Call df.show() first to trigger schema inference
✅ Correct: B (preferred) or A (acceptable but not production-grade)

The _c0, _c1 names appear because header="true" is missing — Spark doesn't know the first row is headers. Option A adds inferSchema which fixes types but reads the file twice and can still guess wrong. Option B is the production-correct answer: define an explicit schema with StructType (correct types, including DecimalType for money) and add header="true". Option C is wrong — the format difference doesn't affect this issue. Option D is wrong — show() is an action that displays data but doesn't change the schema.
Q2. In your Databricks notebook, you write: spark2 = SparkSession.builder.appName("Test").getOrCreate(). After this, you run print(spark2 is spark). What does it print and why?
  • A) False — a brand-new SparkSession named "Test" is created
  • B) True — getOrCreate() returns the existing session; there can only be one active SparkSession per JVM
  • C) Error — you cannot call SparkSession.builder inside a Databricks notebook
  • D) False — spark2 is a separate session but shares the same SparkContext
✅ Correct: B

getOrCreate() checks if a SparkSession already exists in the JVM. Since Databricks pre-creates one, getOrCreate() returns that same object — spark2 is spark evaluates to True. There is only ONE active SparkSession (and ONE SparkContext) per JVM process. Option D describes a real pattern (multiple logical sessions sharing one SparkContext) in advanced multi-tenant setups, but even then getOrCreate() returns the primary session, not a new one.
Q3. You run df = spark.read.parquet("/adls/transactions/"). This folder has 10 Parquet files, each 500MB. How many Spark partitions will the DataFrame have by default, and what controls this?
  • A) 10 partitions — one per file, always
  • B) ~40 partitions — controlled by spark.sql.files.maxPartitionBytes (default 128MB), 500MB ÷ 128MB ≈ 4 per file × 10 files
  • C) 200 partitions — controlled by spark.sql.shuffle.partitions
  • D) 1 partition — Parquet merges all files into one partition for efficiency
✅ Correct: B

Spark splits input files based on spark.sql.files.maxPartitionBytes (default 128MB). Each 500MB Parquet file is logically split into approximately ⌈500/128⌉ = 4 partitions. With 10 files: ~40 partitions total. Option A is wrong — files are split by byte size, not kept whole. Option C is wrong — spark.sql.shuffle.partitions only controls the number of partitions after a shuffle (in Stage 2+), not initial read partitions. Option D is completely wrong.
