Before writing a single line of PySpark, you need to understand what's happening under the hood.

Think of Spark as a factory assembly line. Before you can rearrange products (transformations), you need to understand the factory layout — who gives orders, who does the work, and how data moves.

😴 Lazy Evaluation — The #1 Concept You Must Nail

Lazy Evaluation: Spark does NOT execute transformations the moment you write them. It builds a logical plan (DAG) and only executes when an action is called. This allows Catalyst to optimize the entire pipeline before running a single line.
PySpark 
# ALL of these are LAZY — nothing runs yet
df = spark.read.parquet("/mnt/data/transactions")   # lazy
df2 = df.filter(df.amount > 1000)                    # lazy
df3 = df2.select("account_id", "amount")              # lazy

# This ACTION triggers the entire chain to execute
df3.show()   # ← ONLY now does Spark run!

# Catalyst sees all 3 steps together and may push the filter
# BEFORE the read (predicate pushdown) — saving massive I/O
💡 Interview Gold: "Lazy evaluation allows Catalyst Optimizer to see the full picture and apply optimizations like predicate pushdown and column pruning before any data is actually read."

🔀 Narrow vs Wide Transformations — Critical for Interviews

TypeWhat it meansShuffle?Stage?Examples
NARROW Each output partition depends on only ONE input partition ❌ No shuffle Same stage filter, select, withColumn, drop, cast, alias
WIDE Each output partition depends on MANY input partitions ✅ Shuffle needed New stage groupBy, join, orderBy, distinct, repartition
🌟 All 6 transformations we'll study (select, filter, withColumn, drop, alias, cast) are NARROW — they never cause a shuffle!

Chapter 02

Transformations 101 — The Big Picture

Six core transformations, their syntax patterns, and how to remember each one.

A transformation is a RECIPE STEP — you describe what to do, Spark decides when and how to do it.
🍳 Analogy: Writing a recipe (transformations) vs. actually cooking (action). You can write 10 recipe steps — but the meal only gets cooked when someone says "serve it now" (action like show()).

⚡ The 6 Core Transformations — Quick Reference

#FunctionWhat it doesTypeReturnsMemory Pattern
1 select() Choose specific columns NARROW New DataFrame 📷 "Camera — pick what to photograph"
2 filter() / where() Keep rows matching a condition NARROW New DataFrame 🧹 "Broom — sweep out rows you don't want"
3 withColumn() Add or replace a column NARROW New DataFrame ✏️ "Add a new column to a spreadsheet"
4 drop() Remove one or more columns NARROW New DataFrame 🗑️ "Trash — throw away columns"
5 alias() Rename a column or DataFrame NARROW Column / DataFrame 🏷️ "Nametag — relabel without changing content"
6 cast() Change a column's data type NARROW Column expression 🔄 "Converter — change format, keep value"

🔑 Syntax Pattern — How to Read PySpark Transformations

SYNTAX PATTERN 
# Pattern: new_df = old_df.transformation(what_to_do)
#          ↑            ↑               ↑
#      result df      method         column names / expressions

# All transformations return a NEW DataFrame (immutable!)
# You can chain them: df.filter(...).select(...).withColumn(...)

# THREE ways to reference columns in PySpark:
df.select("col_name")          # 1. String — simplest
df.select(df.col_name)          # 2. Dot notation — df.column
df.select(col("col_name"))      # 3. col() function — most powerful
⚠️ DataFrames are IMMUTABLE — every transformation returns a brand new DataFrame. The original is never modified. This is what enables Spark's fault-tolerance and lazy evaluation.

1
select() — Choose Your Columns
Project specific columns from a DataFrame
Like a SQL SELECT col1, col2 FROM table
📷 Camera analogy: You frame only what you want in the shot. Everything else is cropped out.
PySpark — select() 
# ─── SYNTAX PATTERN ───────────────────────────────────────────────
# df.select("col1", "col2", ...)           → using strings
# df.select(col("col1"), col("col2"))      → using col() function
# df.select(df.col1, df.col2)             → using dot notation
# ──────────────────────────────────────────────────────────────────

# Basic select — pick 3 columns
df_basic = df.select("txn_id", "account_id", "amount")

# Select with expression — compute inline
df_expr = df.select(
    col("txn_id"),
    col("amount"),
    (col("amount") * 1.18).alias("amount_with_gst")   # inline computation!
)

# Select ALL columns except one (common pattern)
df_except = df.select([c for c in df.columns if c != "country_code"])

df_basic.show(3)
Memory: "SELECT in SQL = select() in Spark. Same purpose, same position — first thing after FROM."
⚡ Catalyst Optimization: When you select only 3 columns from a Parquet file with 30 columns, Catalyst applies Column Pruning — it only reads those 3 columns from disk. Massive I/O savings!
2
filter() / where() — Keep Only What You Need
Row-level filtering — identical to SQL WHERE clause
Like SQL WHERE condition — keeps rows that satisfy a condition
🧹 Broom analogy: Sweep out all rows that don't match. filter and where are 100% identical — use whichever reads more naturally.
PySpark — filter() / where() 
# ─── SYNTAX PATTERN ───────────────────────────────────────────────
# df.filter(condition)        OR    df.where(condition)
# condition can be:  col("x") > 100   |  "x > 100" (SQL string)  |  df.x > 100
# ──────────────────────────────────────────────────────────────────

# Filter using col() — recommended ✅
df_flagged = df.filter(col("status") == "FLAGGED")

# Filter using SQL string — great for readability
df_high_value = df.where("amount > 100000")

# Multiple conditions — use & (AND)  | (OR)  ~ (NOT)
# IMPORTANT: wrap each condition in parentheses!
df_risk = df.filter(
    (col("amount") > 100000) &
    (col("status") == "FLAGGED") &
    (col("country_code") != "IN")
)

# Filter using isin() — like SQL IN (...)
df_intl = df.filter(col("country_code").isin(["UK", "AE", "US"]))

df_flagged.show()
Memory: "filter = WHERE in SQL. filter() and where() are ALIASES — same function, different names."
🚀 Predicate Pushdown: Catalyst moves your filter as close to the data source as possible. If reading Parquet from ADLS, the filter is applied at file-scan time — only matching data enters the executor memory!
3
withColumn() — Add or Replace a Column
Derive new columns or overwrite existing ones
Add a computed column or replace an existing one — like adding a formula column in Excel
✏️ Spreadsheet analogy: "Add column D = column B × 1.18" — same row, new derived value.
PySpark — withColumn() 
# ─── SYNTAX PATTERN ───────────────────────────────────────────────
# df.withColumn("new_col_name", expression)
# If "new_col_name" already exists → REPLACES it
# If "new_col_name" is new → ADDS it
# ──────────────────────────────────────────────────────────────────

# Add a constant column (using lit())
df_source = df.withColumn("source_system", lit("CORE_BANKING"))

# Derive amount in USD (multiply)
df_usd = df.withColumn("amount_usd", round(col("amount") / 83.5, 2))

# Conditional column using when().otherwise()
df_risk_level = df.withColumn(
    "risk_level",
    when(col("amount") > 500000, "HIGH")
    .when(col("amount") > 50000,  "MEDIUM")
    .otherwise("LOW")
)

# Parse date column (string → date type)
df_parsed = df.withColumn("txn_date", to_date(col("txn_date"), "yyyy-MM-dd"))

# Chain multiple withColumn calls
df_enriched = (df
    .withColumn("amount_usd",  round(col("amount") / 83.5, 2))
    .withColumn("is_flagged",  col("status") == "FLAGGED")
    .withColumn("source_system", lit("CORE_BANKING"))
)

df_risk_level.show(truncate=False)
Memory: "withColumn = WITH a new COLUMN. Two arguments: what to name it, what to put in it."
⚠️ Performance Note: Chaining 20+ withColumn calls can cause performance issues in older Spark versions (each adds a Project node to the plan). Prefer select() with multiple expressions when adding many columns at once.
4
drop() — Remove Columns
Column pruning — reduce the schema by removing unwanted columns
Remove one or more columns from a DataFrame
🗑️ Trash analogy: Throw away columns you no longer need — the opposite of select().
PySpark — drop() 
# ─── SYNTAX PATTERN ───────────────────────────────────────────────
# df.drop("col_name")                → drop 1 column
# df.drop("col1", "col2", "col3")   → drop multiple columns
# df.drop(*list_of_cols)             → drop from a list (unpack with *)
# ──────────────────────────────────────────────────────────────────

# Drop a single column
df_no_country = df.drop("country_code")

# Drop multiple columns at once
df_minimal = df.drop("country_code", "txn_type", "status")

# Drop from a list (production pattern — configure which cols to drop)
pii_columns = ["customer_name", "account_id"]  # PII columns to mask
df_masked = df.drop(*pii_columns)

# drop() is safe — if column doesn't exist, no error is raised
df_safe = df.drop("non_existent_column")   # No error! Returns df as-is

df_masked.show(3)
Memory: "drop() = select() in reverse. Use drop when you know what to REMOVE; use select when you know what to KEEP."
💡 Use drop() when: You have 30 columns and want all except 2. Use select() when you want only 2 from 30. Both achieve the same result — pick whichever is shorter to write.
5
alias() — Rename Columns or DataFrames
Used for column renaming AND DataFrame aliasing for joins
Give a column or DataFrame a new name — without changing the underlying data
🏷️ Nametag analogy: You put a "Manager" nametag on someone — their identity doesn't change, just how others refer to them.
PySpark — alias() 
# ─── SYNTAX PATTERN ───────────────────────────────────────────────
# Column alias:    col("old").alias("new")   — used inside select()
# DataFrame alias: df.alias("shortname")     — used for joins
# ──────────────────────────────────────────────────────────────────

# Column alias — rename during select
df_renamed = df.select(
    col("txn_id").alias("transaction_id"),
    col("account_id").alias("acc_no"),
    col("amount").alias("txn_amount_inr"),
    col("customer_name").alias("cust_name")
)

# Alias on computed expression
df_computed = df.select(
    col("txn_id"),
    (round(col("amount") / 83.5, 2)).alias("amount_usd"),
    (upper(col("customer_name"))).alias("customer_upper")
)

# DataFrame alias — critical for self-joins / disambiguating columns
txn_a = df.alias("txn_a")
txn_b = df.alias("txn_b")

# Now use qualified names in join condition
df_self_join = txn_a.join(txn_b, col("txn_a.account_id") == col("txn_b.account_id"))

# withColumnRenamed() — alternative for simple column rename
df_simple_rename = df.withColumnRenamed("amount", "txn_amount")

df_renamed.show(3)
Memory: "alias() = 'also known as'. Works on COLUMNS (inside select) and DataFrames (for joins). Two jobs, one function."
6
cast() — Change Data Types
Type coercion — convert a column's data type explicitly
Convert a column from one data type to another — like SQL CAST(col AS INT)
🔄 Currency converter analogy: Same value (₹500), different format (USD equivalent). The number is reinterpreted in a new type system.
PySpark — cast() 
# ─── SYNTAX PATTERN ───────────────────────────────────────────────
# col("col_name").cast(DataType())         → using Type objects
# col("col_name").cast("string")          → using string shorthand
# Always used INSIDE withColumn() or select()
# ──────────────────────────────────────────────────────────────────

# Cast using string shorthand (most common)
df_typed = df.withColumn("amount", col("amount").cast("integer"))

# Cast using Type objects (explicit, production-grade)
df_explicit = df.withColumn("amount", col("amount").cast(IntegerType()))

# Cast inside select — rename + cast in one step
df_clean = df.select(
    col("txn_id"),
    col("amount").cast(DoubleType()).alias("amount_dbl"),   # cast + alias
    col("txn_date").cast(DateType()).alias("txn_date_typed"),
    col("country_code").cast(StringType())
)

# Common type strings you MUST know for interviews:
# "string"    → StringType()
# "integer"   → IntegerType()
# "long"      → LongType()
# "double"    → DoubleType()
# "float"     → FloatType()
# "boolean"   → BooleanType()
# "date"      → DateType()
# "timestamp" → TimestampType()

df_clean.printSchema()   # Verify types changed
Memory: "cast() = type converter. Always used INSIDE withColumn() or select(). Returns a Column — not a DataFrame!"
⚠️ Gotcha: If cast fails (e.g., casting "ABC" to integer), Spark returns null silently — NOT an error! Always validate your data before casting in production.

# Fraud detection: flag high-value international transactions

df_fraud_candidates = (
    df
    .filter(col("country_code") != "IN")         # international only
    .filter(col("amount") > 50000)               # high value
    .withColumn("fraud_score",
        when((col("amount") > 500000) & (col("status") == "FLAGGED"), 0.95)
        .when(col("amount") > 100000, 0.65)
        .otherwise(0.30)

10
Full Pipeline — All 6 Transformations Together
Bronze → Silver transformation: complete end-to-end data quality pipeline
PySpark — Complete Pipeline (Bronze → Silver) 
# ═══════════════════════════════════════════════════════════════
# BRONZE → SILVER transformation pipeline
# Uses ALL 6 core transformations
# ═══════════════════════════════════════════════════════════════

df_silver = (
    df

    # ── STEP 1: cast() — fix data types ──────────────────────────
    .withColumn("amount",    col("amount").cast(DoubleType()))
    .withColumn("txn_date",  col("txn_date").cast(DateType()))

    # ── STEP 2: filter() — quality gate ──────────────────────────
    .filter(col("amount").isNotNull())
    .filter(col("amount") > 0)
    .where(col("txn_id").isNotNull())

    # ── STEP 3: withColumn() — enrich data ───────────────────────
    .withColumn("risk_level",
        when(col("amount") > 500000, "HIGH")
        .when(col("amount") > 10000,  "MEDIUM")
        .otherwise("LOW")
    )
    .withColumn("is_international", col("country_code") != "IN")
    .withColumn("pipeline_version", lit("v2.0"))

    # ── STEP 4: drop() — remove raw/temp cols ────────────────────
    .drop("txn_type")    # already captured in risk_level

    # ── STEP 5: select() + alias() — final schema ────────────────
    .select(
        col("txn_id").alias("transaction_id"),
        col("account_id").alias("account_number"),
        col("amount").alias("txn_amount_inr"),
        col("txn_date").alias("transaction_date"),
        col("status").alias("txn_status"),
        col("country_code").alias("source_country"),
        col("risk_level"),
        col("is_international"),
        col("pipeline_version")
    )
)

# ── STEP 6: Action — this triggers all transformations ────────
df_silver.printSchema()
df_silver.show(truncate=False)
print(f"Silver rows: {df_silver.count()}")

PySpark — AML Screening Pipeline (Databricks) 
# ═══════════════════════════════════════════════════════════════
# AML Transaction Screening — Bronze to Silver
# Runs on: Databricks Runtime 13.x, Azure Databricks cluster
# Storage: ADLS Gen2 (abfss:// via Unity Catalog)
# ═══════════════════════════════════════════════════════════════

from pyspark.sql import SparkSession
from pyspark.sql.functions import (
    col, lit, when, round, upper, sha2,
    to_date, current_timestamp, concat_ws
)
from pyspark.sql.types import DoubleType, DateType, StringType

spark = SparkSession.builder.appName("AML_Screening").getOrCreate()

# ── READ from ADLS Bronze layer ────────────────────────────────
bronze_path = "abfss://bronze@bankadls.dfs.core.windows.net/transactions/"
df_raw = spark.read.format("parquet").load(bronze_path)

# ── TRANSFORMATION 1: cast() — enforce schema ─────────────────
df_typed = (df_raw
    .withColumn("amount",      col("amount").cast(DoubleType()))
    .withColumn("txn_date",    col("txn_date").cast(DateType()))
    .withColumn("account_id",  col("account_id").cast(StringType()))
)

# ── TRANSFORMATION 2: filter() — data quality gate ───────────
df_clean = (df_typed
    .filter(col("amount").isNotNull() & (col("amount") > 0))
    .filter(col("txn_id").isNotNull())
    .filter(col("txn_date").isNotNull())
)

# ── TRANSFORMATION 3: withColumn() — AML risk enrichment ─────
df_enriched = (df_clean
    .withColumn("aml_risk_score",
        when((col("amount") > 1000000) & (col("country_code") != "IN"), 0.98)
        .when((col("amount") > 500000) & (col("status") == "FLAGGED"),    0.85)
        .when(col("country_code").isin(["IR", "KP", "SY"]),                  0.99)
        .when(col("amount") > 100000,                                          0.45)
        .otherwise(0.10)
    )
    .withColumn("requires_sar",      col("aml_risk_score") > 0.80)
    .withColumn("is_cross_border",   col("country_code") != "IN")
    .withColumn("amount_usd",         round(col("amount") / 83.5, 2))
    .withColumn("processed_at",       current_timestamp())
    .withColumn("pipeline_id",        lit("AML-SCR-001"))
)

# ── TRANSFORMATION 4: drop() — remove raw/PII columns ────────
# Mask customer_name before storing in Silver
df_pii_masked = (df_enriched
    .withColumn("customer_hash", sha2(col("customer_name"), 256))
    .drop("customer_name")   # remove raw PII
)

# ── TRANSFORMATION 5: select() + alias() — final schema ──────
df_silver = df_pii_masked.select(
    col("txn_id").alias("transaction_id"),
    col("account_id").alias("account_number"),
    col("customer_hash").alias("customer_identifier"),
    col("amount").alias("txn_amount_inr"),
    col("amount_usd").alias("txn_amount_usd"),
    col("txn_date").alias("transaction_date"),
    col("status").alias("txn_status"),
    col("country_code").alias("source_country"),
    col("aml_risk_score"),
    col("requires_sar"),
    col("is_cross_border"),
    col("processed_at"),
    col("pipeline_id")
)

# ── WRITE to ADLS Silver as Delta ─────────────────────────────
silver_path = "abfss://silver@bankadls.dfs.core.windows.net/aml_transactions/"
(df_silver
    .write
    .format("delta")
    .mode("append")
    .partitionBy("transaction_date", "source_country")
    .save(silver_path)
)

🎤 Interview Questions & Model Answers

Q1 (Direct): "What is the difference between filter() and where() in PySpark?"
They are identical — where() is simply an alias for filter(). Both accept a Column expression or SQL string condition and return a new DataFrame with only the matching rows. I prefer filter() for Python-style code and where() when writing SQL-like queries for readability.
Q2 (Indirect): "In your last project, how did you handle schema enforcement when ingesting data from multiple source systems?"
In our HDFC credit card pipeline on Databricks, raw data from 3 banking systems had inconsistent types — amounts as strings, dates as integers. I used cast() inside withColumn() to enforce a canonical schema, combined with filter(col("amount").isNotNull()) to remove nulls that appear when cast fails. This gave us a consistent Silver layer schema regardless of source differences.
Q3 (Trick): "When would you use drop() over select() to remove columns?"
Use drop() when you want to keep most columns and remove just a few — it's shorter to write and expresses intent clearly. Use select() when you want only a small subset of columns, since listing what you want is cleaner than listing what you don't. Both achieve the same result but express different intent. Importantly, drop() is safe — it won't raise an error if the column doesn't exist, while select("nonexistent") will fail.
Q4 (Advanced): "Why is calling withColumn() 50 times bad for performance?"
In older Spark versions, each withColumn() adds a Project node to the logical plan. With 50 calls, you get 50 nested Project nodes, and Catalyst has to traverse and optimize all of them — this can cause plan analysis overhead and occasionally suboptimal codegen. The fix is to consolidate multiple column additions into a single select() call with all expressions, which produces one Project node that Tungsten can optimize as a whole.
Q5 (Conceptual): "Explain lazy evaluation with respect to filter() and select()."
When you call df.filter(...).select(...), Spark doesn't execute either immediately. It adds nodes to an unresolved logical plan. Only when an action like show() or write is called does Spark pass this plan to the Catalyst Optimizer, which may reorder operations — for example, pushing the filter closer to the data source. This is why writing df.select().filter() versus df.filter().select() often produces the same optimized physical plan.
Always wrap multi-condition filters in parentheses: (col("a") > 1) & (col("b") == "X"). Using Python's and/or instead of &/| is a classic interview trap — it raises an error because Column doesn't support Python boolean operators.

💡 Pro Tips

  • Filter early, select early: Always apply filter() and select() as early as possible in your pipeline. This reduces the data size that flows into expensive wide transformations like groupBy and join.
  • Chain transformations fluently: In production, use parentheses around a chained expression rather than creating intermediate DataFrames for every step. This keeps the logical plan unified and lets Catalyst see all operations together for better optimization.
  • Use col() over string literals: In production code, prefer col("colname") over the string "colname" because col() allows you to do arithmetic, call functions, and chain methods directly — it's a Column expression, not just a name reference.

🧠 Quiz Question 1

You have a DataFrame with 50 columns read from ADLS. You call:

df.filter(col("amount") > 1000).select("txn_id", "amount", "status").show()

Catalyst Optimizer changes this to read only 3 columns from the Parquet file, not all 50. What optimization technique is this?
A Predicate Pushdown
B Column Pruning (Projection Pruning)
C Whole-Stage Code Generation
D Partition Pruning
✅ Answer: B — Column Pruning. Catalyst sees your select("txn_id", "amount", "status") and instructs the Parquet reader to only read those 3 column byte ranges from ADLS — skipping the other 47. Predicate Pushdown (A) is when the filter condition is pushed to the scan; that also happens here, but the question specifically asks about the column count reduction.

🧠 Quiz Question 2

A junior engineer writes this code to find flagged AND high-value transactions:

df.filter(col("status") == "FLAGGED" and col("amount") > 500000).show()

What happens when this runs?
A Works correctly — filters both conditions
B Returns all rows — the filter is ignored
C Raises a ValueError or AnalysisException
D Only the first condition (status == FLAGGED) is applied
✅ Answer: C — Raises an error. Python's and operator does not work on Spark Column objects — it tries to evaluate the truthiness of a Column, which raises a ValueError. The correct operator is & (bitwise AND): (col("status") == "FLAGGED") & (col("amount") > 500000). Always use & (AND), | (OR), ~ (NOT) for Spark Column conditions.

🧠 Quiz Question 3

You cast a column: df.withColumn("amount", col("amount").cast("integer")). Some rows have the value "N/A" in the amount column. What does Spark do with those rows?
A Raises a runtime exception and stops the job
B Skips those rows from the output
C Sets those values to null silently
D Stores the string "N/A" as-is in the integer column
✅ Answer: C — Sets to null silently. Spark's cast() uses a safe cast behavior by default — if a value cannot be converted, it becomes null without raising an error. This is why you should always add a filter(col("amount").isNotNull()) after casting and log/monitor the null counts to catch data quality issues in production.
