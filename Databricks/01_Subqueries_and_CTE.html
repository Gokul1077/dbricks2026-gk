Prerequisites — What You Must Know First

Before we touch CTEs or Subqueries, make sure these building blocks are solid. Every advanced SQL pattern is built on top of these.

Basic SELECT
SELECT column1, column2
FROM   table_name
WHERE  condition;

Every query starts here. SELECT picks columns, FROM picks the table, WHERE filters rows.

Aggregations
SELECT dept_id,
       COUNT(*) AS emp_count,
       SUM(salary) AS total_sal
FROM   employees
GROUP BY dept_id;

GROUP BY collapses rows into groups. Aggregate functions work on each group.

JOINs — The Most Important Pre-req
JOIN TypeWhat it returnsWhen to use
INNER JOIN Only matching rows from both tables You need data from both tables
LEFT JOIN All rows from left + matching from right (NULL if no match) Keep all rows from left table, optional right side
FULL OUTER JOIN All rows from both, NULLs where no match Find data present in one but not the other
CROSS JOIN Every row × every row (cartesian product) Generating combinations (rare, expensive)

Why this matters: You cannot use a column alias from SELECT inside WHERE. The WHERE runs before SELECT!

NULL handling in Oracle
-- NVL replaces NULL with a default value
SELECT NVL(commission, 0) FROM employees;

-- NVL2: if not null → val1, if null → val2
SELECT NVL2(commission, 'Has Comm', 'No Comm') FROM employees;

-- COALESCE returns first non-NULL
SELECT COALESCE(phone, email, 'N/A') FROM customers;

-- NULL comparisons MUST use IS NULL, not = NULL
WHERE commission IS NULL

Subqueries — Query inside a Query

🔷 Concept Snapshot

A subquery (also called an inner query or nested query) is a SELECT statement placed inside another SQL statement. It runs first, and its result is used by the outer query.

Analogy: Like asking a colleague "Tell me which employees earn above average" — first you calculate the average (inner query), then you use that number to filter (outer query). You answer a smaller question first, then use that answer.
Types of Subqueries
TypeWhere it appearsReturnsExample use
Scalar SELECT or WHERE clause 1 row, 1 column Fetch max salary to compare
Row WHERE clause 1 row, multiple columns Compare multiple columns at once
Table (Inline View) FROM clause Multiple rows and columns Pre-aggregate then join
Correlated WHERE clause Depends on outer row Find Nth highest per group
1. Scalar Subquery — returns one value
-- Find all employees who earn above the company average
SELECT employee_id, first_name, salary
FROM   employees
WHERE  salary > (
           SELECT AVG(salary)
           FROM   employees      -- ← inner query runs first
       );
-- Inner query returns ONE number, e.g. 65000
-- Outer query uses that number in WHERE
2. Inline View (Table Subquery in FROM)
-- Find departments where average salary > 70000
SELECT dept_name, avg_sal
FROM (
    SELECT d.department_name  AS dept_name,
           AVG(e.salary)       AS avg_sal
    FROM   employees   e
    JOIN   departments d ON e.department_id = d.department_id
    GROUP BY d.department_name
) dept_summary                    -- ← give the subquery an alias
WHERE avg_sal > 70000;

The subquery in FROM creates a temporary table for the outer query to use. This is called an Inline View.

3. Correlated Subquery — references outer query
-- Find employees who earn MORE than the average of their OWN department
SELECT e1.employee_id, e1.first_name,
       e1.department_id, e1.salary
FROM   employees e1
WHERE  e1.salary > (
           SELECT AVG(e2.salary)
           FROM   employees e2
           WHERE  e2.department_id = e1.department_id  -- ← ties to outer row
       );

-- How it works:
--  For each outer row (e1), the inner query runs fresh
--  using e1.department_id. This is why it's "correlated" —
--  the inner query depends on the current outer row.
⚠️ Watch Out: Correlated subqueries run once per outer row — very slow on large tables. Use Window Functions instead when performance matters.
4. Subquery with IN / EXISTS
-- IN: Match against a list returned by subquery
SELECT first_name, department_id
FROM   employees
WHERE  department_id IN (
           SELECT department_id
           FROM   departments
           WHERE  location_id = 1700
       );

-- EXISTS: Check if subquery returns any rows (TRUE/FALSE)
SELECT d.department_name
FROM   departments d
WHERE  EXISTS (
           SELECT 1                     -- SELECT 1 is a convention
           FROM   employees e
           WHERE  e.department_id = d.department_id
       );
-- EXISTS stops as soon as one match is found → faster than IN
-- Use EXISTS when checking "does any related record exist?"

CTEs (Common Table Expressions) — WITH clause

🔷 Concept Snapshot

A CTE is a named temporary result set defined at the top of your query using the WITH keyword. It exists only during the execution of that query.

Analogy: Think of CTE as a sticky note you write before solving a complex math problem. You jot down intermediate results on sticky notes, then refer to them while solving the main problem. Each sticky note = one CTE block.
CTE Syntax — Step by Step
-- STRUCTURE of a CTE
WITH cte_name AS (          -- Step 1: name your CTE
    SELECT ...               -- Step 2: write the inner query
    FROM   ...
    WHERE  ...
)
SELECT *                     -- Step 3: use the CTE like a table
FROM   cte_name;

-- MULTIPLE CTEs (chain them with comma)
WITH
  cte_one AS (
      SELECT ... FROM table1
  ),
  cte_two AS (              -- cte_two can reference cte_one
      SELECT ... FROM cte_one
  )
SELECT * FROM cte_two;
Full Working Example
-- Find employees who earn above their department average
-- Using CTE (clean, readable)

WITH dept_avg AS (
    -- CTE: calculate avg salary per department
    SELECT department_id,
           AVG(salary) AS avg_salary
    FROM   employees
    GROUP BY department_id
)
-- Main query: join employees with their dept average
SELECT e.employee_id,
       e.first_name,
       e.salary,
       da.avg_salary,
       e.salary - da.avg_salary AS diff_from_avg
FROM   employees e
JOIN   dept_avg da ON e.department_id = da.department_id
WHERE  e.salary > da.avg_salary
ORDER BY diff_from_avg DESC;

Subquery vs CTE — When to Use Which

DimensionSubqueryCTE
Readability ❌ Hard to read when nested deeply ✅ Named blocks — reads like English
Reusability in same query ❌ Must repeat if used multiple times ✅ Define once, reference many times
Debugging ❌ Hard to isolate each step ✅ Run each CTE block independently
Performance in Oracle ✅ Optimizer usually inlines subqueries ✅ Oracle can materialize CTEs for heavy queries
Recursion ❌ Not supported ✅ Recursive CTEs supported

Best for Simple one-time filters or lookups Multi-step logic, readability, recursion
💡 Pro Tip: In Databricks / Spark SQL and Oracle both, CTEs are the preferred style for production queries. They make code review, debugging, and maintenance much easier. Use subqueries only for truly simple, one-off lookups.

Real-World Banking Example

Scenario: Flag High-Risk Customers

Business requirement: Find customers whose total transaction amount in the last 30 days exceeds ₹5,00,000 AND who have more than 2 failed login attempts. (Fraud screening pipeline.)

-- Using CTEs: clean, step-by-step, auditable

WITH

-- Step 1: Aggregate last 30 days transactions per customer
recent_txns AS (
    SELECT   customer_id,
             SUM(amount)   AS total_amount,
             COUNT(*)      AS txn_count
    FROM     transactions
    WHERE    txn_date >= SYSDATE - 30
    GROUP BY customer_id
),

-- Step 2: Find customers with failed logins > 2
failed_logins AS (
    SELECT   customer_id,
             COUNT(*) AS fail_count
    FROM     login_attempts
    WHERE    status     = 'FAILED'
    AND      attempt_dt >= SYSDATE - 30
    GROUP BY customer_id
    HAVING   COUNT(*) > 2
),

-- Step 3: High-value customers (amount > 500000)
high_value AS (
    SELECT customer_id, total_amount, txn_count
    FROM   recent_txns
    WHERE  total_amount > 500000
)

-- Final query: join all CTEs, get customer info
SELECT  c.customer_id,
        c.customer_name,
        c.mobile,
        hv.total_amount,
        hv.txn_count,
        fl.fail_count,
        'HIGH_RISK'      AS risk_flag,
        SYSDATE          AS flagged_on
FROM    customers   c
JOIN    high_value  hv ON c.customer_id = hv.customer_id
JOIN    failed_logins fl ON c.customer_id = fl.customer_id
ORDER BY hv.total_amount DESC;
💡 Why CTEs here? Each step is independent and testable. You can run just the recent_txns CTE to validate it before building the whole query. In production Databricks pipelines, each CTE maps to a logical transformation step.

Interview Insights — CTEs & Subqueries

Q1: "What's the difference between a CTE and a subquery?"
✅ A CTE is a named, reusable temporary result set defined using the WITH clause that makes complex queries modular and readable. A subquery is an unnamed inline query nested inside another query. CTEs can reference each other in a chain, support recursion, and are easier to debug. Subqueries are better for simple, one-time lookups. In production data pipelines — like a Databricks fraud detection job — we prefer CTEs for multi-step logic because they're easier to maintain and test.
Q2: "When would you use EXISTS instead of IN?"
✅ EXISTS is preferred when the subquery may return large result sets or when you just need to check existence rather than match values. EXISTS short-circuits — it stops once the first matching row is found. IN materializes the entire subquery result first and then does the comparison. EXISTS also handles NULLs better; if the subquery returns NULLs, IN can behave unexpectedly. For correlated existence checks on large tables, EXISTS is almost always the right choice.
Q3: "Can a CTE replace a subquery in all cases?"
✅ Mostly yes, but not always. CTEs can only be referenced in the same query block and don't persist beyond it. Scalar subqueries in the SELECT list can't always be directly replaced by a CTE without a JOIN. Also, in Oracle, the optimizer may choose different execution plans for a CTE vs. a subquery. CTEs are not stored objects — use views or materialized views if you need persistence across queries.
🚨 Common Mistakes:
  • Saying "CTE is faster than subquery" — not always true. Performance depends on the query plan, not the syntax.
  • Using ROWNUM < 10 without wrapping in a subquery — ROWNUM filters before ORDER BY in Oracle.
  • Treating CTE as a stored procedure — CTEs are not persistent, they live only during query execution.
🧠 Quiz: What will the following Oracle query return?
SELECT * FROM employees WHERE salary > (SELECT AVG(salary) FROM employees WHERE department_id = 10) AND department_id != 10
  • A) Employees in department 10 who earn above dept 10 avg
  • B) Employees NOT in dept 10 who earn above dept 10 avg
  • C) All employees earning above company average
  • D) Employees in dept 10 only
✅ Answer: B
The inner query calculates the average salary only for department 10. The outer WHERE then filters employees who earn MORE than that average AND are NOT in department 10. The two conditions work independently — the scalar subquery computes dept-10 avg, and the AND filter excludes dept 10 from results.
🎯 What are "FAANG Patterns"?

These are recurring query structures that appear again and again in data engineering interviews at companies like Facebook, Amazon, Apple, Netflix, Google — and top Indian tech/banking companies too. They're not exotic SQL — they're combinations of simple concepts used cleverly.

Analogy: In chess, the basic pieces are simple. But winning requires patterns — the fork, pin, skewer. SQL patterns are the same: the same keywords arranged in specific structures to solve common data problems.
Window Functions — The Foundation of FAANG Patterns

Before the patterns, you must understand Window Functions. They are the engine behind most FAANG-level SQL solutions.

-- SYNTAX of a Window Function
function_name() OVER (
    PARTITION BY column1        -- group rows (like GROUP BY, but doesn't collapse)
    ORDER BY     column2         -- order within each partition
    ROWS BETWEEN ... AND ...    -- optional: define the window frame
)

-- Key Window Functions you MUST know:
-- ROW_NUMBER()  → unique rank (no ties)
-- RANK()        → rank with gaps on tie
-- DENSE_RANK()  → rank without gaps on tie
-- LAG()         → value from previous row
-- LEAD()        → value from next row
-- SUM() OVER()  → running total
-- NTILE(n)      → divide into n buckets
Analogy: Window functions are like looking through a sliding window on a train — you see the current seat AND the context around it (past and future rows), without leaving your seat. GROUP BY collapses rows; Window functions keep all rows but add calculated context.
P1
Ranking Within a Group
ROW_NUMBER / RANK / DENSE_RANK
🤔 Why this pattern? Find Nth highest salary, top 3 products per category, latest transaction per customer — all require ranking within a group. 
-- PROBLEM: Find the 2nd highest salary in each department

WITH ranked AS (
    SELECT
        employee_id,
        department_id,
        salary,
        DENSE_RANK() OVER (
            PARTITION BY department_id    -- rank WITHIN each dept
            ORDER BY     salary DESC      -- highest salary first
        ) AS sal_rank
    FROM employees
)
SELECT * FROM ranked
WHERE  sal_rank = 2;   -- pick rank 2

-- ROW_NUMBER vs RANK vs DENSE_RANK when salaries tie at 5000,5000:
-- ROW_NUMBER:  1,2,3      (arbitrary tiebreak — each row unique)
-- RANK:        1,1,3      (skip rank 2 — tied rows share 1)
-- DENSE_RANK:  1,1,2      (no gap — tied rows share 1, next is 2)
💡 Interview Tip: Always clarify with the interviewer — "What should happen if two employees have the same salary?" The answer determines whether you use RANK or DENSE_RANK.
P2
Running Totals & Moving Averages
SUM() OVER with frame clauses
🤔 Why this pattern? Cumulative sales, rolling 7-day average, YTD metrics — all need running calculations without losing row-level detail. 
-- Running total of transaction amounts per customer
SELECT
    customer_id,
    txn_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY customer_id
        ORDER BY     txn_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS running_total,

    -- 3-day moving average
    AVG(amount) OVER (
        PARTITION BY customer_id
        ORDER BY     txn_date
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW  -- 3 rows window
    ) AS moving_avg_3day
FROM transactions
ORDER BY customer_id, txn_date;

-- ROWS vs RANGE: use ROWS for physical row positions,
-- RANGE for value-based windows (e.g., same date values)
P3
Gaps & Islands
Consecutive sequences, detecting breaks
🤔 Why this pattern? Find consecutive login days, detect gaps in sequential IDs, identify unbroken streaks of activity — common in banking audit trails and user engagement analytics. 
-- PROBLEM: Find streaks of consecutive login days per user

WITH
-- Step 1: For each login, calculate lag (previous login date)
daily_logins AS (
    SELECT DISTINCT
        user_id,
        TRUNC(login_time) AS login_date
    FROM user_sessions
),

-- Step 2: Detect if current row continues the streak
with_gap_flag AS (
    SELECT
        user_id,
        login_date,
        LAG(login_date) OVER (
            PARTITION BY user_id
            ORDER BY login_date
        ) AS prev_date
    FROM daily_logins
),

-- Step 3: Assign group ID — new group when gap > 1 day
with_group AS (
    SELECT
        user_id,
        login_date,
        SUM(
            CASE WHEN login_date - prev_date > 1
                        OR prev_date IS NULL
                 THEN 1 ELSE 0 END
        ) OVER (
            PARTITION BY user_id ORDER BY login_date
        ) AS streak_grp
    FROM with_gap_flag
)

-- Step 4: Count length of each streak
SELECT
    user_id,
    MIN(login_date) AS streak_start,
    MAX(login_date) AS streak_end,
    COUNT(*) AS streak_days
FROM   with_group
GROUP BY user_id, streak_grp
ORDER BY streak_days DESC;
P4
Self Join — Compare Rows Within Same Table
Manager-employee, before-after comparisons
🤔 Why this pattern? When you need to compare a row against other rows in the same table — find employees who earn more than their manager, find duplicate records, or detect YoY growth per row. 
-- PROBLEM: Find employees earning MORE than their manager
SELECT
    e.employee_id,
    e.first_name   AS employee_name,
    e.salary       AS emp_salary,
    m.first_name   AS manager_name,
    m.salary       AS mgr_salary
FROM   employees e
JOIN   employees m         -- same table, different alias
  ON   e.manager_id = m.employee_id
WHERE  e.salary > m.salary;

-- PROBLEM: Find duplicate emails in customers table
SELECT DISTINCT a.email
FROM   customers a
JOIN   customers b
  ON   a.email        = b.email
AND    a.customer_id  != b.customer_id;  -- different rows, same email
P5
PIVOT / Conditional Aggregation
Rotating rows to columns
🤔 Why this pattern? Monthly sales reports, crosstab analysis, turning vertical data horizontal — extremely common in BI and reporting queries. 
-- METHOD 1: CASE WHEN (works everywhere, portable)
SELECT
    product_id,
    SUM(CASE WHEN month_num = 1 THEN amount ELSE 0 END) AS jan_sales,
    SUM(CASE WHEN month_num = 2 THEN amount ELSE 0 END) AS feb_sales,
    SUM(CASE WHEN month_num = 3 THEN amount ELSE 0 END) AS mar_sales
FROM   sales
GROUP BY product_id;

-- METHOD 2: Oracle PIVOT operator (Oracle 11g+)
SELECT *
FROM   sales
PIVOT (
    SUM(amount)
    FOR month_num IN (1 AS jan, 2 AS feb, 3 AS mar)
);

-- UNPIVOT: reverse — convert columns back to rows
SELECT product_id, month_name, sale_amount
FROM   sales_wide
UNPIVOT (sale_amount FOR month_name IN (jan, feb, mar));
P6
Top-N Per Group (Most Common FAANG Question)
Latest record per ID, top products per category
🤔 Why this pattern? Get the latest transaction per account, top 3 selling products per category, most recent address per customer — this is one of the most frequently asked SQL patterns in any interview. 
-- PROBLEM: Get the latest transaction for each account

WITH ranked_txns AS (
    SELECT
        account_id,
        txn_id,
        amount,
        txn_date,
        ROW_NUMBER() OVER (
            PARTITION BY account_id      -- group per account
            ORDER BY     txn_date DESC  -- latest first
        ) AS rn
    FROM transactions
)
SELECT account_id, txn_id, amount, txn_date
FROM   ranked_txns
WHERE  rn = 1;   -- only the latest (rank 1)

-- For Top 3: WHERE rn <= 3
-- For bottom 1: ORDER BY txn_date ASC and WHERE rn = 1
💡 The Golden Rule: CTE + ROW_NUMBER() + WHERE rn = N is the universal template for Top-N per group. Memorize this pattern cold.
P7
Recursive CTE — Hierarchical Data
Org charts, category trees, bill of materials
🤔 Why this pattern? Some data is naturally hierarchical — org charts (employee → manager), product categories (parent → child), geographic hierarchies (country → state → city). Recursive CTEs traverse these trees without needing application code. 
-- PROBLEM: Find all employees under a given manager (full org tree)
-- Table: employees(employee_id, name, manager_id)

WITH RECURSIVE org_tree AS (     -- In Oracle: just WITH (implicit recursive)

    -- ANCHOR: start with the top manager (CEO / root)
    SELECT employee_id, name, manager_id,
           1 AS level_num,
           name AS path
    FROM   employees
    WHERE  manager_id IS NULL          -- root node (no manager)

    UNION ALL

    -- RECURSIVE: join each employee with their manager (already in CTE)
    SELECT e.employee_id, e.name, e.manager_id,
           ot.level_num + 1,
           ot.path || ' → ' || e.name
    FROM   employees e
    JOIN   org_tree   ot ON e.manager_id = ot.employee_id
)
SELECT level_num, name, path
FROM   org_tree
ORDER BY level_num, name;

-- Oracle alternative: CONNECT BY (Oracle-specific, simpler syntax)
SELECT  employee_id, name,
        LEVEL,
        SYS_CONNECT_BY_PATH(name, ' → ') AS hierarchy
FROM    employees
START WITH  manager_id IS NULL         -- root
CONNECT BY  PRIOR employee_id = manager_id  -- parent-child link
ORDER BY LEVEL;
Analogy: Recursive CTE is like peeling an onion — each recursion peels one layer (one level of the hierarchy) until there are no more layers left.

Interview Insights — FAANG Patterns

Q1: "How would you find the top 3 products by sales for each category?"
✅ I'd use the Top-N per Group pattern: first write a CTE that applies ROW_NUMBER() OVER (PARTITION BY category_id ORDER BY sales DESC) to rank products within each category. Then in the outer query, filter WHERE rn <= 3. If ties need to be preserved equally, I'd use DENSE_RANK instead of ROW_NUMBER, since ROW_NUMBER breaks ties arbitrarily.
Q2: "What's the difference between RANK and DENSE_RANK? Give a scenario where the choice matters."
✅ Both rank rows within a partition, but differ on tied values. RANK skips the next rank after a tie (1,1,3) while DENSE_RANK doesn't (1,1,2). Practical scenario: if I'm finding the "2nd highest salary" using DENSE_RANK and two people share rank 1, rank 2 is the true second-highest value. With RANK, rank 2 would be skipped and there might be no results if I filter WHERE rank = 2. For finding "Nth unique value," always use DENSE_RANK.
Q3 (Trick): "Can you write a query to delete duplicate rows, keeping only the most recent?"
✅ Yes — use ROW_NUMBER() to rank duplicates within the group (same key columns, ORDER BY created_date DESC), then delete all rows WHERE rn > 1. In Oracle: DELETE FROM table WHERE rowid IN (SELECT rowid FROM (SELECT rowid, ROW_NUMBER() OVER (PARTITION BY dup_key ORDER BY created_date DESC) rn FROM table) WHERE rn > 1).
🚨 Common Mistakes in FAANG Interviews:
  • Using GROUP BY when you should use PARTITION BY — GROUP BY collapses rows, PARTITION BY doesn't.
  • Forgetting to wrap window functions in a CTE before filtering on them — you cannot use window function alias in WHERE of the same query.
  • Using ROWNUM directly without a subquery for pagination — ROWNUM runs before ORDER BY in Oracle.
  • Not handling NULLs in LAG/LEAD — use the optional default value parameter: LAG(col, 1, 0)
💡 Pro Tips for SQL Interviews:
  • Always think aloud — explain your approach before writing. Interviewers value structured thinking over fast typing.
  • Start with the simplest possible version, then optimize. Say: "First let me solve it with a subquery, then refactor to a CTE."
  • Ask about edge cases: NULLs, ties, empty groups — it shows production thinking, not just textbook knowledge.
🧠 Quiz: A table has (user_id, login_date). How many CTEs do you need to find streaks of consecutive login days?
  • A) 1 — just use GROUP BY on login_date
  • B) 0 — a single window function handles it directly
  • C) 3 — dedup dates, detect gaps with LAG, assign group IDs, then count
  • D) You need a stored procedure, not CTEs
✅ Answer: C
The Gaps & Islands pattern typically needs 3 steps: (1) deduplicate to one row per user per day, (2) use LAG() to detect gaps in dates, (3) use a running SUM to assign a group ID to each streak, and then GROUP BY user + streak_group to count the length. This is a classic multi-CTE pattern.
