Concept Snapshot

Apache Spark is a distributed data processing framework that splits large datasets into chunks, distributes them across a cluster of machines, and processes them in parallel — all in-memory (RAM), which makes it dramatically faster than older tools like Hadoop MapReduce.

🍕 Analogy: Think of Spark like a pizza chain with multiple branches. The Driver is the head office that takes your order and decides which branch makes which part. The Executors are the branch kitchens that actually bake the pizzas in parallel. The Cluster Manager decides how many ovens (resources) each branch gets.

Why Does Spark Exist? (The Problem It Solves)

Before Spark, Hadoop MapReduce had two major pain points:

  • Disk I/O Pain: Every intermediate result (after map, shuffle, reduce) was written to disk. Disk reads/writes are extremely slow — making iterative algorithms like ML training painfully slow.
  • Rigid 1-job-per-shuffle model: MapReduce had exactly one map + one reduce phase per job. Complex pipelines (e.g., filter → join on key A → aggregate on key B) required multiple separate jobs, each reading/writing disk.
✅ Spark's Fix: (1) Intermediate results stay in RAM of worker nodes, not disk. (2) Spark's DAG model handles multiple stages with different grouping keys in a single application — no separate jobs needed.

Core Components Deep Dive

1. Spark Cluster

A collection of machines (nodes) — either in a public cloud (Azure) or on-premise data center — with Spark installed. Every cluster has at least one Driver and one or more Workers.

2. Cluster Manager

A software tool that allocates CPU, memory, and other resources across applications. Think of it as the HR department assigning desks to employees.

Cluster ManagerBest ForNotes
StandaloneDev/testing, simple setupsBuilt into Spark, Master can also be a Worker
Hadoop YARNExisting Hadoop ecosystemsShares cluster with MapReduce, Hive, etc.
KubernetesContainerized, cloud-native workloadsBest for Azure AKS-based Spark
Apache MesosLegacy enterprise clustersLess common now
📌 Databricks Note: In Databricks on Azure, cluster management is handled automatically. You never configure YARN or Standalone directly — Databricks manages it for you.

3. Master Node

The machine that hosts the Cluster Manager. In Standalone mode, it also acts as a Worker. It orchestrates which worker nodes run which executors.

⚠️ Clarification from your notes: The node where the Driver JVM runs is NOT necessarily the Master Node. In client mode, the driver runs on your local machine. In cluster mode, it runs on one of the cluster nodes (but it's still separate from the cluster manager master).

4. Spark Worker

A node in the cluster that receives instructions from the Master and launches Executors on behalf of the Driver.

Key Role: The Worker's only job is to launch Executors. It doesn't process data itself — Executors do that.

5. Spark Executor

A JVM container launched on a Worker node with an allocated amount of cores (threads) and memory. This is where actual data processing happens.

  • Executes Spark tasks assigned by the Driver
  • Stores and caches data partitions in memory (for reuse across stages)
  • Writes intermediate shuffle results to local disk of the worker node
  • Reports results back to the Driver
Executor Core = Logical Thread, NOT a physical CPU core. An executor with 4 cores can run 4 tasks simultaneously. The OS then maps these JVM threads to physical cores.

6. Spark Driver Program vs Driver Process

📄 Driver Program

  • The code you write
  • Contains main() or notebook cells
  • Contains SparkSession creation
  • All transformation + action logic
  • Example: your PySpark script

🖥️ Driver Process

  • The runtime JVM process running that code
  • Contains DAG Scheduler
  • Contains Task Scheduler
  • Communicates with Cluster Manager
  • Collects results from executors
Simple way to remember: Driver Program = your code. Driver Process = the JVM that runs your code. One is the recipe; the other is the chef following the recipe.

7. SparkSession

The single entry point to all Spark functionality. It lives only on the Driver. Executors never have a SparkSession.

from pyspark.sql import SparkSession

# Created ONCE in the driver process
spark = SparkSession.builder \
    .appName("BankingFraudDetection") \
    .config("spark.sql.shuffle.partitions", "200") \
    .getOrCreate()

# SparkSession gives you access to:
spark.sql(...)          # SQL engine
spark.read.parquet(...) # DataFrame API
spark.catalog.listTables() # Catalog
spark.sparkContext      # Underlying SparkContext

Lazy Evaluation

Spark does not execute any code immediately when you write a transformation. It builds up a plan (the logical plan / DAG) and waits. Execution only happens when an action is called.

📚 Analogy: You're telling a chef your entire meal order — "first make soup, then salad, then pasta." The chef listens, optimizes the sequence (maybe prep common ingredients once), and only starts cooking when you say "serve it!" That "serve it" is your action.

How Lazy Evaluation Works Step by Step

# Each line below DOES NOT execute immediately
df = spark.read.parquet("/mnt/adls/transactions/")  # ← builds logical plan node
df_filtered = df.filter(df.amount > 10000)             # ← adds Filter node to plan
df_grouped  = df_filtered.groupBy("country")            # ← adds GroupBy node (shuffle!)
result      = df_grouped.agg({"amount": "sum"})         # ← adds Aggregate node

# 🔴 Nothing has run yet! Spark is just collecting the recipe.

# ✅ THIS triggers actual execution:
result.show()   # ← ACTION → Spark now optimizes + executes the full plan

Benefits of Lazy Evaluation

  • Whole-plan optimization: Catalyst Optimizer sees the full plan before running — it can apply predicate pushdown, column pruning, and join reordering across all steps at once.
  • Avoids unnecessary computation: If you filter before groupBy, Spark processes fewer rows in the shuffle.
  • Pipeline fusion: Multiple narrow transformations can be collapsed into a single stage — no intermediate materialization.
Predicate Pushdown: If you write df.filter(status == "COMPLETED").join(customers), Catalyst moves the filter as close to the data source as possible — even pushing it into the Parquet file reader so that only matching rows are loaded from ADLS. This saves massive I/O.
Projection Pruning (Column Pruning): If your final result only uses 3 columns out of 50, Catalyst removes the unused 47 columns early — reducing memory and shuffle costs.

Transformations vs Actions

Every Spark operation is either a Transformation (lazy — builds the plan) or an Action (eager — triggers execution).

AspectTransformationAction
ExecutionLazy — does NOT run immediatelyEager — triggers full DAG execution
ReturnsA new DataFrame / RDDA result (value, file write, display)
Effect on planAdds a node to the logical planFinalizes plan → optimizes → runs
Creates a Job?NoYes — one action = one Job

Narrow vs Wide Transformations (Critical for Stages)

TypeDefinitionShuffle Needed?Stage ImpactExamples
NarrowEach input partition maps to exactly one output partition — no data moves across the network❌ NoSame stagefilter(), map(), select(), withColumn(), union()
WideInput partitions must be redistributed across the network — multiple input partitions contribute to one output partition✅ YesNew stage boundarygroupBy(), join(), reduceByKey(), distinct(), repartition()
Why Wide Transformations Create New Stages: When data needs to be reshuffled across the network (e.g., all rows with the same country key must go to the same executor for groupBy), Spark cannot do this within a single stage. It must write intermediate results to the worker's local disk, then have the next stage pull the relevant data. This boundary = a shuffle boundary = a new stage.

DAG — Directed Acyclic Graph

A DAG is Spark's execution blueprint. Nodes = RDDs/DataFrames, Edges = transformations, Shuffle boundaries = Stage breaks.

Catalyst Optimizer — Key Optimizations

OptimizationWhat It DoesPerformance Gain
Predicate PushdownMoves filters as close to the data source as possible — even into the file readerReads less data from storage → huge I/O savings
Column PruningRemoves unused columns from the execution planLess memory, less shuffle data
Constant FoldingPre-computes constant expressions at plan timeAvoids repeated runtime computation
Join ReorderingPuts smaller tables first in join orderReduces intermediate dataset sizes

Tungsten Engine

After Catalyst produces the optimized logical plan, Tungsten handles the physical execution engine improvements:

  • Whole-Stage Code Generation (WSCG): Fuses multiple operators into a single compiled Java function — eliminates virtual function calls between operators.
  • Off-heap memory management: Manages memory outside the JVM heap to avoid garbage collection pauses.
  • Cache-friendly data structures: Stores data in columnar, binary format for CPU cache efficiency.

DAG Scheduler → Task Scheduler → Executors

ComponentResponsibility
DAG SchedulerSplits the DAG into Stages at shuffle boundaries. Determines task dependencies across stages. Submits stages to Task Scheduler in order (stage N+1 waits for stage N's shuffle output).
Task SchedulerTakes each stage and creates Tasks (1 task per partition). Schedules tasks on executors based on data locality, executor availability, and resource constraints.
ExecutorsRun the tasks in parallel. Each core (thread) runs one task at a time. Return results to Driver.
Task Dependency Rule: Within a stage, all tasks are independent and run in parallel. Across stages, tasks in stage 2 must wait for all stage 1 tasks to complete their shuffle write before stage 2 can start.

Full Job Execution Flow — Step by Step

Let's trace exactly what happens when you run a Spark job with a 1TB file, 12 worker nodes, and 8 cores each:

Phase 1: Planning (Driver, No Data Moves)

  1. Driver JVM reads the PySpark script and creates a SparkSession.
  2. Driver incrementally builds an Unresolved Logical Plan as each transformation is encountered. No data is read yet — pure lazy evaluation.
  3. When an Action is called, Spark finalizes the plan.
  4. Catalyst Optimizer converts: Unresolved → Resolved → Optimized Logical Plan.
  5. Physical Planner creates multiple candidate physical plans; Cost Optimizer picks the best one.
  6. Tungsten generates optimized Java bytecode.
  7. DAG Scheduler materializes the final RDD lineage DAG and splits it into Stages at shuffle boundaries.

Phase 2: Stage 1 Execution

  1. With 12 workers × 8 cores = 96 parallel task slots.
  2. 1TB file → 8,000 input partitions (assuming ~128MB blocks) → 8,000 tasks created and queued in Task Scheduler.
  3. First 96 tasks dispatched to 96 executor cores simultaneously.
  4. Each task streams only its required byte range from ADLS (Azure Data Lake Storage) — Spark does not copy the full 1TB into worker memory upfront.
  5. Each task processes its partition (filter, map, etc.), stores intermediate results on the local disk of the worker node (shuffle write).
  6. As cores free up, the next batch of tasks from the queue is dispatched — until all 8,000 Stage 1 tasks complete.

Phase 3: Shuffle (Between Stages)

Shuffle is the most expensive operation in Spark — data must move across the network from Stage 1 worker disks to Stage 2 executors.

shuffle.partitions setting: Stage 2's number of tasks is determined by spark.sql.shuffle.partitions, default = 200. This is independent of the input file count. So Stage 2 creates 200 tasks regardless of how many Stage 1 tasks ran. Tune this setting for your cluster size!

Phase 4: Stage 2 Execution

  1. 200 new tasks created for Stage 2 (e.g., groupBy + aggregation).
  2. Each Stage 2 task pulls relevant shuffle data from the worker node disks (Stage 1 output).
  3. Performs final aggregation (groupBy + sum).
  4. Writes final results as 200 Parquet files to the output directory on ADLS.

Block vs Partition — They Are NOT the Same

AspectBlock (HDFS/ADLS)Partition (Spark)
NaturePhysical storage unitLogical processing unit
When it existsBefore Spark runs — on diskCreated when Spark reads data or during transformations
PurposeHow data is storedHow data is processed

Managed byHDFS / ADLS / storage layerSpark engine
Relationship: By default, Spark creates approximately one partition per HDFS block when reading. But you can control this with spark.sql.files.maxPartitionBytes (default 128 MB) or by calling repartition() / coalesce(). Check partition count with df.rdd.getNumPartitions().

Parallelism Formula

Spark Parallelism = min(number of partitions, number of available cores)

Parallelism is NOT determined by: data size, node count, executor memory, or CPU speed.
More partitions → more tasks. More cores → more concurrent task execution.

Real-World Banking Production Example

Scenario: Daily fraud detection pipeline — process 5 billion transaction records from Azure Data Lake, join with customer data, filter suspicious transactions, aggregate fraud patterns by country, and write results to Azure Synapse Analytics.

Pipeline Implementation

from pyspark.sql import SparkSession
from pyspark.sql import functions as F

# ─── DRIVER: SparkSession created (runs on Driver JVM only) ───
spark = SparkSession.builder \
    .appName("FraudDetectionPipeline") \
    .config("spark.sql.shuffle.partitions", "400")  # tuned for cluster size \
    .config("spark.sql.files.maxPartitionBytes", "134217728")  # 128MB \
    .getOrCreate()

# ─── LAZY: Builds logical plan. No data read yet. ───

# Narrow Transformation 1: Read from ADLS Gen2
transactions = spark.read.parquet(
    "abfss://raw@bankadls.dfs.core.windows.net/transactions/2024/"
)

# Narrow Transformation 2: Filter (Catalyst will pushdown into Parquet reader)
suspicious = transactions.filter(
    (F.col("amount") > 50000) &
    (F.col("status") == "FLAGGED")
)

# Narrow Transformation 3: Column pruning
suspicious = suspicious.select("txn_id", "customer_id", "country", "amount")

# Read customer reference data (smaller dataset → broadcast join candidate)
customers = spark.read.parquet(
    "abfss://ref@bankadls.dfs.core.windows.net/customers/"
)

# Wide Transformation: JOIN → triggers shuffle → new Stage boundary
# Spark Catalyst will automatically use BroadcastHashJoin if customers is small
enriched = suspicious.join(
    F.broadcast(customers),  # forces broadcast — avoids shuffle for join!
    on="customer_id",
    how="left"
)

# Wide Transformation: groupBy → triggers shuffle → new Stage boundary
fraud_summary = enriched.groupBy("country", "risk_tier") \
    .agg(
        F.count("txn_id").alias("fraud_count"),
        F.sum("amount").alias("total_fraud_amount"),
        F.avg("amount").alias("avg_fraud_amount")
    )

# ─── ACTION: THIS triggers full DAG execution ───
# Spark now: Optimizes plan → creates DAG → splits stages →
# creates tasks → dispatches to 96 executor cores → collects results
fraud_summary.write \
    .mode("overwrite") \
    .format("delta") \
    .save("abfss://gold@bankadls.dfs.core.windows.net/fraud_summary/")

# ─── DAG breakdown for this job: ───
# Stage 1: Read parquet + filter (Narrow) — 40,000 tasks (5B rows / 128MB)
# Stage 2: groupBy + agg (Wide) — 400 tasks (spark.sql.shuffle.partitions=400)
# Broadcast join does NOT create a new stage — data is sent to each executor

What Happens Internally

  1. Stage 1 (40,000 tasks): Each executor core streams its 128MB partition from ADLS, applies filter + select (narrow — same stage), writes shuffle data to local disk.
  2. Shuffle: Stage 2 tasks pull relevant country-grouped data from all Stage 1 worker disks across the network.
  3. Stage 2 (400 tasks): Each task aggregates its country partition, writes 400 Parquet files to ADLS Gold zone as Delta Lake format.
  4. Result: Azure Data Factory then triggers a pipeline to load from Delta Lake into Azure Synapse Analytics SQL Pool for reporting.

Interview Insights

Explain Spark architecture. What are the main components?
"Spark follows a master-worker architecture. The Driver is the JVM process that runs your application code — it contains the SparkSession, DAG Scheduler, and Task Scheduler. It doesn't process data itself; it orchestrates execution. The Cluster Manager (YARN, Kubernetes, or Standalone) allocates CPU and memory resources. Worker Nodes host Executors — JVM containers that actually process data partitions in parallel. In my banking pipeline on Databricks with Azure, we had 12 worker nodes × 8 cores each = 96 parallel task slots, processing 1TB of transaction data across 8,000 tasks."
What is lazy evaluation in Spark and why does it matter?
"Spark doesn't execute transformations immediately — it builds an unresolved logical plan as you add transformations. Only when an action like show() or write() is called does Spark finalize the plan, pass it through the Catalyst Optimizer, and execute it. This matters because Catalyst can see the full plan at once — it can push filters down to the data source to read less data, remove unused columns early, and reorder joins. In production this reduced our ADLS read costs by over 60% through predicate pushdown on Parquet files."
What's the difference between a narrow and wide transformation?
"A narrow transformation keeps each input partition's data on the same executor — no data crosses the network. Examples: filter, select, withColumn. A wide transformation requires data redistribution (shuffle) across the network because rows with the same key must end up on the same executor. Examples: groupBy, join, distinct. Wide transformations create new stage boundaries in the DAG — Stage N must fully complete its shuffle write before Stage N+1 can begin. In practice, minimizing shuffles by filtering early and using broadcast joins for small tables dramatically improves performance."
A job runs fine locally but is very slow on a cluster. What would you check?
"First, I'd check partition count — too few partitions means most executor cores are idle. Run df.rdd.getNumPartitions(). Second, I'd check for data skew in shuffle stages — one task taking 10x longer than others. Third, I'd look at spark.sql.shuffle.partitions — the default 200 may be too high (small cluster) or too low (large data). Fourth, I'd check if large table joins could use broadcast joins to avoid shuffles entirely. Finally, I'd check executor memory for spill-to-disk warnings in the Spark UI."
What happens when you call collect() on a large DataFrame?
"Calling collect() triggers the full DAG execution and then pulls all data from every executor back to the Driver JVM. If the DataFrame has millions of rows, this will likely cause an OutOfMemoryError on the driver. In production, you should never use collect() on large datasets. Use write() to save results to ADLS, or use take(n)/show(n) for sampling. I once caught a colleague's code that was collecting 10GB to the driver — we replaced it with a write to Delta Lake."

🚨 Common Mistakes Candidates Make

  • Saying "the Driver processes data" — the Driver only orchestrates; Executors process data.
  • Confusing "Cluster Manager" (software) with "Master Node" (machine). They're related but distinct.
  • Saying transformations run immediately — they're always lazy until an action is called.
  • Not knowing that spark.sql.shuffle.partitions controls Stage 2+ task count, not input partitions.
  • Saying a join always creates a new stage — broadcast joins are narrow and don't shuffle.
  • Confusing HDFS Blocks (physical storage) with Spark Partitions (logical processing units).

Pro Tips

⚡
Tune shuffle.partitions for your cluster: The default spark.sql.shuffle.partitions=200 is rarely optimal. For a 12-node × 8-core cluster (96 slots), use 96–192 shuffle partitions. Too many = small files + overhead. Too few = underutilized cores during Stage 2. In Databricks, enable Adaptive Query Execution (AQE) — it auto-tunes shuffle partition count at runtime based on actual data statistics.
🔍
Always use the Spark UI to diagnose slow jobs: In Databricks, the Spark UI shows the DAG visualization, stage timelines, and task-level metrics. Look for: (1) skewed tasks (one task 10× longer than others = data skew), (2) large shuffle read/write sizes (reduce with repartitioning before groupBy), (3) GC time > 10% of task time (executor memory too low). Most production performance issues are visible in the Spark UI within 5 minutes.
📦
Broadcast joins can eliminate an entire stage: If one side of a join is less than spark.sql.autoBroadcastJoinThreshold (default 10MB), Spark automatically broadcasts it. You can force it with F.broadcast(df). This converts a wide transformation (which causes a shuffle and a new stage) into a narrow transformation — no data moves across the network. In banking workloads where we join large transaction tables with small reference tables (currency codes, country mappings), broadcast joins are always the right choice.

Quiz Question

Scenario: A Databricks job processes a 2TB transaction file on a cluster of 10 worker nodes, each with 8 executor cores. The PySpark code has this plan: read() → filter() → withColumn() → groupBy("country").agg(sum()) → write(). The default spark.sql.shuffle.partitions=200 is unchanged. How many Stages will the DAG have, and what determines the number of tasks in Stage 2?
  • A) 1 Stage total; Stage 2 doesn't exist because filter() removes most data
  • B) 2 Stages; Stage 2 has 80 tasks (10 nodes × 8 cores)
  • C) 2 Stages; Stage 2 has 200 tasks (spark.sql.shuffle.partitions default)
  • D) 3 Stages; groupBy and agg are separate wide transformations creating 2 boundaries
✅ Correct Answer: C

The DAG has 2 Stages. Stage 1 covers all narrow transformations: read → filter → withColumn — these all operate on the same partitions with no data movement. The groupBy() is a wide transformation that triggers a shuffle, creating a stage boundary between Stage 1 and Stage 2. Stage 2 task count is not based on input partitions or core count — it is determined by spark.sql.shuffle.partitions, which defaults to 200. So Stage 2 creates exactly 200 tasks, each pulling shuffled data from Stage 1 worker disks. Option B is a common mistake — core count controls parallelism (how many tasks run at once), not how many tasks are created.
