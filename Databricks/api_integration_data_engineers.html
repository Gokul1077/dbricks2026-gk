Prerequisites — What You Must Know First

🔷 Why Does a Data Engineer Need APIs?
Core Concept

APIs (Application Programming Interfaces) are the primary way external systems expose data. As a Data Engineer, you will routinely pull data from APIs (bank transactions, payment gateways, CRM systems, weather feeds, market data) and land it into ADLS / Delta Lake for further processing.

Analogy: An API is like a waiter at a restaurant. You (the client) tell the waiter (API) what you want. The waiter goes to the kitchen (server/database) and brings back what you asked for. You never enter the kitchen directly.

🧠 GET Pattern — How to Remember

requests.get(url, params={}, headers={}, timeout=N)

Mnemonic: "GET P-H-T" → Params, Headers, Timeout

⚡ Sessions — Reuse Connections for Performance
💡

Production tip: Always use requests.Session() when making multiple calls to the same API. It reuses the TCP connection (keep-alive), reducing latency by ~40% and avoiding re-sending headers each time.

      

        

        Python — Session Pattern (Production Best Practice)
      

      import requests

# ── Session Pattern ──
# Set headers ONCE, reused for every call
with requests.Session() as session:
    session.headers.update({
        "Authorization": f"Bearer {token}",
        "Accept": "application/json"
    })
    session.headers.update({"X-Client-ID": "databricks-pipeline-v2"})

    # All calls reuse the session (TCP connection pooling)
    r1 = session.get(f"{BASE_URL}/transactions", timeout=30)
    r2 = session.get(f"{BASE_URL}/accounts", timeout=30)
    r3 = session.get(f"{BASE_URL}/customers", timeout=30)
    # Connection auto-closed when 'with' block exits

🏆 OAuth 2.0 — Client Credentials Flow (Production Standard)
Why OAuth 2.0?

OAuth 2.0 Client Credentials is the standard for machine-to-machine (M2M) authentication — perfect for data pipelines. You exchange client_id + client_secret for a short-lived access token. If the token is stolen, it expires quickly (minutes).

        Python — OAuth2 Client Credentials (Full Production Pattern)
      

      import requests
import time

class OAuth2TokenManager:
    """Manages OAuth2 token lifecycle with auto-refresh."""
    
    def __init__(self, token_url, client_id, client_secret):
        self.token_url = token_url
        self.client_id = client_id
        self.client_secret = client_secret
        self._token = None
        self._token_expiry = 0
    
    def _fetch_token(self):
        """Exchange client credentials for access token."""
        response = requests.post(
            url=self.token_url,
            data={                            # OAuth uses form data, NOT json
                "grant_type": "client_credentials",
                "client_id": self.client_id,
                "client_secret": self.client_secret,
                "scope": "transactions.read accounts.read"
            },
            timeout=30
        )
        response.raise_for_status()
        token_data = response.json()
        
        self._token = token_data["access_token"]
        # expires_in is seconds — subtract buffer of 60s for safety
        self._token_expiry = time.time() + token_data.get("expires_in", 3600) - 60
        return self._token
    
    def get_token(self):
        """Return valid token, refreshing if expired."""
        if not self._token or time.time() >= self._token_expiry:
            self._fetch_token()
        return self._token

# ── Usage ──
token_mgr = OAuth2TokenManager(
    token_url="https://login.bank.com/oauth2/token",
    client_id="pipeline-client-id",      # from Azure Key Vault
    client_secret="super-secret"           # from Azure Key Vault
)

headers = {"Authorization": f"Bearer {token_mgr.get_token()}"}
response = requests.get("https://api.bank.com/v2/transactions",
                         headers=headers, timeout=30)

Pagination — Fetching Large Datasets

🔷 Why Pagination Exists
Problem → Solution

APIs never return all records at once (imagine 10 million transactions in one response — it would crash everything). Instead they return a page of results and tell you how to get the next page. As a DE, you must loop through all pages to get the complete dataset.

Analogy: Like reading a book — you read page 1, then page 2, etc. The API tells you "there are 200 more pages after this one".
📄 Pattern 1 — Offset/Limit Pagination
🧠 Memory Pattern

?limit=100&offset=0 → page 1 | ?limit=100&offset=100 → page 2

Think: LIMIT = page size, OFFSET = how many records to skip

      

        

        Python — Offset Pagination
      

      def fetch_all_offset(base_url, headers, page_size=100):
    """Fetch all pages using offset pagination."""
    all_records = []
    offset = 0

    while True:
        response = requests.get(
            url=base_url,
            params={"limit": page_size, "offset": offset},
            headers=headers,
            timeout=30
        )
        response.raise_for_status()
        data = response.json()
        
        records = data.get("data", [])
        if not records:         # empty page = done
            break
        
        all_records.extend(records)
        offset += page_size
        
        # Safety check using total count
        total = data.get("total_count", None)
        if total and offset >= total:
            break

        print(f"Fetched {len(all_records)}/{total} records")
    
    return all_records
📄 Pattern 2 — Cursor / Next-Page-Token Pagination (Most Common in Modern APIs)
🧠 Memory Pattern

API returns next_page_token or cursor. Send it back in next request. When null/missing → last page.

Think: cursor pagination = "API hands you a bookmark" — keep using it until the bookmark is gone

      

        

        Python — Cursor Pagination
      

      def fetch_all_cursor(base_url, headers, page_size=100):
    """Fetch all pages using cursor/token pagination."""
    all_records = []
    next_token = None

    while True:
        params = {"limit": page_size}
        if next_token:
            params["page_token"] = next_token    # attach cursor

        response = requests.get(url=base_url, params=params,
                                  headers=headers, timeout=30)
        response.raise_for_status()
        data = response.json()
        
        all_records.extend(data.get("data", []))
        
        # Get next token — API-specific field names vary!
        next_token = (
            data.get("next_page_token") or   # common name
            data.get("cursor") or             # alt name
            data.get("pagination", {}).get("next")  # nested
        )

        if not next_token:   # None or "" = last page
            break
        
        print(f"Page done. Total so far: {len(all_records)}")
    
    return all_records
📄 Pattern 3 — Page Number Pagination
      

        

        Python — Page Number Pagination
      

      def fetch_all_pages(base_url, headers, page_size=100):
    all_records = []
    page = 1

    while True:
        resp = requests.get(base_url,
                            params={"page": page, "per_page": page_size},
                            headers=headers, timeout=30)
        resp.raise_for_status()
        data = resp.json()
        
        records = data.get("results", [])
        if not records: break
        
        all_records.extend(records)
        
        # Check if there's a next page
        has_next = data.get("has_next", False)
        if not has_next: break
        page += 1
    
    return all_records

Error Handling — Building Resilient Pipelines

🔷 Why Error Handling Matters in Production
Production Reality

APIs fail regularly in production — network blips, server overloads, token expiry, maintenance windows. A pipeline without retry logic will silently drop data or crash at 3 AM. Robust error handling is what separates junior from senior data engineers.

WHAT

Catching exceptions, checking status codes, logging failures, and deciding whether to retry or fail-fast.

WHY

Network is unreliable. Servers fail. Tokens expire. Without handling, one bad API call crashes the entire pipeline.

WHEN TO RETRY

5xx errors (server problem), 429 (rate limit), connection timeouts. NOT for 4xx (your bug — fix the request).

HOW

try/except + raise_for_status() + exponential backoff with jitter. Use the tenacity library in production.

🛡️ Full Error Handling Pattern
      

        

        Python — Complete Error Handling Pattern
      

      import requests
import logging

logger = logging.getLogger(__name__)

def safe_api_call(url, headers, params=None, timeout=30):
    """
    Makes a safe API call with complete error handling.
    Returns (data, None) on success, (None, error_message) on failure.
    """
    try:
        response = requests.get(
            url=url, headers=headers, params=params, timeout=timeout
        )
        
        # ── Handle by status code ──
        if response.status_code == 200:
            return response.json(), None
        
        elif response.status_code == 401:
            logger.error("AUTH FAILED — token expired or invalid")
            raise ValueError("Authentication failed")
        
        elif response.status_code == 403:
            logger.error("PERMISSION DENIED — check API scopes")
            raise PermissionError("Forbidden")
        
        elif response.status_code == 404:
            logger.warning(f"NOT FOUND: {url}")
            return None, "404 Not Found"
        
        elif response.status_code == 429:
            retry_after = int(response.headers.get("Retry-After", 60))
            logger.warning(f"RATE LIMITED — retry after {retry_after}s")
            raise requests.exceptions.RetryError(f"Rate limited: wait {retry_after}s")
        
        else:
            response.raise_for_status()  # handles all other 4xx/5xx
    
    except requests.exceptions.ConnectionError as e:
        logger.error(f"CONNECTION ERROR: {e}")
        raise   # re-raise for retry logic to catch
    
    except requests.exceptions.Timeout as e:
        logger.error(f"TIMEOUT after {timeout}s: {e}")
        raise
    
    except requests.exceptions.HTTPError as e:
        logger.error(f"HTTP ERROR: {e.response.status_code} — {e}")
        raise
    
    except requests.exceptions.RequestException as e:
        logger.error(f"UNEXPECTED ERROR: {e}")
        raise

Rate Limiting & Exponential Backoff

⏱️ What is Rate Limiting?
Definition

APIs limit how many requests you can make per second/minute/hour to prevent abuse. When you exceed the limit, you get a 429 Too Many Requests response. Your pipeline must slow down and retry intelligently.

Analogy: Like a highway toll booth — it processes X cars per minute. If you try to push through faster, you get blocked. You must wait your turn.
🔄 Exponential Backoff with Jitter
🧠 Memory Pattern

Wait = base * (2^attempt) + random_jitter

Attempt 1: wait 2s → Attempt 2: wait 4s → Attempt 3: wait 8s → Attempt 4: wait 16s
Jitter = adds randomness so all parallel workers don't retry at the same time (thundering herd)

      

        

        Python — Exponential Backoff + tenacity Library
      

      import time
import random
import requests

# ── Manual Implementation ──
def retry_with_backoff(url, headers, max_retries=5):
    for attempt in range(max_retries):
        try:
            response = requests.get(url, headers=headers, timeout=30)
            
            if response.status_code == 429:
                retry_after = int(response.headers.get("Retry-After", 0))
                wait = max(retry_after, 2 ** attempt + random.uniform(0, 1))
                print(f"Rate limited. Waiting {wait:.1f}s (attempt {attempt+1}/{max_retries})")
                time.sleep(wait)
                continue
            
            if response.status_code >= 500:
                wait = 2 ** attempt + random.uniform(0, 1)
                print(f"Server error {response.status_code}. Retrying in {wait:.1f}s")
                time.sleep(wait)
                continue
            
            response.raise_for_status()
            return response.json()
        
        except (requests.exceptions.ConnectionError, requests.exceptions.Timeout):
            if attempt == max_retries - 1:
                raise
            wait = 2 ** attempt + random.uniform(0, 1)
            time.sleep(wait)
    
    raise Exception(f"All {max_retries} retries failed for {url}")


# ── Production: Use tenacity library ──
from tenacity import (retry, stop_after_attempt, wait_exponential,
                       retry_if_exception_type, before_sleep_log)
import logging

logger = logging.getLogger(__name__)

@retry(
    stop=stop_after_attempt(5),
    wait=wait_exponential(multiplier=1, min=2, max=60),
    retry=retry_if_exception_type((
        requests.exceptions.ConnectionError,
        requests.exceptions.Timeout
    )),
    before_sleep=before_sleep_log(logger, logging.WARNING)
)
def fetch_with_tenacity(url, headers):
    response = requests.get(url, headers=headers, timeout=30)
    response.raise_for_status()
    return response.json()
🚦 Throttling — Proactive Rate Control
      

        

        Python — Throttle Requests (Stay Below Limit)
      

      import time

class RateLimiter:
    """Stay within N requests per second."""
    
    def __init__(self, max_per_second=5):
        self.min_interval = 1.0 / max_per_second   # 0.2s between calls
        self.last_call = 0
    
    def wait(self):
        elapsed = time.time() - self.last_call
        sleep_time = self.min_interval - elapsed
        if sleep_time > 0:
            time.sleep(sleep_time)
        self.last_call = time.time()

# ── Usage: 5 requests/second max ──
limiter = RateLimiter(max_per_second=5)

for account_id in account_ids:
    limiter.wait()   # ensure we don't exceed rate limit
    data = fetch_transactions(account_id)

def fetch_transactions(token, from_date, to_date):
    headers = {"Authorization": f"Bearer {token}", "Accept": "application/json"}
    all_txns = []
    next_token = None
    
    while True:
        params = {"from_date": from_date, "to_date": to_date, "limit": 500}
        if next_token: params["page_token"] = next_token

            try:
                resp = requests.get(f"{API_BASE}/transactions",
                                     params=params, headers=headers, timeout=60)
                if resp.status_code == 429:
                    wait = int(resp.headers.get("Retry-After", 2 ** attempt))
                    time.sleep(wait); continue
                resp.raise_for_status()
                break
            except requests.exceptions.RequestException:
                if attempt == 4: raise
                time.sleep(2 ** attempt)

        next_token = data.get("next_page_token")

        if not next_token: break
    
    return all_txns

# ═══════════════════════════════════════════════
# STEP 4: Save raw JSON to ADLS (landing zone)
# ═══════════════════════════════════════════════
def save_raw_to_adls(transactions, run_date):
    raw_path = f"{ADLS_PATH}/dt={run_date}/raw.json"
    # Save each record as newline-delimited JSON
    json_lines = "\n".join([json.dumps(t) for t in transactions])
    dbutils.fs.put(raw_path, json_lines, overwrite=True)
    logger.info(f"Saved {len(transactions)} records to {raw_path}")
    return raw_path

# ═══════════════════════════════════════════════
# STEP 5: Transform and write to Delta Lake
# ═══════════════════════════════════════════════
def write_to_delta(spark, raw_path):
    df = spark.read.json(raw_path)
    
    # Add metadata columns
    from pyspark.sql.functions import current_timestamp, lit
    df = (df
          .withColumn("ingested_at", current_timestamp())
          .withColumn("pipeline_version", lit("v2.1"))
          .dropDuplicates(["transaction_id"])
    )
    
    # MERGE into Delta (upsert = no duplicate transactions)
    if DeltaTable.isDeltaTable(spark, DELTA_PATH):
        delta_tbl = DeltaTable.forPath(spark, DELTA_PATH)
        (delta_tbl.alias("target")
         .merge(df.alias("source"),
                "target.transaction_id = source.transaction_id")
         .whenMatchedUpdateAll()
         .whenNotMatchedInsertAll()
         .execute())
    else:
        df.write.format("delta").mode("overwrite").save(DELTA_PATH)
    
    logger.info(f"Delta merge complete: {df.count()} records")

# ═══════════════════════════════════════════════
# MAIN EXECUTION
# ═══════════════════════════════════════════════
spark = SparkSession.builder.appName("BankAPIIngestion").getOrCreate()
run_date = date.today().isoformat()

token = get_access_token()
transactions = fetch_transactions(token, from_date=run_date, to_date=run_date)
raw_path = save_raw_to_adls(transactions, run_date)
write_to_delta(spark, raw_path)

print(f"✅ Pipeline complete: {len(transactions)} transactions ingested")

          Python — Banking Reconciliation Pipeline
        

        class TransactionReconciliationPipeline:
    
    GATEWAY_API = "https://api.paymentgateway.com/v3"
    RATE_LIMIT_RPS = 1.5  # 1.5 req/sec = 90/min (below 100/min limit)
    
    def __init__(self, spark):
        self.spark = spark
        self.token_mgr = OAuth2TokenManager(
            token_url=f"{self.GATEWAY_API}/auth/token",
            client_id=dbutils.secrets.get("kv", "gw-client-id"),
            client_secret=dbutils.secrets.get("kv", "gw-client-secret")
        )
        self.limiter = RateLimiter(self.RATE_LIMIT_RPS)
        self.session = requests.Session()
    
    def run(self, process_date):
        try:
            # 1. Authenticate
            self.session.headers.update({
                "Authorization": f"Bearer {self.token_mgr.get_token()}"
            })
            
            # 2. Fetch all transactions with pagination + rate limit
            transactions = self._fetch_paginated(process_date)
            
            # 3. Land raw to ADLS
            raw_path = f"abfss://raw@bank.dfs.core.windows.net/txn/dt={process_date}"
            self._save_raw(transactions, raw_path)
            
            # 4. Transform and upsert to Delta
            self._upsert_delta(raw_path)
            
            return {"status": "SUCCESS", "count": len(transactions)}
        
        except Exception as e:
            # Log to Azure Monitor via structured logging
            raise Exception(f"Pipeline FAILED for {process_date}: {e}") from e
    
    def _fetch_paginated(self, date):
        records, next_token = [], None
        while True:
            self.limiter.wait()        # throttle
            params = {"date": date, "limit": 1000}
            if next_token: params["cursor"] = next_token
            
            r = self.session.get(
                f"{self.GATEWAY_API}/transactions",
                params=params, timeout=60
            )
            r.raise_for_status()
            page = r.json()
            records.extend(page["transactions"])
            next_token = page.get("cursor")
            if not next_token: break
        return records

Interview Insights — Ace Every Question

🎤 Interview Questions & Model Answers

Q1 (Direct): "How would you ingest data from an external REST API into a data lake?"
✅ Model Answer: "I'd use Python's requests library in a Databricks notebook. First, fetch OAuth2 token from Azure Key Vault using dbutils.secrets. Then loop through paginated API responses using cursor-based pagination, applying exponential backoff for retries on 5xx/429 errors. Land raw JSON to ADLS Gen2 as partitioned files, then read with Spark, apply transformations, and upsert into Delta Lake using MERGE for idempotency."
Q2 (Architecture): "The API has a 100 requests/minute limit. How do you handle this with 50,000 accounts?"
✅ Model Answer: "I'd implement a RateLimiter class that spaces requests to ~1.5 req/sec (well below the limit). For 50K accounts at 1.5/sec, that's ~9 hours — so I'd parallelize using Spark parallelForEach or batch the accounts and use Databricks workflows for concurrent jobs, each handling a subset of accounts. I'd also check if the API supports bulk endpoints to reduce call count."
Q3 (Tricky): "How do you ensure no duplicate records when the same API is called multiple times?"
✅ Model Answer: "Use Delta Lake's MERGE (upsert) operation with whenMatchedUpdateAll().whenNotMatchedInsertAll() keyed on a unique transaction ID. Also deduplicate within each batch using dropDuplicates(['transaction_id']) before writing. For the raw landing zone, partition by date and use overwrite mode per date — rerunning the pipeline for a date replaces the partition cleanly."
Q4 (Security): "Where do you store API credentials in production?"
✅ Model Answer: "Always in Azure Key Vault — never hardcoded. In Databricks, I use dbutils.secrets scopes backed by Key Vault. Credentials are rotated regularly and the service principal used for pipeline access has minimum permissions (read-only scope on the API, write-only on target storage). I also ensure secrets never appear in logs using structured logging that masks sensitive headers."
Q5 (Debugging): "The API returns 200 but the data looks wrong. How do you debug?"
✅ Model Answer: "First, print response.json() to inspect the full structure — often the data is nested deeper than expected. Check if there's a pagination issue (last page returns empty, not 404). Verify the request URL and params are what you intended using response.url. Check response headers for any warnings. Compare field names exactly — APIs are case-sensitive."
🚨 Common Mistakes:
  • Not setting timeout — pipeline hangs forever on slow API
  • Using params in URL manually instead of using params={} (URL encoding errors)
  • Not handling empty pages in pagination — infinite loop!
  • Catching generic Exception and silently swallowing errors
  • Hardcoding credentials — immediate disqualification in security-focused interviews
  • Not using raise_for_status() — 4xx/5xx responses return "successfully" without it

💡 Production Pro Tips

  • Schema on read vs write: Always define explicit schemas when reading API data into Spark. Auto-inferred schemas scan all data (expensive) and can change silently between runs — breaking your pipeline days later.
  • Idempotent pipelines: Design every pipeline to be safely re-runnable. Use MERGE not INSERT, use partition overwrite not append, and always log which date/batch was processed.
  • Circuit breaker pattern: If an API fails 5 consecutive times, stop retrying and alert. Don't let a broken API burn your rate limit budget or keep a pipeline zombie-running for hours.

🧠 Quiz

Scenario: Your Databricks pipeline fetches daily transactions from a banking API (cursor pagination, 429 rate limiting). The pipeline has been running for 2 months. Today it fails midway through page 47 of 200 due to a network timeout. When it restarts, what is the BEST strategy?

  • A) Restart from page 1 and re-fetch all 200 pages. Use INSERT mode so no data is lost.
  • B) Skip today's run and pick up tomorrow. The API will have today's data tomorrow too.
  • C) Track the last successful cursor/page token in a checkpoint file/Delta table, resume from there. Use MERGE to upsert into Delta (idempotent).
  • D) Increase the timeout to 10 minutes and retry the failed page immediately, ignoring all other pages.
✅ Answer: C — Checkpoint the cursor so you resume from where you failed (saves 46 pages of wasted API calls). Using MERGE ensures even if pages overlap, there are no duplicate records. This is the production-standard pattern combining checkpointing + idempotent writes.
