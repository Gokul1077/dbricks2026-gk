Prerequisites — JVM, Bytecode & Distributed Computing Basics

What is the JVM?

☕

Java Virtual Machine (JVM)

A virtual runtime engine that allows Java (or Scala/Kotlin) code to run on any operating system without recompiling. Think of it as a universal translator — you write code once, JVM runs it everywhere.

Why does Spark care about JVM? Apache Spark is written in Scala, which compiles to JVM bytecode. Every Spark Executor, Driver process, and Worker runs inside a JVM. Even PySpark talks to JVM via a bridge called Py4J.

🔵 Key Mental Model Java Source Code (.java) → compiler → Bytecode (.class file) → JVM interprets at runtime on any OS/hardware.

Key Terms You Must Know Before Everything Else

🧱
Node
A single machine (physical server or virtual machine) in a cluster. Has its own CPU, RAM, and disk.
🖥️
Cluster
A collection of nodes connected by a network, working together as one logical unit.
📦
Block
A fixed-size physical chunk of data stored on disk. In HDFS, default = 128 MB. Exists BEFORE Spark runs.
🔀
Partition
A logical unit of data Spark creates at runtime. Defines how much data one task will process. Configurable.
⚙️
Process vs Thread
A process = isolated JVM instance. A thread = lightweight execution unit inside a process. Executor cores = threads.
🦥
Lazy Evaluation
Spark does NOT execute transformations immediately. It builds a plan and executes only when an action (like .count()) is called.

File Systems — Regular FS vs Distributed File System (DFS)

💾

The Core Problem

A single machine's disk can hold maybe 10–20 TB. Modern enterprises deal with petabytes. You need a way to spread data across hundreds of machines and still access it as if it's one file system.

What is a Regular File System?

A regular file system (like ext4 on Linux, NTFS on Windows) manages how data is stored and retrieved on a single machine's disk. It knows where files start and end on the local disk.

Limitation: If your file is 5 TB and your disk is 1 TB — you can't store it. You can't process it either without reading it sequentially.

What is a Distributed File System (DFS)?

A DFS splits large files into blocks and distributes those blocks across multiple machines. It presents a unified namespace — you see one big file, but internally it's spread across 100 nodes.

❌ Regular FS One machine. One disk. If disk fails → data lost. Can't scale beyond hardware limit.
✅ Distributed FS (DFS/HDFS) Many machines. Data replicated. If one node fails → other replicas serve data. Scales horizontally.

FS vs DFS — Comparison

HDFS — Architecture, Internals & Interview Deep Dive

🗄️

Hadoop Distributed File System (HDFS)

HDFS is a distributed, fault-tolerant file system designed to store very large files (GBs to TBs) across a cluster of commodity hardware. It is the storage backbone of the Hadoop ecosystem.

The 2-Component Architecture

HDFS has exactly two types of components — NameNode (the brain) and DataNode (the storage worker):

📐 HDFS Architecture Diagram
🧠
NameNode (Master)
Stores only metadata (file names, permissions, block-to-DataNode mappings). It does NOT store actual file data. The single point of contact for all clients. If NameNode goes down → entire HDFS is unavailable (SPOF).
💿
DataNode (Worker)
Stores actual data blocks. Sends heartbeats to NameNode every 3 seconds. Handles block replication with other DataNodes. Multiple DataNodes per cluster.
📑
Secondary NameNode
⚠️ Misleading name! It is NOT a hot standby. Its job is to periodically merge the fsimage + edit logs to reduce NameNode startup time. Full HA needs separate NameNode HA setup.
🔢
Block Size
Default = 128 MB (Hadoop 2.x). Large block size reduces metadata overhead. A 1 GB file = 8 blocks of 128 MB each.

How HDFS Stores a File — Step by Step

Client asks NameNode: "I want to store a 384 MB file." NameNode responds with a list of DataNodes for each of the 3 blocks (384 / 128 = 3 blocks).
Client writes Block-1 to DataNode-1. DataNode-1 pipelines (replicates) Block-1 to DataNode-2, which replicates to DataNode-3.
Same for Block-2 and Block-3 — different DataNodes used to spread load.
NameNode updates its metadata with block locations. Client never talks to NameNode for actual data — only for metadata.
DataNodes send heartbeats every 3 seconds. If a node dies, NameNode detects it and triggers re-replication of its blocks elsewhere.

HDFS — Key Properties for Interviews

PropertyValue / BehaviorWhy It Matters
Default block size128 MB (Hadoop 2), 64 MB (Hadoop 1)Reduces NameNode metadata load
Replication factor3 (default)1 original + 2 copies = fault tolerance
Write modelWrite-once, append-onlyNo random writes; optimized for batch
Data localityNameNode knows which DataNode has which blockSpark moves computation to data, not data to computation
SPOF riskSingle NameNode = single point of failureHDFS HA (High Availability) needed for production
Best forLarge sequential reads/writesBad for random access, small files, real-time

Data Locality — The Killer Feature of HDFS

In HDFS, Spark knows exactly which DataNode has which block. So Spark schedules tasks to run on the same node where the data lives, avoiding expensive network transfers.

✅ Data Locality Benefit Without data locality: Read 128 MB over network = slow.
With data locality: Read 128 MB from local disk = 10-100× faster.
Spark's task scheduler honors locality levels: PROCESS_LOCAL → NODE_LOCAL → RACK_LOCAL → ANY

HDFS Limitations (Why We Moved On)

  • SPOF: Single NameNode = entire cluster down if it fails (unless HA configured)
  • Small files problem: Millions of small files overwhelm NameNode memory
  • On-premise only: Requires dedicated hardware, high ops cost
  • Not cloud-native: Cannot scale elastically like ADLS or S3
  • No ACID transactions: No update/delete support natively
  • Tightly coupled with compute: Storage and compute cannot scale independently

🎤 HDFS Interview Questions

Q1. What is the role of NameNode vs DataNode in HDFS?
Model Answer: NameNode is the master that stores metadata (file-to-block mapping, block-to-DataNode mapping, permissions) but never stores actual data. DataNodes store the physical blocks and handle all read/write operations. Clients contact NameNode to discover block locations, then interact with DataNodes directly.
Q2. What happens when a DataNode fails?
Model Answer: DataNode sends heartbeats every 3 seconds. If NameNode doesn't receive a heartbeat for 10 minutes, it marks that node as dead and instructs other DataNodes to re-replicate the blocks that were on the failed node, restoring the replication factor back to 3.
Q3 (Trick). Is Secondary NameNode a backup for NameNode?
❌ Common mistake: Saying "yes, Secondary NameNode is a standby." ✅ Correct: Secondary NameNode is a checkpointing service — it merges the fsimage and edit logs periodically to prevent edit log from growing too large. For true HA, you need HDFS NameNode HA with a Standby NameNode and ZooKeeper.

MapReduce — How It Works & Why It Hurts

🗺️

MapReduce (Hadoop)

A programming model for processing large datasets by breaking work into two phases: Map (filter & transform data in parallel) and Reduce (aggregate results). Every intermediate result is written to disk.

How MapReduce Works — Word Count Example

📐 MapReduce Word Count Flow

Pain Point 1: Disk I/O Hell

Every intermediate result — after Map, after Shuffle — is written to disk (HDFS). Then the Reduce phase reads it back from disk. For iterative algorithms (like ML training with 100 iterations), this means 200 disk read/write operations.

🔴 MapReduce Bottleneck RAM access speed: ~100 GB/s  |  Disk access speed: ~200 MB/s
MapReduce forces everything through disk. Spark keeps intermediate data in RAM. Result: Spark is 10–100× faster for iterative workloads.

Pain Point 2: Rigid One Shuffle Workflow

Every MapReduce job has exactly one Map phase and one Reduce phase. If your pipeline needs two different grouping keys (e.g., join on customer_id then aggregate on country), you need two separate MapReduce jobs — each reading and writing from HDFS.

-- Real-world query needing 2 MapReduce jobs:
-- "Total order amount per country for COMPLETED orders only"

-- MapReduce Job 1: Filter completed orders + JOIN orders with customers
-- Intermediate result → written to HDFS disk

-- MapReduce Job 2: Read from HDFS again → GROUP BY country → SUM amount
-- Final result → written to HDFS

-- Spark does this in ONE DAG, in-memory, with ZERO intermediate HDFS writes
df = spark.read.parquet("orders/")\
  .filter("status = 'COMPLETED'")\
  .join(customers, "customer_id")\
  .groupBy("country")\
  .agg(sum("amount").alias("total"))
# One DAG. In-memory. No HDFS intermediate writes.

How Spark Solves Each MapReduce Pain Point

✅ In-Memory Processing Spark stores intermediate results in executor RAM, not disk. Only shuffles (stage boundaries) write to disk. For iterative ML (100 epochs), Spark reads data once into memory and iterates in RAM.
✅ DAG Execution Model Spark builds a full DAG (Directed Acyclic Graph) of all transformations before executing. This allows Catalyst Optimizer to reorder, prune, and merge operations — impossible in MapReduce's fixed Map→Reduce model.

Spark Execution Flow — Jobs → Stages → Tasks

The Full Pipeline: From Code to Results

📐 Spark Execution Pipeline

Jobs, Stages, Tasks — The Hierarchy

📐 Mental Model Application → contains multiple Jobs → each Job splits into Stages → each Stage splits into Tasks
Jobs triggered by actions. Stages split at shuffle boundaries. Tasks map 1:1 to partitions.
UnitCreated WhenDetermined ByRuns Where
JobAction called (count(), collect(), write())Number of actions in your codeConceptual — triggered on driver
StageShuffle boundary in the DAGNarrow vs Wide transformationsDAG Scheduler on driver
TaskOne per partition per stageNumber of partitionsExecutor cores on worker nodes

Narrow vs Wide Transformations

✅ Narrow Transformation (same stage) Each input partition → exactly one output partition. No data movement across nodes.

Examples: filter(), map(), select(), withColumn()
⚠️ Wide Transformation (new stage = shuffle) One input partition's data → goes to MULTIPLE output partitions. Requires network shuffle.

Examples: groupBy(), join(), distinct(), orderBy()

Parallelism Formula — The Most Asked Interview Formula

📐 Spark Parallelism Effective Parallelism = min(number of partitions, total available executor cores)

If you have 200 partitions and only 96 cores → first 96 tasks run, then next 96, then final 8.
Data size, node count, executor memory, CPU speed do NOT directly determine parallelism.

Partition Types

Input Partitions
Created when Spark reads data. Controlled by spark.sql.files.maxPartitionBytes (default 128MB). Check with df.rdd.getNumPartitions().
Shuffle Partitions
Created after a shuffle (groupBy, join). Controlled by spark.sql.shuffle.partitions (default 200). Often the biggest performance tuning lever.
Output Partitions
Controls how many files are written to storage. Too many = small files problem. Use coalesce() or repartition() to control.

How Spark Reads from ADLS (No NameNode!)

ADLS does not have a NameNode. Instead, Spark uses the ABFS (Azure Blob File System) driver to communicate with ADLS via HTTP range requests:

ABFS driver authenticates with ADLS using service principal / managed identity / OAuth token.
Lists files and gets metadata (file sizes, schema for Parquet/Delta).
Spark creates partitions based on file sizes (maxPartitionBytes setting).
Each executor sends HTTP GET requests with byte-range headers to fetch only its partition's data.
Azure internally serves data from the nearest replica (LRS/ZRS). Spark doesn't know or care — it just receives bytes.
# Accessing ADLS Gen2 from Databricks / PySpark
# ABFS URI format: abfss://<container>@<account>.dfs.core.windows.net/path

spark.conf.set(
  "fs.azure.account.auth.type.<account>.dfs.core.windows.net", 
  "OAuth"
)
spark.conf.set(
  "fs.azure.account.oauth.provider.type.<account>.dfs.core.windows.net",
  "org.apache.hadoop.fs.azurebfs.oauth2.ClientCredsTokenProvider"
)

# Read Parquet from ADLS — Spark creates HTTP range-read requests internally
df = spark.read.parquet(
  "abfss://bronze@myadls.dfs.core.windows.net/transactions/"
)
df.printSchema()
print(f"Partitions: {df.rdd.getNumPartitions()}")

🏦 Real-World Banking Pipeline — Azure + Databricks

Scenario: Daily Transaction Reconciliation Pipeline

Business need: Every night, process 50M+ banking transactions from core banking system, join with customer master data, detect anomalies, and produce reconciliation reports per region.

Architecture Components

Azure Event Hubs
Ingests real-time transaction events from core banking. Partitioned Kafka-compatible stream.
ADLS Gen2 (Bronze layer)
Raw transaction files landed in Parquet format from Event Hubs capture. Immutable raw zone.
Databricks Spark Cluster
Processes Bronze → Silver → Gold transformations using PySpark. Auto-scaling cluster.
Delta Lake (Silver/Gold)
ACID-compliant table format on ADLS. Enables MERGE (upsert), time-travel, schema enforcement.
Unity Catalog
Metadata catalog managing table locations, access control, lineage for all Delta tables.
Azure Key Vault
Stores ADLS credentials, service principal secrets. Databricks Secret Scope reads from it.

Data Flow

INGEST
Core banking → Event Hubs → ADLS Bronze
50M transaction events/day land as Parquet files in ADLS Bronze container. 128MB files = ~400 files = ~400 initial partitions for Spark.
STAGE 1 — Bronze → Silver
Cleansing + Schema Enforcement
Spark reads 400 partitions. Narrow transformations (filter, cast, withColumn) — single stage, no shuffle. 400 tasks run in parallel across executor cores.
STAGE 2 — Join
Transactions JOIN Customer Master
Wide transformation. Triggers shuffle. Spark creates 200 shuffle partitions (spark.sql.shuffle.partitions). Catalyst may use BroadcastHashJoin if customer table < 10MB.
STAGE 3 — Aggregate
GROUP BY region → SUM amounts
Second shuffle. Catalyst applies predicate pushdown and column pruning to minimize data scanned. Results written to Gold Delta table.
OUTPUT
Delta Lake Gold → Power BI / Reports
Gold Delta table partitioned by region_date. Power BI via Databricks SQL Warehouse for reporting. Unity Catalog enforces row-level security per analyst's region access.

Production PySpark Code

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, sum, broadcast, to_date
from delta.tables import DeltaTable

# ── SparkSession (Driver creates this once) ─────────────────────────
spark = SparkSession.builder \
  .appName("TxnReconciliation") \
  .config("spark.sql.shuffle.partitions", "400")  # tuned for our data size
  .config("spark.sql.files.maxPartitionBytes", "134217728")  # 128MB
  .getOrCreate()

# ── Read Bronze (Parquet from ADLS) ─────────────────────────────────
bronze_path = "abfss://bronze@bankadls.dfs.core.windows.net/transactions/"
transactions = spark.read.parquet(bronze_path)
# Spark creates partitions lazily — no data read yet (lazy evaluation)
print(f"Input partitions: {transactions.rdd.getNumPartitions()}")

# ── Stage 1: Narrow transformations (no shuffle, single stage) ──────
silver_txns = transactions \
  .filter(col("status") == "COMPLETED") \                  # narrow
  .filter(col("txn_date") == "2024-01-15") \               # narrow
  .select("txn_id", "customer_id", "amount", "txn_date")  # column pruning
# Catalyst will push the filter + select all the way to Parquet read

# ── Stage 2: Wide transformation — JOIN (triggers shuffle) ──────────
customers_path = "abfss://silver@bankadls.dfs.core.windows.net/customers/"
customers = spark.read.parquet(customers_path)
# If customers table is small, Catalyst uses BroadcastHashJoin (no shuffle!)
enriched = silver_txns.join(broadcast(customers), "customer_id")

# ── Stage 3: Wide transformation — GROUP BY (triggers shuffle) ──────
gold_df = enriched \
  .groupBy("region", "txn_date") \
  .agg(
    sum("amount").alias("total_amount"),
    count("*").alias("txn_count")
  )

# ── Write to Delta Lake (action — triggers DAG execution) ───────────
gold_path = "abfss://gold@bankadls.dfs.core.windows.net/reconciliation/"
gold_df.write \
  .format("delta") \
  .mode("overwrite") \
  .partitionBy("region", "txn_date") \
  .save(gold_path)
# ↑ THIS is the action. ONLY now does Spark build the DAG and run tasks.
# All previous lines were just building the logical plan (lazy!).
PYSPARK — PRODUCTION
✅ What Catalyst Does Here 1. Predicate Pushdown: filter(status=COMPLETED) pushed into Parquet file scan → reads only relevant row groups
2. Column Pruning: select(4 columns) → Parquet reads only those 4 column chunks, not all 30
3. Broadcast Join: If customers table < spark.sql.autoBroadcastJoinThreshold → BroadcastHashJoin (zero shuffle!)
4. Stage merging: filter + select merged into Stage 1 (narrow pipeline)

🎤 Master Interview Q&A + Quiz

🎤 HDFS + Spark Interview Questions

Q1. Walk me through what happens when you submit a Spark job.
Model Answer: (1) Driver program starts, SparkSession created. (2) Transformations build an unresolved logical plan (lazy). (3) Action triggers job creation. (4) Catalyst Optimizer resolves and optimizes the logical plan (predicate pushdown, column pruning). (5) Physical plan created via Tungsten code generation. (6) DAG Scheduler splits plan into stages at shuffle boundaries. (7) Task Scheduler converts each stage into tasks (1 per partition). (8) Tasks assigned to available executor cores based on locality. (9) Executors run tasks in parallel, cache intermediate results in RAM. (10) Results returned to driver.
Q2. What determines the number of tasks in a Spark stage?
Model Answer: Number of tasks = number of partitions for that stage. For Stage 1 (input stage): determined by number of input file blocks (controlled by spark.sql.files.maxPartitionBytes). For Stage 2+ (post-shuffle stages): determined by spark.sql.shuffle.partitions (default 200). Executors and cores determine how many tasks run concurrently, not the total task count.
Q3. What is the difference between a Spark Driver and a Spark Executor?
Model Answer: Driver is the JVM process that runs your application code, contains SparkSession, DAG Scheduler, and Task Scheduler. It orchestrates everything but doesn't process data. Executor is a JVM process on worker nodes that actually runs tasks on data partitions. Executors have fixed cores (task slots) and memory. One application has one driver and many executors.
Q4 (Trick). If I have 1000 partitions and 50 executor cores, how many stages are there?
❌ Common mistake: Confusing partitions with stages. ✅ Correct: Stage count is determined by the number of shuffle boundaries in the DAG, NOT by partitions or cores. With 50 cores, Spark will process 50 tasks at a time, taking 20 rounds (1000/50). But the number of stages depends entirely on whether there are wide transformations (groupBy, join, etc.) in your code.
Q5. Why did modern data engineering move away from HDFS?
Model Answer: (1) HDFS tightly couples storage and compute — can't scale independently. (2) Requires dedicated on-premise hardware with high ops overhead. (3) SPOF risk with NameNode. (4) No ACID transactions. (5) Cloud object storage (ADLS Gen2) offers elastic scaling, pay-per-use, zero ops, and geo-redundancy. Delta Lake adds ACID transactions on top of ADLS, giving the best of both worlds. Storage/compute decoupling with Databricks clusters means you pay for compute only when processing.
Q6. What is lazy evaluation in Spark and why does it matter?
Model Answer: Lazy evaluation means Spark does NOT execute transformations (filter, select, groupBy) immediately. It builds an unresolved logical plan. Only when an action (count, collect, write) is called does Spark compile, optimize, and execute. This matters because: (1) Catalyst Optimizer can see the complete pipeline and apply optimizations across all steps. (2) No wasted computation — if a filter reduces 90% of rows, it's pushed down before joins. (3) Enables pipelining — consecutive narrow transformations merge into a single stage.

💡 Pro Tips for Interviews

  • Mention Catalyst Optimizer by name. Interviewers love when candidates can explain predicate pushdown and column pruning — these show you understand WHY Spark is fast, not just that it IS fast.
  • Use the "partition = task" rule confidently. When asked about parallelism, state: "In Spark, parallelism = min(partitions, available executor cores). The default shuffle partition count of 200 is often suboptimal and needs tuning."
  • Distinguish spark.sql.shuffle.partitions vs repartition(). shuffle.partitions is a config that applies to ALL shuffles; repartition() is an explicit operation that introduces a full shuffle. For skewed data, explain salting or AQE (Adaptive Query Execution).

🧠 Scenario Quiz

A banking pipeline reads 500 Parquet files (totalling 60 GB) from ADLS. After reading, the code does: filter() → join(customers) → groupBy("region") → write(). The cluster has 3 workers, each with 1 Executor of 8 cores. spark.sql.shuffle.partitions = 200. How many STAGES does this job have, and how many TOTAL TASKS run in Stage 1?

A 1 stage, 500 tasks
B 2 stages, 200 tasks in Stage 1
C 3 stages, 500 tasks in Stage 1 (or ~470 depending on maxPartitionBytes)
D 1 stage, 24 tasks (limited by cores)
✅ Answer: C — 3 stages, ~500 tasks in Stage 1

Why 3 stages? There are 2 wide transformations: join (shuffle 1) and groupBy (shuffle 2). Each shuffle = new stage. So: Stage 1 (filter+read → join shuffle write), Stage 2 (join output → groupBy shuffle write), Stage 3 (groupBy output → write).
Why ~500 tasks in Stage 1? Stage 1's task count = number of input partitions, which roughly equals number of files = 500. The 24 cores (3×8) determines how many tasks run CONCURRENTLY, not the total count. Option D confuses parallelism with task count — a classic mistake.
