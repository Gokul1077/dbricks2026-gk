Chapter 0
Prerequisites — Know These First
Terms you must understand before Unity Catalog makes sense

🗄️ Metastore

A central repository that stores metadata — information about your data: table names, schemas, column types, file locations, permissions.

Think: The library catalogue — it doesn't hold the books, it tells you where every book is and who can borrow it.

Without it: You'd have to manually track every table location across dozens of storage accounts.

🏢 Workspace

An isolated Azure Databricks environment with its own notebooks, clusters, jobs, and users. One Azure subscription can have multiple workspaces (e.g., dev, test, prod).

Banking example: Retail banking workspace, Investment banking workspace, Risk workspace — each separate but needing shared data.

📦 Hive Metastore (Legacy)

The old default metadata store in Databricks. Each workspace had its own isolated Hive metastore — tables registered in workspace A were invisible in workspace B.

This is exactly the problem Unity Catalog solves.

🛡️ RBAC vs ABAC

RBAC (Role-Based): Access based on role — "All Analysts can read". ABAC (Attribute-Based): Access based on data attributes — "Only see rows where branch = your branch".

Unity Catalog supports both — table-level RBAC + row/column-level ABAC.

🏦 Banking Domain Glossary

Bronze Layer — Raw ingested data (transactions, core banking feeds). Read-only. Append-only.
Silver Layer — Cleaned, enriched, joined data. Validated, conformed.
Gold Layer — Business aggregates, reports, ML features. Business-ready.

Unity Catalog governs access to ALL three layers — different teams get different access levels per layer.

Chapter 2
Why Do We Need Unity Catalog?
The real problems it solves in production

😰 Problem 1
Workspace Silos

Dev team creates transactions table. Prod team can't see it. Every workspace is an island. Data Engineers manually copy schemas between workspaces.

Banking: Risk team can't access Retail team's customer table even though both are on Databricks.

😰 Problem 2
No Audit Trail

RBI/SOX audit asks: "Who accessed customer PAN data in the last 6 months?" Without Unity Catalog — no answer. Hive metastore doesn't log access.

Banks face regulatory penalties without complete data access audit logs.

😰 Problem 3
No Data Lineage

A downstream fraud model gives wrong results. Where does the data come from? Which transformation broke it? Without lineage — you're blind.

Root cause analysis takes days instead of minutes.

😰 Problem 4
Scattered Permissions

Permissions set in 5 different places: Hive metastore, ADLS ACLs, cluster-level policies, notebook permissions, job permissions. Out of sync, hard to audit.

Offboarding an employee requires changes in 10 places.

😰 Problem 5
No PII Protection

Customer Aadhaar numbers, PAN, salary data visible to all users who can query the table. No row-level or column-level security in Hive metastore.

DPDP Act violation risk in Indian banking context.

😰 Problem 6
No Data Discovery

New Data Analyst joins. They don't know what tables exist, where they are, what they contain. Has to ask senior engineers. Months to onboard.

Productivity lost, errors from using wrong/stale tables.

✅
Unity Catalog solves ALL 6 problems: Unified namespace across workspaces · Complete audit logs · Auto data lineage · Centralized permissions · Column/row-level masking · Searchable data catalog.

🌍 Metastore — The Root

What: Top-level container. One per Azure region per account. All catalogs, schemas, tables live inside it. Attached to multiple workspaces.

Who creates it: Databricks Account Admin (not workspace admin).

Banking analogy: The RBI's master registry for all banks in a region. Every bank (catalog) is registered here.

⚠️
Interview key: You can only have ONE metastore per region per Databricks account. Multiple workspaces share one metastore.

📂 Catalog — Namespace Organizer

What: First level of the 3-part name. Organizes schemas. Use it to separate environments, domains, or business units.

Naming patterns used in banking:

  • retail_banking — Retail domain catalog
  • risk_management — Risk domain catalog
  • dev_retail_banking — Dev environment
  • prod_retail_banking — Prod environment

Built-in catalogs: main (default), hive_metastore (legacy), system (audit logs).

🗂️ Schema (Database) — Logical Group

What: Container for tables, views, volumes, functions within a catalog. Equivalent to a database in traditional SQL.

Banking patterns:

  • bronze / silver / gold — Medallion layers
  • transactions, customers, loans — Domain schemas
  • reporting, ml_features — Use-case schemas

📋 Tables, Views & More

Managed Table: UC owns storage. Drop table → data deleted. Stored under metastore's managed storage location.

External Table: You own storage (ADLS Gen2). Drop table → metadata deleted, data stays.

View: Saved SELECT query. Great for column masking and row-level security.

Volume: UC-managed folder for unstructured files (PDFs, images, CSVs). New in UC!

Function: SQL UDFs registered and governed centrally.

🧠 Memory Trick — 3-Part Naming

catalog.schema.table = building.floor.room
prod_retail_banking.gold.risk_scores = In the Retail Bank building → Gold floor → Risk Scores room.
Always three parts. Always this order. Never skip levels.

Chapter 5
Creating Unity Catalog — Step by Step
From account setup to your first governed table
🔑
Prerequisites for setup: Databricks Account Admin role (not just workspace admin) · Azure Databricks Premium SKU · ADLS Gen2 storage account · Service Principal with Storage Blob Data Contributor role on ADLS

Setup Flow

  1. Create a Metastore — Done once per region by Account Admin in Databricks Account Console. Choose a root storage location (ADLS Gen2 path for managed tables).
  2. Assign Metastore to Workspaces — Link one or more workspaces to the metastore. All linked workspaces now share the same catalog/schema/table namespace.
  3. Create Storage Credential — Register a Managed Identity or Service Principal that UC uses to access ADLS Gen2.
  4. Create External Location — Register an ADLS Gen2 path (e.g., abfss://bronze@bank.dfs.core.windows.net/) tied to the storage credential.
  5. Create Catalogs & Schemas — Organize your namespace. Set owners. Grant access.
  6. Create Tables — Managed (UC stores data) or External (your ADLS). Grant SELECT/MODIFY as needed.

Step 1–2: Metastore (Done in Account Console UI — or via Terraform)

# Terraform approach (Infrastructure as Code — recommended for banking)
resource "databricks_metastore" "bank_metastore" {
  name           = "bank-east-us-metastore"
  storage_root   = "abfss://unity-catalog@bankmetastore.dfs.core.windows.net/"
  owner          = "data-platform-admin@bank.com"
  region         = "eastus"
}

resource "databricks_metastore_assignment" "prod_assign" {
  workspace_id         = databricks_mws_workspaces.prod.workspace_id
  metastore_id         = databricks_metastore.bank_metastore.id
  default_catalog_name = "prod_retail_banking"
}

Step 3: Storage Credential — Tell UC How to Access ADLS

-- In Databricks SQL or Notebook
CREATE STORAGE CREDENTIAL bank_adls_credential
  WITH AZURE_MANAGED_IDENTITY
  (CREDENTIAL = '/subscriptions/xxx/resourceGroups/rg-bank/providers/Microsoft.Databricks/accessConnectors/bank-uc-connector')
  COMMENT 'Managed Identity for Unity Catalog ADLS access';

-- Validate it works
VALIDATE STORAGE CREDENTIAL bank_adls_credential;

Step 4: External Location — Register ADLS Path

CREATE EXTERNAL LOCATION bronze_location
  URL 'abfss://bronze@bankdatalake.dfs.core.windows.net/'
  WITH (STORAGE CREDENTIAL bank_adls_credential)
  COMMENT 'Bronze layer - raw transaction data';

-- Test access
LIST 'abfss://bronze@bankdatalake.dfs.core.windows.net/';

Step 5: Create Catalog, Schema

-- Create catalog for production retail banking
CREATE CATALOG IF NOT EXISTS prod_retail_banking
  COMMENT 'Production catalog for retail banking domain';

-- Set the catalog context
USE CATALOG prod_retail_banking;

-- Create schemas for medallion layers
CREATE SCHEMA IF NOT EXISTS bronze COMMENT 'Raw ingested data';
CREATE SCHEMA IF NOT EXISTS silver COMMENT 'Cleaned and enriched data';
CREATE SCHEMA IF NOT EXISTS gold   COMMENT 'Business-ready aggregates';

Step 6: Create Tables (Managed vs External)

-- MANAGED TABLE (UC controls storage)
CREATE TABLE prod_retail_banking.gold.risk_scores (
  customer_id   BIGINT,
  risk_score    DOUBLE,
  risk_category STRING,
  score_date    DATE,
  model_version STRING
)
USING delta
COMMENT 'Customer credit risk scores - updated daily'
TBLPROPERTIES ('delta.enableChangeDataFeed' = 'true');

-- EXTERNAL TABLE (data lives in your ADLS)
CREATE TABLE prod_retail_banking.bronze.transactions_raw
USING delta
LOCATION 'abfss://bronze@bankdatalake.dfs.core.windows.net/transactions/'
COMMENT 'Raw transactions from core banking system (external)';

-- VOLUME (for unstructured files like KYC docs)
CREATE VOLUME prod_retail_banking.bronze.kyc_documents
  LOCATION 'abfss://bronze@bankdatalake.dfs.core.windows.net/kyc_docs/'
  COMMENT 'KYC PDFs and images';
Chapter 6
Access Control — GRANT, REVOKE, Masking
The most interview-critical chapter

Privilege Hierarchy — Permissions Flow Down

LevelPrivilegeWhat it allowsBanking use case
MetastoreCREATE CATALOGCreate new catalogsData Platform team only
CatalogUSE CATALOGAccess catalogs and schemas withinAll engineers need this minimum
SchemaUSE SCHEMA CREATE TABLEAccess schema; create objectsDE team: CREATE TABLE in bronze
TableSELECT MODIFY ALL PRIVILEGESRead / write / full controlAnalysts: SELECT on gold only
Column/RowDynamic Views / MaskingFilter rows or mask column valuesMask PAN for junior analysts

GRANT Syntax — Pattern to Remember

GRANT <privilege> ON <object_type> <object_name> TO <principal>;
-- Pattern: GRANT [what] ON [what type] [which object] TO [who]
-- principal = user, group, or service principal 
-- Step 1: Always grant USE CATALOG first (required for any access)
GRANT USE CATALOG ON CATALOG prod_retail_banking
  TO `analysts-group`;

-- Step 2: Grant USE SCHEMA
GRANT USE SCHEMA ON SCHEMA prod_retail_banking.gold
  TO `analysts-group`;

-- Step 3: Grant table-level SELECT
GRANT SELECT ON TABLE prod_retail_banking.gold.risk_scores
  TO `analysts-group`;

-- Data Engineers: full access to bronze + silver
GRANT ALL PRIVILEGES ON SCHEMA prod_retail_banking.bronze
  TO `data-engineers-group`;

-- REVOKE access when someone leaves
REVOKE SELECT ON TABLE prod_retail_banking.gold.risk_scores
  FROM `ex-employee@bank.com`;

-- Check who has what access
SHOW GRANTS ON TABLE prod_retail_banking.gold.risk_scores;
SHOW GRANTS TO `analysts-group`;
⚠️
Interview Gotcha! You must grant USE CATALOG AND USE SCHEMA BEFORE table-level grants work. Forgetting this is the #1 "why can't I access the table?" issue in production. Think: you need a building pass (catalog) AND floor access (schema) before you can enter a room (table).

Column Masking — Protect PII (DPDP/GDPR Compliance)

-- 1. Create a masking function
CREATE OR REPLACE FUNCTION prod_retail_banking.silver.mask_pan(pan STRING)
  RETURN CASE
    WHEN IS_MEMBER('compliance-team') THEN pan        -- full PAN for compliance
    ELSE CONCAT('XXXXX', RIGHT(pan, 5))              -- masked for everyone else
  END;

-- 2. Apply mask to the column
ALTER TABLE prod_retail_banking.silver.customers
  ALTER COLUMN pan_number
  SET MASK prod_retail_banking.silver.mask_pan;

-- Now: analysts see XXXXX1234X, compliance sees full PAN

Row-Level Security — Branch-Based Data Isolation

-- Row filter: each analyst sees only their branch's transactions
CREATE OR REPLACE FUNCTION prod_retail_banking.gold.branch_filter(branch_id STRING)
  RETURN IS_MEMBER('admin-group')
    OR branch_id = CURRENT_USER()
    OR IS_MEMBER(CONCAT('branch-', branch_id));

ALTER TABLE prod_retail_banking.gold.daily_summary
  SET ROW FILTER prod_retail_banking.gold.branch_filter
    ON (branch_id);

-- Chennai analyst → sees only Chennai rows automatically
-- Admin → sees all rows
Chapter 7
Data Engineer Workflow with Unity Catalog
What to check at every stage of a project
🏗️
As a Data Engineer, your UC checklist has 5 phases: Project Start → Ingestion → Transformation → Output → Monitoring. Each phase has specific UC interactions.

📋 Phase 1: Project Kickoff — What to Check

  1. Does the metastore exist? Check Account Console. If not — block: no UC work possible until Account Admin creates it.
  2. Is the metastore attached to your workspace? Run SELECT CURRENT_METASTORE(); in a notebook. If returns nothing — escalate to admin.
  3. What catalogs exist? Run SHOW CATALOGS; — understand the namespace structure already defined.
  4. What's the storage location strategy? Confirm ADLS paths for bronze/silver/gold and which external locations are registered.
  5. What groups exist? Coordinate with admin: data-engineers, analysts, ml-team groups should be in Azure AD/Entra ID.

📋 Phase 2: Ingestion (Bronze Layer)

# PySpark in Databricks Notebook — Bronze Ingestion
from pyspark.sql import functions as F

# Always use 3-part name: catalog.schema.table
df_raw = spark.read.format("csv") \
    .option("header", "true") \
    .load("abfss://landing@bankdatalake.dfs.core.windows.net/transactions/")

# Write to UC-governed external table in Bronze
df_raw.write \
    .format("delta") \
    .mode("append") \
    .option("path", "abfss://bronze@bankdatalake.dfs.core.windows.net/transactions/") \
    .saveAsTable("prod_retail_banking.bronze.transactions_raw")
# ↑ saveAsTable with 3-part name = registered in Unity Catalog

# What DE must check:
# 1. Does external location cover this ADLS path? → SHOW EXTERNAL LOCATIONS;
# 2. Does the service principal on the job cluster have WRITE access?
# 3. Does the target schema exist?

📋 Phase 3: Transformation (Silver/Gold)

# Always qualify table names with catalog.schema.table
transactions = spark.table("prod_retail_banking.bronze.transactions_raw")
customers    = spark.table("prod_retail_banking.silver.customers_enriched")

# UC enforces: if job SP doesn't have SELECT on bronze → fails with AnalysisException
enriched = transactions \
    .join(customers, "customer_id") \
    .withColumn("risk_band",
        F.when(F.col("credit_score") < 600, "HIGH")
         .when(F.col("credit_score") < 750", "MEDIUM")
         .otherwise("LOW"))

# Write to Silver — UC creates lineage link: bronze.transactions_raw → silver.loan_risk
enriched.write \
    .format("delta") \
    .mode("overwrite") \
    .option("overwriteSchema", "true") \
    .saveAsTable("prod_retail_banking.silver.loan_risk")

📋 Phase 4: Output — Check Permissions Before Delivery

-- Before delivering a table to analysts, set up governance:

-- 1. Add tags for data classification
ALTER TABLE prod_retail_banking.gold.risk_scores
  SET TAGS ('domain' = 'risk', 'pii' = 'false', 'sla' = 'daily');

-- 2. Add column-level comments (helps data discovery)
ALTER TABLE prod_retail_banking.gold.risk_scores
  ALTER COLUMN risk_score COMMENT 'Credit risk score 0-1000. Higher = riskier.';

-- 3. Grant access to consumer group
GRANT USE CATALOG ON CATALOG prod_retail_banking TO `risk-analysts`;
GRANT USE SCHEMA   ON SCHEMA  prod_retail_banking.gold TO `risk-analysts`;
GRANT SELECT       ON TABLE   prod_retail_banking.gold.risk_scores TO `risk-analysts`;

-- 4. Verify grants look correct
SHOW GRANTS ON TABLE prod_retail_banking.gold.risk_scores;

📋 Phase 5: Monitoring — UC Health Checks

-- Audit: who queried sensitive tables today?
SELECT
  event_time, user_name, action_name, request_params.table_full_name
FROM system.access.audit
WHERE
  action_name IN ('selectTable', 'commandSubmit')
  AND request_params.table_full_name
      LIKE 'prod_retail_banking.silver.customers%'
  AND DATE(event_time) = CURRENT_DATE()
ORDER BY event_time DESC;

-- Check all tables and their owners
SELECT table_catalog, table_schema, table_name, table_owner, created_by
FROM system.information_schema.tables
WHERE table_catalog = 'prod_retail_banking';

-- Find tables with no comments (documentation debt)
SELECT table_name FROM system.information_schema.tables
WHERE table_catalog = 'prod_retail_banking'
  AND comment IS NULL;
Chapter 8
Data Lineage & Audit Logs
Automatic tracking — the compliance superpower

🔗 Data Lineage — Automatic

Unity Catalog automatically captures column-level lineage every time you run a Spark/SQL query. No configuration needed — it just works.

What it tracks:

  • Which tables were read to produce this table
  • Which notebooks/jobs wrote to a table
  • Column-level: risk_score was derived from credit_history.score
  • Upstream + downstream impact analysis

Banking use case: A data quality issue in transactions_raw — UC lineage shows all 12 downstream tables affected in seconds.

📊 system.access.audit

Unity Catalog writes ALL data access events to a system table you can query with SQL. Available to all UC users.

Key columns:

  • event_time — exact timestamp
  • user_name — who accessed
  • action_name — selectTable, createTable, dropTable…
  • request_params — full query context
  • workspace_id — which workspace

Banking use case: RBI audit asks for all accesses to customer PII tables in Q3 → single SQL query, 5 minutes.

Chapter 9
Syntax Patterns — Remember Them Easily
Every UC SQL command with a memorable pattern

The Master Pattern Card

-- CREATE pattern: what → name → options → location
CREATE [CATALOG|SCHEMA|TABLE|VIEW|VOLUME|FUNCTION] <3-part-name> [options];

-- GRANT pattern: privilege → object_type → name → who
GRANT <privilege> ON <type> <name> TO <principal>;

-- ACCESS pattern (always 3 levels: catalog → schema → table)
GRANT USE CATALOG ... → GRANT USE SCHEMA ... → GRANT SELECT ...

-- SHOW pattern: what you want to inspect
SHOW [CATALOGS|SCHEMAS IN <catalog>|TABLES IN <schema>|GRANTS ON <object>];

-- USE pattern: set default context to avoid repeating catalog/schema
USE CATALOG <catalog>; USE SCHEMA <schema>;

🔍 Discovery Commands

SHOW CATALOGS;
SHOW SCHEMAS IN prod_retail_banking;
SHOW TABLES  IN prod_retail_banking.gold;
SHOW VOLUMES IN prod_retail_banking.bronze;
DESCRIBE TABLE EXTENDED
  prod_retail_banking.gold.risk_scores;
SHOW GRANTS ON CATALOG prod_retail_banking;
SELECT CURRENT_METASTORE();
SELECT CURRENT_CATALOG();

🏷️ Tag & Document

-- Tag a table
ALTER TABLE prod_retail_banking.silver.customers
  SET TAGS ('pii'='true', 'domain'='retail');

-- Add column comment
ALTER TABLE prod_retail_banking.silver.customers
  ALTER COLUMN pan_number
  COMMENT 'PAN card number - DPDP sensitive';

-- Search by tag in catalog UI
SELECT * FROM system.information_schema.tables
WHERE array_contains(tags, 'pii');

🧠 Memory Trick — The 3-Level Access Staircase

Always climb 3 stairs:
Stair 1: GRANT USE CATALOG → enter the building
Stair 2: GRANT USE SCHEMA → take the elevator to the floor
Stair 3: GRANT SELECT/MODIFY → open the specific room
Miss any stair → error! The staircase is not optional.

Chapter 10
Interview Insights — Q&A with Model Answers
Direct, indirect, and scenario-based questions
Q1 (Direct): "What is Unity Catalog and why would you use it over Hive Metastore?"
✅ Model Answer: "Unity Catalog is Databricks's unified governance layer that provides a single metastore shared across all workspaces. The key difference from Hive Metastore is that Hive is workspace-isolated — tables in one workspace aren't visible in another, there's no audit log, no lineage, and no centralized permissions. Unity Catalog solves all of this: one catalog across all workspaces, automatic data lineage, SQL-based access control at catalog/schema/table/column level, and a system audit log. In banking, this is critical for regulatory compliance — we need to prove who accessed customer data, when, and trace where risk models get their data from."
Q2 (Scenario): "A new analyst joins your bank's data team. Walk me through how you'd give them access to the Gold risk tables."
✅ Model Answer: "First, I'd ensure the analyst's Azure AD account is in the right Databricks group — say risk-analysts. Then in Unity Catalog I'd run three grants in order: GRANT USE CATALOG ON CATALOG prod_retail_banking TO 'risk-analysts', then GRANT USE SCHEMA ON SCHEMA prod_retail_banking.gold TO 'risk-analysts', then GRANT SELECT ON TABLE prod_retail_banking.gold.risk_scores TO 'risk-analysts'. The key thing is the order — you must grant catalog access first, then schema, then table. If there's sensitive PAN data in any of those tables, I'd also check if column masking is applied so they only see masked values. Finally I'd verify with SHOW GRANTS ON TABLE."
Q3 (Indirect — "Describe your data pipeline"): They're listening for whether you use 3-part table names and mention governance
✅ What to say: "...we write to prod_retail_banking.bronze.transactions_raw using saveAsTable with the full 3-part name so Unity Catalog registers it. Downstream, the silver job reads from the same UC-registered table, which means lineage is automatically captured. We use Delta Lake format throughout so UC can track column-level lineage too. Before the gold table goes to analysts, we apply column masking for PII fields and set row filters for branch isolation..."
Q4 (Tricky): "What's the difference between a managed table and external table in Unity Catalog? When would you use each?"
✅ Model Answer: "In Unity Catalog, a managed table means UC owns both the metadata and the underlying data files — when you DROP TABLE, both are deleted. An external table means you own the files in ADLS; dropping the table only removes the metadata, files remain. In banking, I use external tables for Bronze — raw data from core banking systems lands in ADLS and we register it, but we never want a DROP command accidentally deleting raw data. For Gold tables I use managed tables — UC owns the lifecycle, storage is abstracted, and it's simpler to manage. The tradeoff is that external tables require an External Location and Storage Credential to be set up first."
Q5 (Scenario): "Your RBI auditor asks: show me everyone who accessed the customers table containing Aadhaar data in the last 3 months."
✅ Model Answer: "With Unity Catalog this is straightforward. I'd query the system.access.audit table: SELECT event_time, user_name, action_name FROM system.access.audit WHERE request_params.table_full_name LIKE '%customers%' AND event_time >= DATEADD(month, -3, CURRENT_DATE()). This gives a complete timestamped log of every access — including reads. Without Unity Catalog, this would be impossible as Hive Metastore doesn't track access events."
