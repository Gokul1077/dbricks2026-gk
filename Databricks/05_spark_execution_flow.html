📖 Chapter 0 · Prerequisites

What You Must Know First

Before understanding Spark execution flow, you need to be comfortable with these concepts:

🗂️ Distributed Computing

What: Instead of one powerful machine processing all data, work is split across many machines.
Why Spark? Banks process terabytes of transactions daily – one server cannot handle this. Spark splits the work.

🧠 RDD – Resilient Distributed Dataset

What: Spark's core data structure. An RDD is an immutable, fault-tolerant collection of data split into partitions and distributed across cluster nodes.
Memory trick: Resilient = recovers from failure · Distributed = across nodes · Dataset = collection of records

📦 Partition

What: A logical chunk of data. If you have 10 million bank transactions and 10 partitions, each partition holds ~1 million records.
Key rule: One Task processes one Partition. This is the atomic unit of Spark work.

🔄 Transformations vs Actions

Transformations (lazy) → filter(), map(), groupBy() — builds a plan, does nothing yet.
Actions (trigger execution) → count(), show(), write() — this is when a Job is created!

🧠 Memory Trick – Lazy Evaluation

Think of transformations like a recipe. Writing down steps doesn't cook food. Only when you call the waiter (action) does the kitchen actually start cooking. Spark waits for an action before executing any transformation.

🌐 Chapter 1 · Overview

Big Picture: What is Apache Spark?

Apache Spark is an in-memory distributed data processing engine. It processes data stored across many machines in parallel, making it 100x faster than traditional MapReduce for many workloads.

Banking Example: A bank runs a nightly Spark job on Azure Databricks to process 50 million daily transactions — detecting fraud, calculating interest, and generating regulatory reports — all in under 30 minutes.

🚀 Chapter 2 · Spark Application

Spark Application — The Outermost Container

Spark Application

A Spark Application is the entire program you submit to the cluster. It lives as long as your program runs. It has two key components:

👑 Driver (The Boss)

  • Runs the main() of your application
  • Creates the SparkContext / SparkSession
  • Builds the execution plan (DAG)
  • Coordinates all Executors
  • Collects final results

Banking analogy: The Branch Manager who assigns work to tellers.

⚙️ Executors (The Workers)

  • JVM processes on each worker node
  • Actually run the Tasks
  • Each has its own memory and CPU cores
  • Cache data in memory for fast re-use
  • Report status back to Driver

Banking analogy: Bank tellers who each handle a set of customer transactions.

SparkSession — Entry Point

Every Spark application starts with creating a SparkSession. On Azure Databricks, it is automatically available as spark.

# On Databricks – SparkSession auto-created as 'spark'
# On standalone cluster or local – create it manually:
from pyspark.sql import SparkSession

spark = SparkSession.builder \
    .appName("BankFraudDetection") \
    .config("spark.executor.memory", "8g") \
    .config("spark.executor.cores", "4") \
    .getOrCreate()

# SparkContext is accessible via:
sc = spark.sparkContext
print(sc.appName)  # BankFraudDetection

🧠 Memory Trick – Application

Think of a Spark Application like a Bank Branch operation for the day. The Driver = Branch Manager, Executors = Tellers, SparkSession = Opening the branch. The branch runs all day (many jobs), then closes (application ends).

💼 Chapter 3 · Jobs

Job — Triggered by an Action

Every time you call an Action on a DataFrame or RDD, Spark creates a Job. A single application can have many jobs, running sequentially or in parallel.

Banking Example — Multiple Jobs in One Application

# SPARK APPLICATION: Daily Bank Reconciliation

# Load data (no job yet)
transactions_df = spark.read.format("delta") \
    .load("abfss://bronze@bankdatalake.dfs.core.windows.net/transactions")

# --- JOB 1 triggered by .count() ---
total_txns = transactions_df.count()  # ← Action → JOB 1
print(f"Total transactions today: {total_txns}")

# --- JOB 2 triggered by .show() ---
fraud_df = transactions_df \
    .filter(transactions_df.amount > 100000) \
    .filter(transactions_df.is_international == "Y")

fraud_df.show(10)  # ← Action → JOB 2

# --- JOB 3 triggered by .write() ---
fraud_df.write.format("delta") \
    .mode("append") \
    .save("abfss://gold@bankdatalake.dfs.core.windows.net/fraud_alerts")
# ← Action → JOB 3

🧠 Memory Trick – Job

"Action = Job submission"
Think of an Action like pressing Submit on a banking form. Nothing happens until you press Submit. The moment you press it (call an action), a Job starts.

🎬 Chapter 4 · Stages

Stage — Divided by Shuffle Boundaries

A Job is divided into Stages. The key concept here is the Shuffle. Spark splits stages wherever data needs to be physically moved across nodes — called a shuffle.

✅ Narrow Transformations

Each partition's output depends on only one input partition. No data movement across nodes. These stay in the SAME Stage.

Examples: filter(), select(), map(), withColumn()

🏦 Banking: Filter transactions where amount > 10000 (each partition filters independently)

🔀 Wide Transformations (Shuffle)

Partition output depends on multiple input partitions. Data is shuffled across nodes. This creates a NEW Stage.

Examples: groupBy(), join(), orderBy(), distinct(), repartition()

🏦 Banking: Group by account_id to sum all transactions (data from many partitions must come together)

DAG — Directed Acyclic Graph

Spark builds a DAG (Directed Acyclic Graph) of all transformations before executing. The DAG Scheduler then identifies shuffle boundaries and creates Stages.

🧠 Memory Trick – Stages & Shuffle

"Shuffle = Stage boundary = Data highway crossing"
Imagine bank employees in 3 floors. Work within a floor = same stage (narrow). Moving files between floors = shuffle = new stage (wide). Every time data must cross to another node, a new stage begins.

Code Example: Stages in Banking

# Stage 1: Narrow transformations (filter + withColumn)
flagged = transactions_df \
    .filter(transactions_df.amount > 10000) \              # narrow
    .withColumn("currency_flag",                              # narrow
                F.when(transactions_df.currency != "INR", "FOREIGN")
                 .otherwise("DOMESTIC"))

# Stage 2: Wide transformation (groupBy = SHUFFLE)
summary = flagged \
    .groupBy("account_id", "currency_flag") \    # WIDE → new Stage
    .agg(
        F.sum("amount").alias("total_amount"),
        F.count("*").alias("txn_count")
    )

# Stage 3: Another narrow + write action
summary.filter(F.col("total_amount") > 500000) \         # narrow
       .write.format("delta").mode("overwrite") \
       .save("abfss://gold@bankdatalake.dfs.core.windows.net/high_risk")
# ↑ write() = Action = triggers Job with 3 stages
⚡ Chapter 5 · Tasks

Task — The Smallest Unit of Work

A Task is the actual work unit that runs on a single Executor Core. Each Task processes exactly one Partition.

🔑 Golden Rule

1 Partition = 1 Task = 1 Executor Core

If your DataFrame has 200 partitions, Spark creates 200 Tasks for that Stage. If you have 50 executor cores, they run 50 tasks in parallel, cycling through all 200.

Partitioning in Code

# Check current number of partitions
print(transactions_df.rdd.getNumPartitions())  # e.g., 200

# Repartition: increases partitions (involves shuffle!)
df_repartitioned = transactions_df.repartition(400)

# Coalesce: reduces partitions (NO shuffle, faster)
df_coalesced = transactions_df.coalesce(50)

# Best practice: align partitions to executor cores
# If you have 50 cores → use multiples of 50 (200, 400…)
df_tuned = transactions_df.repartition(200, F.col("account_id"))
# ↑ partitions by account_id for balanced distribution

🧠 Memory Trick – Tasks

"Task = One Teller serving One Customer"
One bank teller (Executor Core) handles one customer at a time (one Partition = one Task). More tellers = more tasks done in parallel = faster processing.

☁️ Chapter 7 · Azure Databricks

Spark Execution on Azure Databricks — Banking Pipeline

Azure Databricks is Microsoft Azure's managed Apache Spark platform. It handles cluster provisioning, scaling, and optimization automatically.

Azure Databricks Components

  • Workspace — collaborative notebooks, jobs, clusters
  • Cluster — Spark Driver + Worker nodes (Executors)
  • Job Cluster — ephemeral, created per job run (cost-effective)
  • Interactive Cluster — always on for development
  • Databricks Runtime — optimized Spark with Delta Lake built-in
  • Unity Catalog — governance & data lineage

ADLS Gen2 Integration

  • Data stored in Azure Data Lake Storage Gen2
  • Databricks reads via abfss:// protocol
  • Delta Lake format on top of ADLS = ACID + versioning
  • Service Principal / Managed Identity for auth
  • Medallion Architecture: Bronze → Silver → Gold

Full Banking Pipeline — Azure Databricks Notebook

# ============================================================
# Bank Transaction Risk Scoring Pipeline – Azure Databricks
# SparkSession is auto-available as 'spark' in Databricks
# ============================================================
from pyspark.sql import functions as F
from pyspark.sql.window import Window

# ── STEP 1: Read from Bronze (raw ADLS) ──────────────────────
transactions = spark.read \
    .format("delta") \
    .load("abfss://bronze@mybank.dfs.core.windows.net/transactions")

customers = spark.read \
    .format("delta") \
    .load("abfss://bronze@mybank.dfs.core.windows.net/customers")

# ── STEP 2: Narrow Transformations → STAGE 1 ─────────────────
cleaned = transactions \
    .filter(F.col("status") == "COMPLETED") \         # narrow
    .withColumn("txn_date", F.to_date("txn_timestamp")) \  # narrow
    .withColumn("hour", F.hour("txn_timestamp"))           # narrow

# ── STEP 3: Shuffle – Join with customers → NEW STAGE ────────
# JOIN = Wide transformation = shuffle boundary
enriched = cleaned.join(customers, on="customer_id", how="left")

# ── STEP 4: Another Shuffle – Aggregate → NEW STAGE ──────────
window_spec = Window.partitionBy("customer_id") \
                    .orderBy("txn_timestamp") \
                    .rowsBetween(-29, 0)

risk_df = enriched \
    .withColumn("rolling_30d_total",
                F.sum("amount").over(window_spec)) \      # wide
    .withColumn("risk_score",
                F.when(F.col("rolling_30d_total") > 1000000, "HIGH")
                 .when(F.col("rolling_30d_total") > 100000, "MEDIUM")
                 .otherwise("LOW"))                          # narrow

# ── STEP 5: ACTION triggers JOB → Write to Gold ──────────────
risk_df.write \
    .format("delta") \
    .mode("overwrite") \
    .partitionBy("txn_date") \
    .option("mergeSchema", "true") \
    .save("abfss://gold@mybank.dfs.core.windows.net/risk_scores")
# ↑ write() → JOB created → DAG → 3+ Stages → Tasks per partition

# ── VIEW EXECUTION PLAN ───────────────────────────────────────
risk_df.explain(True)  # See DAG, stages, physical plan

🎯 Chapter 8 · Interview Insights

Interview Insights — How to Answer These Questions

Q1: "Explain Spark's execution model."

✅ Structured Answer:

"When we write a Spark program on Azure Databricks, it runs as a Spark Application managed by a Driver node. The Driver creates a SparkSession and builds a logical plan called the DAG as we chain transformations. Nothing executes until we call an Action like .write() or .count() — that triggers a Job. The DAG Scheduler then splits each Job into Stages, where stage boundaries are determined by shuffle operations like joins and group-bys. Each Stage is finally broken into Tasks, one per partition, and those Tasks are distributed to Executor Cores across the cluster to run in parallel."

Q2: "What is a shuffle and why does it matter?"

✅ Structured Answer:

"A shuffle happens when data needs to be physically redistributed across nodes — for example during a groupBy or join. It's the most expensive operation in Spark because it involves disk I/O and network transfer. In our banking fraud pipeline, when we groupBy(account_id) to sum transactions, all records for the same account_id must land on the same partition — that's a shuffle. To minimize shuffles, we use techniques like broadcast joins for small tables, or partition the data by join key upfront."

Q3 (Indirect): "Why is my Spark job slow?" (They mean: data skew, too few partitions, too many shuffles)

✅ Structured Answer:

"I'd start by checking the Spark UI on Databricks — looking at task duration variance in a Stage. If some tasks take 10x longer than others, that's data skew — one partition has far more data. I'd also check if we're doing too many wide transformations that cause unnecessary shuffles, or if the number of partitions is misconfigured relative to executor cores. In production banking pipelines, we also look at shuffle read/write sizes to detect expensive joins."

Q4: "What is the difference between repartition and coalesce?"

✅ Structured Answer:

"repartition(N) reshuffles all data across N partitions and involves a full shuffle — it can both increase and decrease partition count, and gives even distribution. coalesce(N) only reduces partitions by merging them without a full shuffle — it's faster but may produce uneven partition sizes. For our banking pipeline writing to Delta Lake, we use coalesce() before writing if we want fewer output files, but repartition() when we need balanced distribution for downstream processing."

⚡ Possible Indirect Questions

  • "How many tasks will run if I have 200 partitions and 50 executor cores?" → 200 tasks, running 50 at a time (4 waves)
  • "Why does my groupBy create a new stage?" → Because groupBy is a wide transformation that causes a shuffle = stage boundary
  • "What happens if I call .count() twice?" → Two separate Jobs are created unless you cache/persist the DataFrame first
  • "What's the role of the Driver in failure recovery?" → Driver re-submits failed tasks; if Driver fails, the whole application fails
  • "In Databricks, what manages the cluster?" → Databricks Runtime + Azure Resource Manager; Cluster Manager handles resource allocation
