1. Why Spark? — Motivation

Problems with Hadoop MapReduce

Before Spark, Hadoop MapReduce was the dominant big-data processing engine. But it had serious problems:

😓 Difficult to Program

Even a simple Word Count required 61 lines of Java in Hadoop MR. The same job in Spark is just 1 line in the interactive shell.

🐢 Performance Bottlenecks

Every MapReduce job writes data to disk 3+ times. A typical Hive query compiles to 3–5 MR jobs, meaning 18–30 disk scans per query.

🔁 Bad for Iterative Jobs

Machine learning algorithms (e.g. gradient descent) need many passes over data. MR re-reads from disk every iteration — very slow.

Slide from deck — Hadoop MR (61 lines) vs Spark (1 line) for Word Count

What Spark Offers

⚡ Lazy Computations

Spark does NOT execute transformations immediately. It builds a plan and optimises before running — only when an action is triggered.

🧠 In-Memory Caching

Scan HDFS only once, then serve subsequent operations from RAM. Dramatically reduces I/O.

🔗 Efficient Pipelining

Spark chains multiple operations inside a single stage, avoiding unnecessary disk writes between steps.

🗣️ Interview-க்கு தமிழில் புரிந்துகொள்ளுங்கள் (Tamil Explanation)

Hadoop MapReduce-ல் ஒவ்வொரு step-லயும் data-ஐ hard disk-ல write பண்ணும் — அது மிகவும் slow. ஒரு Hive query run ஆகும்போது 18–30 முறை disk scan நடக்கும். Spark இதை சரி செய்தது: Lazy evaluation (உடனடியா run பண்ணாம plan மட்டும் வைக்கும்), in-memory caching (RAM-ல data வைத்து வேலை செய்யும்), மற்றும் pipelining (disk touch பண்ணாம operations chain ஆகும்). இதனால Spark, MapReduce-ஐ விட 10x–100x வேகமாக இயங்கும்.

2. RDD — Resilient Distributed Dataset

RDD is the fundamental data structure in Spark. Every piece of data you process in Spark lives in an RDD (or is built on top of it).

Simple View

An RDD is a collection of data items split into partitions and stored in memory across the worker nodes of the cluster.

Complex (Real) View

An RDD is actually an interface for data transformation. It doesn't necessarily hold data — it knows how to compute data. It can refer to:

  • Data in a persistent store (HDFS, Cassandra, HBase)
  • Data in cache (memory / disk)
  • Another RDD (lineage)

RDD Metadata (5 key properties)

PropertyWhat it stores
PartitionsSet of data splits associated with this RDD
DependenciesList of parent RDDs involved in computing this one
ComputeFunction to compute each partition given its parent partitions
Preferred LocationsBest place to run computation — for data locality
PartitionerHow data is split across partitions (Hash, Range, Custom)

Resilient — Fault Tolerance

If a partition is lost (node crash), Spark recomputes it from the lineage — it doesn't need to replicate data like HDFS does. The dependency chain (DAG) is the recovery mechanism.

Two Classes of RDD Operations

🔄 Transformations

Produce a new RDD from an existing one. They are lazy — only change metadata, no actual computation yet.

Examples: map(), filter(), flatMap(), reduceByKey()

⚡ Actions

Trigger actual execution. Once an action is called, Spark evaluates the entire DAG built so far.

Examples: count(), collect(), saveAsTextFile(), foreach()

Lazy Computation Model — Transformations only update metadata; the Action triggers real execution

Code Example — Basic RDD Operations

# Create an RDD from a text file
lines = sc.textFile("hdfs://path/to/file.txt")

# Transformation: split each line into words (lazy - nothing runs yet)
words = lines.flatMap(lambda line: line.split(" "))

# Transformation: map each word to (word, 1) tuple
pairs = words.map(lambda word: (word, 1))

# Transformation: sum counts per key
counts = pairs.reduceByKey(lambda a, b: a + b)

# ACTION: this triggers all the lazy transformations above
counts.saveAsTextFile("hdfs://output/wordcount")
# PySpark (Spark 2.x+) - Using SparkContext directly
from pyspark import SparkContext

sc = SparkContext("local[4]", "MyApp")

rdd = sc.parallelize([1, 2, 3, 4, 5], numSlices=2)
result = rdd.filter(lambda x: x % 2 == 0).collect()
print(result)  # [2, 4]
🗣️ RDD — தமிழ் விளக்கம்

RDD என்பது "Resilient Distributed Dataset" — cluster-ல உள்ள multiple machines-ல பிரிந்து (partitions) இருக்கும் data collection. இதில் 2 முக்கியமான operations: Transformations — "இப்படி செய்" என்று plan மட்டும் வைக்கும் (lazy), உடனே execute ஆகாது. Actions — இதை call பண்ணும்போது மட்டுமே actual computation நடக்கும். ஒரு node crash ஆனா, Spark, lineage (parent RDD chain) பார்த்து அந்த partition-ஐ மறுபடியும் compute பண்ணும் — இதுவே "Resilient".

3. DAG — Directed Acyclic Graph

DAG is the execution plan Spark builds from your transformations. It tells Spark how to compute the final result step by step.

Node

Each node in the DAG is an RDD partition.

Edge

Each edge is a transformation applied on a partition (map, filter, etc.)

Acyclic

The graph never loops back — data flows forward only, enabling fault recovery via lineage.

Directed

Each transformation moves data from state A → state B. Direction is always forward.

WordCount DAG — each line is a transformation; foreach is the action that triggers execution

WordCount DAG visualised — HDFS → RDD → flatMap → map → reduceByKey (SHUFFLE) → foreach

Same DAG split into Stage 1 (before shuffle) and Stage 2 (after shuffle)

DAG Scheduler

The DAG Scheduler inside the Driver takes the DAG and:

1
Splits DAG into StagesEach stage contains transformations that can be pipelined without a shuffle.
2
Identifies Stage BoundariesWide transformations (like reduceByKey, join) require a shuffle — these form stage boundaries.
3
Submits Tasks to Task SchedulerEach partition in a stage becomes one Task. The Task Scheduler sends these to Executors.

Narrow vs Wide Dependencies

✅ Narrow Dependency

Each parent partition is used by at most one child partition. No shuffle needed.
Examples: map, filter, flatMap

⚠️ Wide Dependency

Multiple child partitions depend on one parent partition. Requires shuffle across the network.
Examples: reduceByKey, groupByKey, join

🗣️ DAG — தமிழ் விளக்கம்

DAG என்பது ஒரு plan (வழிமுறை) — எந்த order-ல operations நடக்கணும் என்று Spark-க்கு சொல்கிறது. இதில் Stage என்பது shuffle இல்லாம ஒரே sequence-ல run ஆகும் operations குழு. Shuffle boundary-ல (reduceByKey போன்றவை), ஒரு stage முடியும், புது stage ஆரம்பிக்கும். Narrow dependency = ஒரு parent partition, ஒரு child partition-க்கு மட்டும் — fast, no shuffle. Wide dependency = ஒரு parent partition, பல child partitions-க்கு செல்லும் — slow, network shuffle தேவை.

4. Spark Cluster Architecture

A Spark cluster has two main roles — the Driver and the Executors.

Spark Cluster — Detailed: Client Node (Driver JVM + SparkContext) & Worker Nodes (Executor JVMs with Tasks)

Spark Cluster — PySpark: Python Driver + Worker Monitor + separate Python Worker Processes on each Executor

Driver

The Driver is the brain of the Spark application. It runs on the Client Node (or a dedicated master node).

  • Entry point of the Spark Shell (Scala, Python, R)
  • Creates and hosts the SparkContext
  • Translates the RDD lineage into an execution graph (DAG)
  • Splits the DAG into Stages
  • Schedules Tasks and controls their execution on Executors
  • Stores metadata about all RDDs and their partitions
  • Hosts the Spark Web UI (port 4040)

Executor

Executors are JVM processes running on Worker Nodes that do the actual computation.

  • Stores data in cache (JVM heap or disk)
  • Reads data from external sources (HDFS, S3, Cassandra, etc.)
  • Writes output data to external storage
  • Runs the actual Tasks sent by the Driver
  • Multiple Executors can run on a single Worker Node
  • Each Executor can run multiple Tasks in parallel (up to spark.executor.cores)

SparkContext

# Spark 1.x / low-level
from pyspark import SparkContext, SparkConf

conf = SparkConf().setAppName("MyApp").setMaster("yarn")
sc = SparkContext(conf=conf)

# Spark 2.x+ (preferred) — SparkSession wraps SparkContext
from pyspark.sql import SparkSession

spark = SparkSession.builder     .appName("MyApp")     .config("spark.executor.memory", "4g")     .config("spark.executor.cores", "2")     .getOrCreate()

sc = spark.sparkContext  # access low-level SparkContext if needed

Key Configuration Parameters

ParameterWhat it controls
spark.driver.memoryRAM for the Driver JVM (e.g. 2g)
spark.executor.memoryRAM per Executor JVM (e.g. 4g, default 512MB)
spark.executor.coresCPU cores per Executor (controls task parallelism)
spark.task.cpusCPU cores each individual Task requires
spark.local.dirDirectory for shuffle / spill files on disk

PySpark — Extra Layer

When you use PySpark, there is an extra layer: the Python Driver process communicates with the Driver JVM via a Worker Monitor. On each Executor, a separate Python Worker Process runs per task — the JVM and Python talk via sockets. This adds overhead, which is why PySpark can be slower than Scala for certain operations.

🗣️ Architecture — தமிழ் விளக்கம்

Spark cluster-ல இரண்டு main roles: Driver — boss மாதிரி. Plan செய்யும், stages பிரிக்கும், tasks schedule பண்ணும், Web UI காட்டும். Executor — worker மாதிரி. Driver சொன்ன tasks execute பண்ணும், data cache-ல வைக்கும், disk-ல write பண்ணும். ஒரு Worker Node-ல பல Executors இருக்கலாம். ஒரு Executor-ல பல Tasks parallel-ஆ run ஆகும். PySpark-ல ஒரு extra step இருக்கும் — Python process, JVM Executor-ஓட socket மூலம் பேசும். அதனால சின்ன overhead வரும்.

5. Application Decomposition

Every Spark program has a clear hierarchy:

A
Application A single instance of SparkContext. Contains the full data processing logic. Can schedule multiple Jobs sequentially or in parallel (SparkContext is thread-safe).
J
Job A complete set of transformations on an RDD that ends with an Action (count(), save(), collect()). Each action triggers one Job.
S
Stage A set of transformations that can be pipelined and executed by a single independent worker without a shuffle. Stage boundaries are at wide transformations (shuffle points).
T
Task The execution of one stage on a single data partition. It is the basic unit of scheduling. If a stage has 100 partitions → 100 tasks are created.

Summary Table

LevelTriggered byScope
ApplicationSparkContext creationWhole program lifetime
JobAction (count, save)One DAG evaluation
StageDAG Scheduler (at shuffle boundary)Group of pipeline-able operations
TaskTask Scheduler (one per partition)One partition, one stage
# This single script = 1 Application
spark = SparkSession.builder.getOrCreate()
sc = spark.sparkContext

# Job 1: triggered by count() action
rdd = sc.textFile("hdfs://data/logs.txt")  # lazy
filtered = rdd.filter(lambda x: "ERROR" in x)  # lazy
error_count = filtered.count()  # ACTION → triggers Job 1

# Job 2: triggered by saveAsTextFile() action
pairs = filtered.map(lambda x: (x.split()[0], 1))
grouped = pairs.reduceByKey(lambda a, b: a+b)  # SHUFFLE → new stage
grouped.saveAsTextFile("hdfs://output/")  # ACTION → triggers Job 2
🗣️ Application Decomposition — தமிழ் விளக்கம்

Spark Application → Job → Stage → Task — இது hierarchy. Application: SparkContext ஆரம்பிக்கும்போது தொடங்கும், முழு program. Job: ஒவ்வொரு Action (count, save) ஒரு Job ஆகும். Stage: Shuffle இல்லாம ஒரே pipeline-ல நடக்கும் operations grouping. Shuffle வந்தா, stage boundary. Task: ஒரு partition-ஐ process பண்றது = 1 Task. 100 partitions இருந்தா, 100 tasks parallel-ஆ run ஆகும்.

6. Persistence & Executor Memory

Executor Memory Layout — JVM Heap is divided into Storage, Unroll, Shuffle, and Safe regions

Executor Memory Regions (JVM)

RegionSizeConfig KeyPurpose
JVM Heap512MB defaultspark.executor.memoryTotal JVM heap size
Safe90% of Heapspark.storage.safetyFractionUsable memory (10% reserved for JVM overhead)
Storage60% of Safespark.storage.memoryFractionRDD caching / persistence
Unroll20% of Storagespark.storage.unrollFractionDeserializing block to check if it fits in memory
Shuffle20% of Safespark.shuffle.memoryFractionData during shuffle operations

Persistence Levels

LevelDescription
MEMORY_ONLYStore as deserialized Java objects in JVM. If no space, partition is recomputed. Default
MEMORY_AND_DISKStore in JVM; if no space, spill to disk. Reads from disk when needed.
MEMORY_ONLY_SERStore as serialized byte arrays. More space-efficient, more CPU-intensive.
MEMORY_AND_DISK_SERSerialized in memory, spills to disk instead of recomputing.
DISK_ONLYStore partitions only on disk. Slowest option.
MEMORY_ONLY_2 etc.Same as above but replicated on 2 nodes for fault tolerance.

Spark's Memory as LRU Cache

Spark treats the storage memory as a cache with LRU (Least Recently Used) eviction. When memory is full, the least recently used partition is evicted. If disk is involved, it's spilled there instead of being dropped.

from pyspark import StorageLevel

rdd = sc.parallelize(range(1000))

# Cache in memory (default MEMORY_ONLY)
rdd.cache().count()

# Custom persistence level — memory first, spill to disk, serialized
rdd.persist(StorageLevel.MEMORY_AND_DISK_SER).count()

# Free memory when done
rdd.unpersist()
🗣️ Persistence & Memory — தமிழ் விளக்கம்

Executor-ல JVM Heap memory பல பகுதிகளாக பிரிந்திருக்கும்: Storage — RDD cache வைக்க (60%). Shuffle — shuffle நடக்கும்போது data வைக்க (20%). MEMORY_ONLY: RAM-ல மட்டும் வைக்கும், பாழாய்போனா recompute பண்ணும். MEMORY_AND_DISK: RAM நிரம்பினா disk-ல spill பண்ணும் — slow ஆனா data போகாது. Spark memory ஒரு LRU cache மாதிரி — பழைய data out, புது data in. Interview-ல கேட்டா: "cache() vs persist()" — cache() = MEMORY_ONLY shortcut; persist() = custom StorageLevel choice.

7. Spark Shuffle

A shuffle is the redistribution of data across partitions — data moves over the network between Executors. It is the most expensive operation in Spark.

Spark has three shuffle implementations:

Hash Shuffle (pre-1.2) Sort Shuffle (default now) Tungsten Sort Shuffle (optimised)

① Hash Shuffle (Legacy)

Each map task creates one output file per reducer. So if there are M mapper tasks and R reducers:

Total files = M × R

With many mappers and reducers, this creates a massive number of tiny files on disk — huge I/O pressure.

② Hash Shuffle with Consolidation

Hash Shuffle With Consolidation — one file per core (not per task), reducing file count

Consolidation groups mapper output files by executor core instead of individual tasks. File count reduces from M×R to (cores_per_executor × R). Better, but still scales poorly.

③ Sort Shuffle (Default since 1.2)

Each map task writes one sorted output file with an index file. Reducers fetch only their range from this single sorted file. Total files = 2M (one data + one index per mapper).

Internally, it uses an AppendOnlyMap in memory. When memory fills up, data is sorted and spilled to disk. Multiple spills are merged using a MinHeap merge at the end.

# Sort shuffle is default. To tune it:
spark = SparkSession.builder   .config("spark.shuffle.manager", "sort")   .config("spark.shuffle.memoryFraction", "0.2")   .config("spark.local.dir", "/mnt/fast-ssd/spark-shuffle")   .getOrCreate()

④ Tungsten Sort Shuffle (Optimised)

Tungsten is Spark's memory management overhaul. Tungsten Sort Shuffle works off-heap — it stores data as raw binary in a LinkedList<MemoryBlock>, with a compact array of (data pointer, partition ID) pairs as long[].

Benefits:

No JVM GC pressure

Data stored outside JVM heap — no garbage collection pauses on shuffle data.

Cache-friendly sorting

Sorting is done on the compact pointer array (longs), not on actual objects — CPU cache-friendly.

Efficient merging

Output files with per-partition sections are merged into a final consolidated file with an index.

Shuffle Comparison

TypeFiles CreatedMemory StructureBest For
Hash ShuffleM × RIn-memory hash mapVery few reducers
Hash + Consolidationcores × RIn-memory hash mapModerate reducers
Sort Shuffle2M (data + index)AppendOnlyMap + sort/spillGeneral use Default
Tungsten Sort2M (merged)Off-heap binary + pointer sortHigh performance, large shuffles
🗣️ Shuffle — தமிழ் விளக்கம்

Shuffle என்பது data-ஐ different machines-க்கு இடையே network வழியா move பண்றது. இது மிகவும் expensive operation — disk write + network transfer + disk read நடக்கும். Hash Shuffle: ஒவ்வொரு mapper task-உம் ஒவ்வொரு reducer-க்கும் ஒரு file எழுதும் — M×R files! பல files ஆனா I/O பிரச்சனை. Sort Shuffle (default): ஒவ்வொரு mapper-உம் ஒரே ஒரு sorted file + index file எழுதும் — மிகவும் efficient. Tungsten: JVM heap-க்கு வெளியே off-heap memory-ல store பண்ணும் — GC இல்லை, fast sorting. Interview-ல: "reduceByKey vs groupByKey" — reduceByKey shuffle data குறைக்கும் (combiner போல); groupByKey எல்லா data-வையும் shuffle பண்ணும் — avoid பண்ணணும்.

8. Spark DataFrame

DataFrame Idea — multiple language frontends (SQL, Python, Scala, R) compiled to a single logical plan, then optimised for multiple backends

What is a DataFrame?

A DataFrame is an RDD with a schema — field names, field data types, and statistics. It provides a higher-level, SQL-like API on top of RDDs.

Interface

Unified API

Same transformations work in SQL, Python, Java, Scala, and R. All transformations are compiled down to JVM operations — even PySpark DataFrames run JVM-side.

RDD Interop

A DataFrame can be converted to an RDD of Row objects for low-level operations if needed.

Catalyst Optimizer

The Catalyst query optimizer rewrites and optimises the logical plan before execution — joins, filters, projections are reordered for efficiency.

Internals

Row-Columnar Format

Data is stored in a row-columnar format internally. Chunk size controlled by spark.sql.inMemoryColumnarStorage.batchSize.

Partition Pruning

Each column in each partition stores min-max statistics. Spark skips entire partitions if a filter can't match (pushdown).

Better Compression

Columnar layout allows better compression ratios than standard RDDs because similar data is stored together.

Faster Column Subsets

When you select only 3 out of 50 columns, Spark reads only those columns — much faster than row-based RDDs.

Code Examples

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, count, when

spark = SparkSession.builder.appName("DFDemo").getOrCreate()

# Read from HDFS / S3 / Delta Lake
df = spark.read.parquet("hdfs://data/transactions/")

# DataFrame transformations (lazy)
result = df   .filter(col("amount") > 1000)   .groupBy("customer_id")   .agg(count("*").alias("txn_count"))

# Action: triggers execution
result.show()

# SQL style — same performance as DataFrame API
df.createOrReplaceTempView("transactions")
spark.sql("""
    SELECT customer_id, COUNT(*) as txn_count
    FROM transactions
    WHERE amount > 1000
    GROUP BY customer_id
""").show()
# DataFrame → RDD (low-level access if needed)
rdd_of_rows = df.rdd
first_row = rdd_of_rows.first()
print(first_row["customer_id"])

# Schema inspection
df.printSchema()
df.describe().show()  # min, max, count, stddev per column

🗣️ DataFrame — தமிழ் விளக்கம்

DataFrame = RDD + Schema (column names + types). RDD-ல நாம் manually transformation எழுதணும், optimizer எதுவும் help பண்ணாது. DataFrame-ல Catalyst Optimizer automatically query-ஐ optimize பண்ணும் — filter pushdown, join reorder எல்லாம் automatic. Tungsten memory engine, data-ஐ off-heap binary format-ல store பண்ணும் — GC இல்லை, fast. Column-wise storage (columnar format) — 50 columns-ல 3 மட்டும் select பண்ணினா, மூன்று columns மட்டும் read ஆகும் — மிகவும் efficient. Interview-ல: "DataFrame uses Catalyst optimizer + Tungsten execution engine" — இது key answer.
