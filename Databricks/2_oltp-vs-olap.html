Phase 01 · Foundation → Concept #1

OLTP vs OLAP
Why Analytics Needs a Different Stack

The root reason data engineering exists. Understand this and every tool — Spark, Databricks, Delta Lake — starts to make sense.

⚡ Interview-Ready Explanations Included
The Big Picture

Start with a Story — Why Two Systems?

Imagine you run an e-commerce company like Amazon. Two very different things happen every second:

OPERATIONAL (OLTP)
  • A customer places an order
  • Payment is processed
  • Inventory count decreases by 1
  • A shipping record is created
  • User updates their address
ANALYTICAL (OLAP)
  • What were total sales last quarter?
  • Which product category is trending?
  • Which cities have the most returns?
  • What's the avg order value by age group?
  • Predict next month's demand
💡 Simple Analogy

OLTP is like the cash register at a store — handles one transaction at a time, fast, precise.
OLAP is like the accountant's monthly report — reads everything, finds patterns, never writes new sales.

These two needs are so fundamentally different that trying to use the same database for both causes serious problems. That's exactly why analytics needs its own stack.

Real SQL Examples

See the Difference in Actual Queries

Same database, completely different query patterns — this is why they need different storage engines.

🔴 OLTP Query — Place an Order
-- Touches 1 row. Needs to be FAST.
-- Runs thousands of times per second

INSERT INTO orders (
  order_id, user_id,
  product_id, amount, status
)
VALUES (
  10291, 4451,
  8823, 499.00, 'pending'
);

-- Immediate consistency required
-- Must not fail silently
🔵 OLAP Query — Monthly Sales Report
-- Scans MILLIONS of rows
-- Runs once a day, result cached

SELECT
  DATE_TRUNC('month', order_date)
    AS month,
  category,
  SUM(amount) AS total_revenue,
  COUNT(*) AS order_count
FROM orders
JOIN products USING (product_id)
WHERE order_date >= '2024-01-01'
GROUP BY 1, 2
ORDER BY total_revenue DESC;
💡 Key Insight

The OLTP query touches 1 row and needs to complete in 5ms. The OLAP query touches 50 million rows and 5 seconds is fine. Optimizing one database for both is like designing a car that's both a racing car and a truck — it ends up being bad at both.

The Core Problem

Why Running Analytics on OLTP Breaks Everything

This is the "cause" that created the entire data engineering industry. When companies tried to run analytical queries on their production OLTP databases, three things broke:

🐢
Performance Collapse
Row-based storage reads entire rows even when you only need 2 columns. A query on 50M orders reads all columns — 10× slower than needed.
💥
Kills Production DB
A heavy analytics query locks tables and hogs CPU/IO. Your checkout page starts timing out because a dashboard is running in the background.
📏
Can't Scale to History
OLTP databases keep only current data. Analyzing 5 years of orders across 200M rows? The normalized schema requires expensive JOINs across 10+ tables.
💡 Real-World Incident Pattern

A BI analyst runs SELECT * FROM orders WHERE year = 2023 on the production MySQL DB at 9am. The query takes 8 minutes. During those 8 minutes, customers can't check out. This is the exact incident that forced companies to build separate analytical infrastructure.

How the Analytics Stack Evolved

The solution was to copy data out of OLTP into a separate system optimized for reads. This gave birth to the entire data engineering pipeline.

The Technical Why

Row Storage vs Columnar Storage — The Hidden Reason

This is the technical core. OLTP and OLAP use fundamentally different ways to store data on disk, which is why you can't swap one for the other.

ROW STORAGE (OLTP)
  • Stores all columns of a row together on disk
  • Fast for reading/writing entire rows (one record at a time)
  • Inserting a new order = write one block
  • Analytics = reads ALL columns even if you need only 2
  • Examples: PostgreSQL, MySQL
Disk: [id=1, name="Ali", age=28, city="Chennai", amount=499]
       [id=2, name="Raj", age=31, city="Mumbai", amount=299]
COLUMNAR STORAGE (OLAP)
  • Stores each column together on disk, separately
  • Analytics query needs only "amount"? Reads only that column
  • Massive compression — same values cluster together
  • Writes are slower — but analytics is read-heavy
  • Examples: Parquet (Delta Lake), BigQuery, Snowflake
id col:    [1, 2, 3, 4, 5 ...]
amount col: [499, 299, 799, 149 ...]
city col:   [Chennai, Mumbai, Delhi ...]
💡 Why This Matters for Databricks

Databricks stores data in Parquet format inside Delta Lake — which is columnar. When you run SELECT SUM(amount) FROM orders on 100M rows, Databricks reads only the amount column file — not names, cities, or emails. This is why Databricks queries are so fast on large data.

Real World Scenarios

Which System Handles What?

🔴 Use OLTP For
User signs up on your app

Write 1 row to users table. Needs immediate consistency. If it fails, user can't login. → PostgreSQL

🔵 Use OLAP For
Product team asks: "What % of users who signed up in Jan 2024 made a purchase within 7 days?"

Scan 2M user rows + 10M event rows. Join, filter, aggregate. Takes 4 seconds. Fine. → Databricks

🔴 Use OLTP For
Bank processes a money transfer

Debit one account, credit another. Must be atomic — if step 2 fails, step 1 must roll back. ACID transaction. → MySQL / Oracle

🔵 Use OLAP For
Finance team: "Total transaction volume by city for Q3, broken by customer segment"

Historical scan of 500M transactions. GROUP BY, multi-dimension aggregation. Data freshness: next-day is fine. → Redshift / BigQuery / Databricks

Interview Prep

Answer These Without Hesitation

Most-asked interview questions on this topic
Q: "What is the difference between OLTP and OLAP?"
Answer: OLTP is designed for transactional operations — fast reads and writes on individual rows, used by application backends. OLAP is designed for analytical queries — aggregating large volumes of historical data to produce business insights. They differ in schema design, storage engine, query patterns, and the users who consume them.
Q: "Why can't we run analytics directly on the production database?"
Answer: Three reasons — first, heavy analytical queries compete with write transactions and degrade production performance. Second, OLTP uses row-based storage which is inefficient for scanning millions of rows. Third, OLTP databases only hold current state, while analytics needs years of historical data with a denormalized schema for efficient aggregations.
Q: "What is a Lakehouse and how does it relate to OLAP?"
Answer: A Lakehouse combines the scalability and cost of a data lake (store raw files in S3/ADLS) with the ACID transactions and query performance of a data warehouse. Databricks with Delta Lake is a Lakehouse — it stores data in columnar Parquet format with transactional guarantees, serving both batch OLAP analytics and streaming workloads from a single platform.
Q: "What is columnar storage and why does it matter for analytics?"
Answer: Columnar storage keeps each column's values together on disk. When an analytics query aggregates only 2–3 columns out of 50, it reads only those column files — skipping 95% of data. This makes aggregation queries 10–100× faster and allows high compression since similar values are adjacent. Parquet (used by Delta Lake in Databricks) is columnar.
