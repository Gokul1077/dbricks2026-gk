Designing a Data Governance Architecture:
Azure → Databricks

Security · Lineage · Compliance · Auditability
Beginner → Intermediate Azure + Databricks Interview Focus

Your company is moving multiple datasets from Azure SQL Database, Azure Blob Storage, and Azure Data Lake Storage Gen2 (ADLS Gen2) into Databricks for analytics and machine learning. The leadership team is concerned about data security, lineage, and compliance.

Question: How would you design a data governance architecture to ensure that data ingestion from Azure to Databricks is secure, auditable, and compliant?

01
🧭 WHAT IS DATA GOVERNANCE? WHAT

Data Governance is a framework of policies, processes, roles, and technologies that ensures data is accurate, secure, accessible, and compliant throughout its lifecycle — from ingestion to consumption.

In simple terms: It's the "rules of the road" for your data — who can see it, who can touch it, where it came from, and where it's going.

Think of an Investment Bank's Trade Lifecycle. When a trader executes a deal, every step — from order placement to settlement — is recorded, audited, and governed by compliance rules (MiFID II, Basel III). Nobody can just "move money" without a paper trail. Similarly, in data governance, no data moves without access controls, audit logs, and lineage tracking.

The three pillars the leadership team cares about map directly to real concerns:

🔐 Security
  • Who can access which data?
  • Is data encrypted in transit & at rest?
  • Are secrets stored securely?
🔍 Lineage
  • Where did data originate?
  • What transformations happened?
  • Which dashboards use this data?
📋 Compliance
  • GDPR / CCPA data residency rules
  • PII masking & classification
  • Retention & deletion policies
📓 Auditability
  • Who accessed what and when?
  • Who changed the schema?
  • Full audit trail for regulators
🗂️ Data Quality
  • Is ingested data clean?
  • Schema validation checks
  • Null/duplicate detection
02
❓ WHY DO WE NEED IT? WHY

Without governance, you risk:

  • 🚨 Data breaches — unauthorized access to sensitive customer or financial data
  • 💸 Regulatory fines — GDPR violations can cost up to 4% of global revenue
  • 🌀 Data swamps — data lakes with no cataloguing become unusable
  • 🔎 Untraceable errors — a wrong model result with no lineage = impossible to debug
  • 📉 Trust erosion — leadership stops trusting analytics if data quality is unknown

In investment banking, a rogue trader scenario (like Nick Leeson at Barings Bank) happens when there's no governance over who can do what. Data governance prevents the equivalent — a rogue pipeline that overwrites production data, deletes PII without consent, or silently corrupts ML training sets.

1
SECURE THE IDENTITY LAYER (Azure AD + Managed Identity)

Never use username/password for service connections. Use Azure Managed Identity — a service principal that Azure manages automatically. Databricks uses this identity to talk to ADLS Gen2, SQL DB, and Key Vault without embedding any secrets in your code.

# Mount ADLS Gen2 using Managed Identity (no secrets in code) configs = { "fs.azure.account.auth.type": "OAuth", "fs.azure.account.oauth.provider.type": "org.apache.hadoop.fs.azurebfs.oauth2.ClientCredsTokenProvider", "fs.azure.account.oauth2.client.id": dbutils.secrets.get("kv-scope", "client-id"), "fs.azure.account.oauth2.client.secret": dbutils.secrets.get("kv-scope", "client-secret"), "fs.azure.account.oauth2.client.endpoint": "https://login.microsoftonline.com/<tenant-id>/oauth2/token" } dbutils.fs.mount( source = "abfss://raw@mydatalake.dfs.core.windows.net/", mount_point = "/mnt/raw", extra_configs = configs )
2
STORE SECRETS IN AZURE KEY VAULT (Never Hardcode!)

All connection strings, passwords, and tokens go into Azure Key Vault. Databricks connects to Key Vault via a Secret Scope. This way, auditors can see who accessed which secret, and rotating a password doesn't require redeploying code.

💡 Interview Tip: Always mention Key Vault + Secret Scopes when asked about secrets management. This shows security maturity.
3
INGEST USING AZURE DATA FACTORY + AUTO LOADER

Use Azure Data Factory (ADF) to orchestrate bulk or scheduled moves from Azure SQL DB and Blob. For streaming/incremental files landing in ADLS Gen2, use Databricks Auto Loader — it automatically detects new files and ingests them incrementally.

# Auto Loader — incrementally ingest new files from ADLS Gen2 df = (spark.readStream .format("cloudFiles") .option("cloudFiles.format", "parquet") .option("cloudFiles.schemaLocation", "/mnt/checkpoints/schema") .load("/mnt/raw/transactions/") ) (df.writeStream .format("delta") .option("checkpointLocation", "/mnt/checkpoints/txn") .outputMode("append") .table("bronze.transactions") )
4
GOVERN WITH UNITY CATALOG (The Central Governance Layer)

Unity Catalog is Databricks' unified governance solution. It provides a three-level namespace: catalog.schema.table. It enforces:

  • ✅ Fine-grained access control — column-level security, row filters, data masking
  • ✅ Automated data lineage — tracks where data came from and where it flows
  • ✅ Centralized audit logs — every read/write logged with user + timestamp
  • ✅ Data classification — tag PII columns (e.g., email, SSN) for masking
-- Unity Catalog: Grant column-level access GRANT SELECT (trade_id, amount, currency) ON TABLE finance.silver.trades TO `analyst-group@company.com`; -- Row-level filter: analysts see only their own desk's data CREATE ROW ACCESS POLICY desk_filter AS (desk_id STRING) RETURNS BOOLEAN RETURN desk_id = current_user_attribute('desk');
5
IMPLEMENT MEDALLION ARCHITECTURE IN DELTA LAKE

Store data in Bronze → Silver → Gold layers on ADLS Gen2 using Delta format. Delta's ACID transactions and Time Travel are key governance features — you can audit what data looked like at any point in time.

Medallion = Trade Settlement Lifecycle: Bronze = raw trade order (unverified), Silver = validated and enriched trade, Gold = settled, aggregated P&L report ready for the risk desk. Each layer has progressively stricter quality and access controls.

6
INTEGRATE MICROSOFT PURVIEW FOR ENTERPRISE LINEAGE

Microsoft Purview is Azure's enterprise data catalog. It scans Azure SQL, Blob, ADLS Gen2, and connects with Unity Catalog to provide an end-to-end lineage map — from source systems to dashboards. Compliance teams can see the full data journey without opening Databricks.

7
ENABLE ENCRYPTION + NETWORK SECURITY
  • 🔒 In-transit: TLS 1.2+ enforced on all connections
  • 🔒 At-rest: Azure Storage encryption (AES-256, Microsoft-managed or customer-managed keys via Key Vault)
  • 🌐 Network: Use Azure Private Link + VNet injection for Databricks so traffic never traverses public internet
  • 🛡️ Databricks IP Access Lists: Whitelist approved corporate IPs only
04

🎤 INTERVIEW INSIGHTS

DIRECT QUESTION
"How would you design a data governance architecture for Azure to Databricks ingestion?"
STRUCTURED ANSWER
Formula: Problem → Pillars → Tools → Architecture → Trade-offs

"I would approach this in layers. First, for security, I'd use Azure AD Managed Identity and Azure Key Vault to avoid any hardcoded credentials, combined with Private Link to keep traffic off public internet. For ingestion, I'd use Azure Data Factory for batch and Databricks Auto Loader for incremental file ingestion, landing data in a Bronze layer on ADLS Gen2 in Delta format. For governance, I'd enable Unity Catalog as the single control plane — it gives fine-grained access control, automatic lineage, and audit logs at no extra cost. I'd also connect Microsoft Purview for enterprise-wide lineage visibility. Data flows through a Medallion architecture — Bronze for raw, Silver for cleaned, Gold for business-ready. Delta's Time Travel ensures we can always audit historical states for compliance. Finally, I'd enforce column masking on PII fields like email and SSN, and use row-level security to restrict analysts to their permitted data subsets."
INDIRECT / TRICK QUESTION
"A compliance officer asks: 'Can you show me who accessed our customer PII data last Tuesday?' — how would you fulfill this?"
STRUCTURED ANSWER
"Unity Catalog automatically logs every data access event — table reads, writes, schema changes — with the username, timestamp, and query. I would query the system.access.audit table in Unity Catalog, filtering by date and the specific table containing PII. I could also cross-reference with Azure Monitor logs for the Key Vault access that day. This gives a complete audit trail meeting GDPR's right-to-audit requirements."
SCENARIO QUESTION
"What's the difference between Unity Catalog and Microsoft Purview? Why would you use both?"
STRUCTURED ANSWER
"Unity Catalog is Databricks-native — it governs everything inside the Databricks platform: who can query which table, column masking, row filters, and fine-grained lineage within Databricks jobs. Microsoft Purview is Azure's enterprise catalog — it scans across Azure SQL, Blob, ADLS, Databricks, and even on-premises systems to give business stakeholders a single map of all data assets. Think of Unity Catalog as your internal trading desk's risk system, and Purview as the bank-wide compliance reporting system. They complement each other — Unity Catalog for enforcement, Purview for enterprise visibility."

🧠 QUIZ TIME

A data engineer hardcodes the Azure SQL Database password directly inside a Databricks notebook to speed up development. The security team flags this as a critical risk.

Which TWO changes should be made to fix this the RIGHT way?
A) Move the password to an environment variable inside the notebook cell
B) Store the password in Azure Key Vault and reference it via a Databricks Secret Scope
C) Encrypt the password using a custom Base64 encoder and hardcode the encoded value
D) Use Azure Managed Identity so no password is needed at all — Azure handles authentication automatically
✅ Correct: B and D

B (Azure Key Vault + Secret Scope) removes credentials from code entirely. The notebook calls dbutils.secrets.get("scope", "key") — the secret is never visible in logs or version control.

D (Managed Identity) is even better — it eliminates passwords entirely. Azure issues short-lived tokens automatically, and there's nothing to rotate or leak.

❌ A is wrong — environment variables in notebooks are still visible to anyone who can run the cell.
❌ C is wrong — Base64 is encoding, NOT encryption. It provides zero security.
