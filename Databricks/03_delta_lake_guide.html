🔧 Prerequisites — Must Know Before Delta Lake
Parquet · HDFS vs Object Storage · ACID · Data Lake problems · Spark basics
Why This Matters

Delta Lake is built on top of Parquet files and solves the core problems of raw Data Lakes. To truly understand Delta Lake, you need to know what came before and why it was broken.

Parquet Format

📦 What is Parquet?

Parquet is a columnar binary file format — data is stored column-by-column instead of row-by-row. This makes analytics queries (reading 2 out of 100 columns) extremely fast.

Row format (CSV):
Row1: [id=1, name="Alice", amount=5000, city="Chennai"]
Row2: [id=2, name="Bob", amount=3000, city="Mumbai"]
Columnar format (Parquet):
id col: [1, 2, 3, 4 ...]
name col: ["Alice","Bob" ...]
amount col: [5000, 3000 ...]
If you only need amount, Parquet reads ONLY the amount column from disk. CSV reads ALL columns. On 1 TB data with 100 columns, this is a 50x speedup.
Data Lake Problems

🚨 Why Raw Parquet on a Lake Fails

  • No ACID: Two writers simultaneously → corrupted data
  • No schema check: Anyone can write garbage columns
  • No "undo": Deleted data is gone forever
  • No upserts: Can't update a specific row in Parquet
  • Small files: Thousands of tiny files → slow queries
  • No history: Can't query "what did data look like last week?"
💡 Delta Lake solves ALL of these by adding a _delta_log transaction log on top of Parquet files.
ACID — Quick Recap

⚗️ ACID Properties (you must know this cold)

A — Atomicity

All or nothing. A fund transfer either fully completes or fully rolls back. No partial state.

C — Consistency

DB moves from one valid state to another. Total balance never changes mid-transfer.

I — Isolation

Concurrent transactions don't see each other's in-progress changes. Readers see a stable snapshot.

D — Durability

Once committed, data survives a crash. Written to durable storage (ADLS / disk).

Spark Basics Recap

⚡ Spark Concepts Needed for Delta Lake

SparkSession

Entry point to Spark. Lives on driver. You create it once per application.

DataFrame

Distributed table-like structure. Lazy — nothing runs until you call an action like .write or .show().

Partition

A logical chunk of a DataFrame. Each partition is processed by one Spark task on one core.

Action vs Transform

Transforms are lazy (filter, select). Actions trigger execution (write, count, show).

ADLS Gen2

Azure Data Lake Storage. Where Delta table files physically live. Accessed via abfss:// path.

Catalog / Metastore

Metadata registry. Knows table names → ADLS paths. Unity Catalog in Databricks.

01
What is Delta Lake?
ACID on your Data Lake · _delta_log · Delta vs plain Parquet · Interview Q: "What problem does Delta Lake solve?"
Concept Snapshot

🔷 One-liner definition

Delta Lake is an open-source storage framework (created by Databricks) that adds a transaction log (_delta_log) on top of Parquet files stored in a Data Lake (ADLS Gen2) — giving you ACID transactions, schema enforcement, time travel, and upserts on cheap cloud storage.

🏦 Bank Analogy: Plain Parquet files on ADLS = a bank vault with no ledger. Delta Lake = that same vault but now with a complete audit ledger — every deposit/withdrawal is recorded, you can roll back mistakes, and two tellers can't corrupt the same account simultaneously.
Internals

🔧 How Delta Lake Actually Works — The _delta_log

When you write a Delta table, Spark creates two things in your ADLS path:

abfss://bronze@bankdatalake.dfs.core.windows.net/transactions/
  ├── part-00001-abc123.snappy.parquet ← actual data file (version 0 write)
  ├── part-00002-def456.snappy.parquet ← actual data file (version 0 write)
  ├── part-00003-ghi789.snappy.parquet ← actual data file (version 1 append)
  └── _delta_log/ ← transaction log folder
      ├── 00000000000000000000.json ← version 0: initial write
      ├── 00000000000000000001.json ← version 1: append
      ├── 00000000000000000002.json ← version 2: MERGE (update)
      └── 00000000000000000010.checkpoint.parquet ← checkpoint every 10 versions

📄 version 0 (initial write)

metaData: schema, partitionBy, config
+ add: part-00001.parquet (stats: 50K rows)
+ add: part-00002.parquet (stats: 50K rows)

📄 version 2 (MERGE update)

- remove: part-00001.parquet (has changed rows)
+ add: part-00004.parquet (rewritten with updates)
💡 Key insight: Delta Lake NEVER modifies existing Parquet files. Updates work by marking old files as removed in the log and writing new files with the changes. The old files remain on disk until you run VACUUM. This is how time travel works!
Delta vs Plain Parquet

⚖️ Delta Table vs Plain Parquet — Side by Side

Schema Enforcement❌ Anyone writes anything✅ Rejects mismatched schema writes
Row Updates/Deletes❌ Impossible natively✅ MERGE, UPDATE, DELETE

🎤 Interview Insights — "What problem does Delta Lake solve?"

Q: What problem does Delta Lake solve?
Plain Data Lakes on ADLS suffer from four critical problems: (1) No ACID transactions — concurrent writes corrupt data. (2) No schema enforcement — garbage data gets ingested. (3) No update/delete capability — Parquet files are immutable. (4) No history — you can't audit or recover from bad writes. Delta Lake adds a transaction log (_delta_log) on top of Parquet files, solving all four problems without changing the cheap object storage underneath.
Q: Is Delta Lake a replacement for ADLS or Parquet?
No. Delta Lake is a storage framework that sits ON TOP of ADLS. It still stores data as Parquet files on ADLS — it just adds a _delta_log transaction log folder alongside them. Think of ADLS as the vault and Delta Lake as the ledger system for that vault. They work together, not as alternatives.
🚨 Mistake: Saying "Delta Lake replaces Parquet" — Delta Lake uses Parquet as its storage format. Delta IS Parquet + transaction log.

PySpark Code

💻 Creating Delta Tables — All Methods

# ─────────────────────────────────────────────────────────────────
# METHOD 1: Write DataFrame as Delta (external path on ADLS)
# ─────────────────────────────────────────────────────────────────
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, current_date

spark = SparkSession.builder.appName("DeltaDemo").getOrCreate()

# Sample banking transactions DataFrame
data = [(1, "ACC001", 50000.0, "COMPLETED", "Chennai"),
        (2, "ACC002", 12000.0, "PENDING",   "Mumbai"),
        (3, "ACC003", 75000.0, "COMPLETED", "Delhi")]
cols = ["txn_id", "account_no", "amount", "status", "city"]
df = spark.createDataFrame(data, cols)

# Write as Delta — creates Parquet files + _delta_log
delta_path = "abfss://silver@bankdatalake.dfs.core.windows.net/transactions"

df.write \
  .format("delta") \
  .mode("overwrite") \          # options: overwrite, append, ignore, error
  .save(delta_path)

# ─────────────────────────────────────────────────────────────────
# METHOD 2: Register as a named table in Unity Catalog / Hive Metastore
# ─────────────────────────────────────────────────────────────────
df.write \
  .format("delta") \
  .mode("overwrite") \
  .saveAsTable("banking_prod.silver.transactions")
  # catalog.schema.table  ←  Unity Catalog format

# ─────────────────────────────────────────────────────────────────
# METHOD 3: SQL DDL — CREATE TABLE in Databricks notebook
# ─────────────────────────────────────────────────────────────────
spark.sql("""
CREATE TABLE IF NOT EXISTS banking_prod.silver.transactions (
    txn_id     BIGINT,
    account_no STRING,
    amount     DOUBLE,
    status     STRING,
    city       STRING
)
USING DELTA
LOCATION 'abfss://silver@bankdatalake.dfs.core.windows.net/transactions'
COMMENT 'Silver layer — cleaned transaction records'
""")

# ─────────────────────────────────────────────────────────────────
# METHOD 4: Read back the Delta table
# ─────────────────────────────────────────────────────────────────
# By path:
df_read = spark.read.format("delta").load(delta_path)

# By table name (requires metastore registration):
df_read = spark.read.table("banking_prod.silver.transactions")

# SQL:
df_read = spark.sql("SELECT * FROM banking_prod.silver.transactions WHERE status = 'COMPLETED'")
df_read.show()

DeltaTable API

🔗 DeltaTable.forPath() — The Python Object for Delta Operations

DeltaTable.forPath() gives you a Python object representing an existing Delta table — it's the entry point for MERGE, UPDATE, DELETE, VACUUM, OPTIMIZE, and history operations.

from delta.tables import DeltaTable

# Get a handle to the Delta table object
delta_txn = DeltaTable.forPath(spark, delta_path)

# OR by name (if registered in metastore)
delta_txn = DeltaTable.forName(spark, "banking_prod.silver.transactions")

# Inspect the table
print(delta_txn.history().show())    # all commit history
print(delta_txn.detail().show())     # location, format, row count

🎤 Interview Insights — Create & Write

Q: What is the difference between .save(path) and .saveAsTable(name)?
.save(path) writes data to ADLS at that path but does NOT register the table in any metastore — only people who know the exact ADLS path can read it. .saveAsTable(name) both writes the data AND registers it in Unity Catalog/Hive Metastore — anyone can query it by name using SQL or spark.read.table(). In production, always use saveAsTable with Unity Catalog for governance.
Q: When would you use append mode vs overwrite for a banking transaction table?
Overwrite replaces all data — use it for dimension tables (like dim_customer) during a full refresh. Append adds only new records — use it for fact tables (like transactions) during daily incremental loads. Overwriting a transaction table with 3 years of history every day would be extremely wasteful and slow. Always use append or MERGE for large fact tables.
🚨 Mistake: Using overwrite mode for production fact tables — this is a common junior mistake that causes full data refresh every run, wasting hours of compute.

MERGE (Upsert) Operation
DeltaTable.merge() · whenMatchedUpdate · whenNotMatchedInsert · SCD Type 1 & 2
Concept

🔷 What is MERGE (Upsert)?

MERGE = Upsert = Update if exists + Insert if new. It's the most powerful Delta Lake operation — it compares a source DataFrame against the target Delta table and decides what to do for each row.

🏦 Bank Analogy: Every day new customer data arrives. For existing customers, update their address. For new customers, add their record. For customers who've been deleted, remove them. MERGE does all three in one atomic operation.

Code

💻 MERGE — PySpark and SQL Examples

# ─────────────────────────────────────────────────────────────────
# SCENARIO: Daily transaction status update
# - If txn_id exists and status changed → UPDATE
# - If txn_id is new → INSERT
# ─────────────────────────────────────────────────────────────────
from delta.tables import DeltaTable

# Target: existing Delta table
delta_txn = DeltaTable.forPath(spark, delta_path)

# Source: new updates arriving today
new_updates = spark.createDataFrame([
    (2, "ACC002", 12000.0, "COMPLETED", "Mumbai"),  # existing — status changed
    (4, "ACC004", 8500.0,  "PENDING",   "Pune")     # new record
], ["txn_id", "account_no", "amount", "status", "city"])

# MERGE operation
delta_txn.alias("target") \
  .merge(
    new_updates.alias("source"),
    "target.txn_id = source.txn_id"   # match condition (join key)
  ) \
  .whenMatchedUpdate(set={               # row exists in both → UPDATE
    "status": "source.status",
    "amount": "source.amount"
  }) \
  .whenNotMatchedInsertAll() \           # new row in source only → INSERT all cols
  .execute()


# ─────────────────────────────────────────────────────────────────
# SCD TYPE 1 — Overwrite existing (no history kept)
# Use case: Fix typo in customer name
# ─────────────────────────────────────────────────────────────────
delta_customers.alias("t").merge(
  new_customers.alias("s"),
  "t.customer_id = s.customer_id"
).whenMatchedUpdateAll() \               # update ALL columns
 .whenNotMatchedInsertAll() \
 .execute()


# ─────────────────────────────────────────────────────────────────
# SCD TYPE 2 — Keep full history (add new row, close old row)
# Use case: Customer moves from Chennai to Bangalore
# ─────────────────────────────────────────────────────────────────
from pyspark.sql.functions import current_date, lit

delta_dim_customer.alias("t").merge(
  new_addresses.alias("s"),
  "t.customer_id = s.customer_id AND t.is_current = true AND t.city != s.city"
).whenMatchedUpdate(set={               # close old row
    "is_current": lit("false"),
    "valid_to":   current_date()
}).whenNotMatchedInsert(values={        # add new row with new city
    "customer_id": "s.customer_id",
    "city":        "s.city",
    "is_current":  lit("true"),
    "valid_from":  current_date(),
    "valid_to":    lit(None)
}).execute()


# ─────────────────────────────────────────────────────────────────
# SQL MERGE (equivalent, use in Databricks SQL cells)
# ─────────────────────────────────────────────────────────────────
spark.sql("""
MERGE INTO banking_prod.silver.transactions AS target
USING new_updates_view AS source
ON target.txn_id = source.txn_id
WHEN MATCHED THEN
  UPDATE SET target.status = source.status,
             target.amount = source.amount
WHEN NOT MATCHED THEN
  INSERT (txn_id, account_no, amount, status, city)
  VALUES (source.txn_id, source.account_no, source.amount,
          source.status, source.city)
""")

🎤 Interview Insights — MERGE

Q: How do you implement SCD Type 2 using Delta Lake?
SCD Type 2 tracks full history by adding new rows instead of overwriting. I use a Delta MERGE with three conditions: (1) WHEN MATCHED AND the tracked attribute changed → UPDATE the old row to set is_current=false and valid_to=today. (2) WHEN NOT MATCHED BY TARGET → INSERT the new row with is_current=true, valid_from=today. The match condition includes is_current=true so we only compare against active records. This gives a full audit trail — critical in banking for regulatory reporting on customer data at any point in time.
Q: What is the difference between whenMatchedUpdate and whenMatchedUpdateAll?
whenMatchedUpdateAll() updates ALL columns in the target with corresponding source values — equivalent to `SET * = source.*`. whenMatchedUpdate(set={}) lets you specify exactly which columns to update and with what expressions — more fine-grained control. Use UpdateAll for simple SCD Type 1. Use specific set={} when you need partial updates or computed values like audit timestamps.
🚨 Mistake: Using whenMatchedUpdateAll for SCD Type 2 — this would overwrite the valid_to/is_current columns you're trying to preserve on the old row. Always use explicit set={} for SCD Type 2.

🧠 Quiz

A customer's account status changes from PENDING to COMPLETED. Your Delta table has 1 million rows. Which MERGE clause runs for this customer's row?

  • A. whenNotMatchedInsertAll — it's a new row
  • B. whenMatchedUpdate — the row exists in target (same txn_id), update the status column
  • C. whenNotMatchedBySourceDelete — it was deleted from source
  • D. MERGE fails — Delta doesn't support status updates

✅ B. The match condition (txn_id = txn_id) finds the existing row. Since it MATCHED, the whenMatchedUpdate clause runs, updating only the status column. Delta internally marks the old Parquet file as removed and writes a new file with the updated row.

Time Travel & History
DESCRIBE HISTORY · VERSION AS OF · TIMESTAMP AS OF · RESTORE
Concept

🔷 What is Time Travel?

Delta Lake's Time Travel lets you query any historical version of a table — by version number or timestamp — using the transaction log. Since Delta never deletes old Parquet files until VACUUM runs, every version's data is accessible.

🏦 Bank Analogy: A bank's statement system can show your account balance "as of December 31, 2023" even though it's now 2024. Delta Lake gives you the same capability — query your data as it existed at any past point in time.
How It Works

🔧 Time Travel Internals

Version 0 — Initial Load (Jan 1)

100K rows written. Parquet files: part-0001.parquet, part-0002.parquet

Version 1 — Append (Jan 2)

10K new rows appended. New file: part-0003.parquet added to log

Version 2 — MERGE update (Jan 3)

500 rows updated. part-0001.parquet marked as REMOVED, new part-0004.parquet added

VERSION AS OF 0 → reads part-0001 + part-0002

Original 100K rows — the old part-0001 file is still on ADLS, just "logically removed"

After VACUUM (retention expired)

Old files physically deleted. Time travel to versions before VACUUM cutoff no longer possible.

Code

💻 Time Travel — All Commands

# ─────────────────────────────────────────────────────────────────
# STEP 1: See full history of the table
# ─────────────────────────────────────────────────────────────────
from delta.tables import DeltaTable

delta_txn = DeltaTable.forPath(spark, delta_path)
delta_txn.history().show(truncate=False)

# SQL equivalent:
spark.sql("DESCRIBE HISTORY banking_prod.silver.transactions").show()

# Output columns: version, timestamp, userId, operation, operationParameters
# Example output:
# version | timestamp           | operation | operationMetrics
# 2       | 2024-01-03 10:30:00 | MERGE     | {numUpdated: 500}
# 1       | 2024-01-02 08:00:00 | WRITE     | {numFiles: 1}
# 0       | 2024-01-01 06:00:00 | WRITE     | {numFiles: 2}


# ─────────────────────────────────────────────────────────────────
# STEP 2: Query a specific VERSION
# ─────────────────────────────────────────────────────────────────
# PySpark — by version number
df_v0 = spark.read.format("delta") \
    .option("versionAsOf", 0) \
    .load(delta_path)
print(f"Version 0 row count: {df_v0.count()}")  # 100,000

df_v2 = spark.read.format("delta") \
    .option("versionAsOf", 2) \
    .load(delta_path)
print(f"Version 2 row count: {df_v2.count()}")  # 110,000 (after merge)

# SQL — by version
spark.sql("SELECT * FROM banking_prod.silver.transactions VERSION AS OF 0")

# SQL — by timestamp
spark.sql("SELECT * FROM banking_prod.silver.transactions TIMESTAMP AS OF '2024-01-02'")


# ─────────────────────────────────────────────────────────────────
# STEP 3: RESTORE — Rollback to a previous version
# Use case: A bad pipeline wrote corrupted data — undo it!
# ─────────────────────────────────────────────────────────────────
# Restore by version number
delta_txn.restoreToVersion(1)

# Restore by timestamp
delta_txn.restoreToTimestamp("2024-01-02T08:00:00")

# SQL equivalent:
spark.sql("RESTORE TABLE banking_prod.silver.transactions TO VERSION AS OF 1")
spark.sql("RESTORE TABLE banking_prod.silver.transactions TO TIMESTAMP AS OF '2024-01-02'")

# Important: RESTORE itself creates a new version!
# If you were at version 5 and restored to version 1 → new version 6 is created
# The full history is preserved — this is an auditable rollback.
PySpark + SQL
Time Travel Limits

⚠️ Time Travel — What You Must Know

Default Retention: 30 days

Delta keeps old files for 30 days by default. You can query any version within this window. After VACUUM runs, old files are deleted and time travel to those versions is gone.

# Change retention period
spark.sql("""
ALTER TABLE banking_prod.silver.transactions
SET TBLPROPERTIES (
  'delta.logRetentionDuration' = 'interval 60 days',
  'delta.deletedFileRetentionDuration' = 'interval 60 days'
)
""")

RESTORE creates a new version

RESTORE is non-destructive — it creates a new version entry in the log. Your history from version 1 to 5 is preserved even after restoring to version 1. This is an auditable rollback — important for compliance.

🎤 Interview Insights — Time Travel

Q: How do you recover accidentally deleted data in Delta Lake?
Delta Lake's time travel feature lets me recover from accidental data loss. First, I use DESCRIBE HISTORY to identify the last good version before the deletion. Then I either use spark.read with versionAsOf to read the good version and re-insert the missing rows, or I use RESTORE TABLE to roll the entire table back to that version. RESTORE is preferred for full table recovery since it's atomic and creates a new auditable version in the log.
Q: Can you query yesterday's data in Delta Lake without keeping a separate backup?
Yes, that's exactly what time travel provides. SELECT * FROM transactions TIMESTAMP AS OF '2024-01-01' gives you the table as it existed at midnight on Jan 1, using the transaction log to reconstruct which Parquet files were active at that timestamp. No separate backup is needed — as long as VACUUM hasn't deleted the old files (default 30-day retention).
🚨 Mistake: Forgetting that time travel stops working after VACUUM deletes old files. Always set a sufficiently long retention period before running VACUUM aggressively.
05
VACUUM
Remove old files · Retention threshold · What happens if you VACUUM too aggressively?
Concept

🔷 What is VACUUM?

VACUUM is a Delta Lake maintenance command that physically deletes old Parquet files from ADLS that are no longer part of the current table version and have exceeded the retention threshold. Without VACUUM, your ADLS storage grows unboundedly even though logically the data was "deleted."

🏦 Bank Analogy: VACUUM is like the annual shredding of old bank records. Documents are "logically deleted" from the active system but physically still exist in the archive room (ADLS). VACUUM shreds (physically deletes) files older than the retention period — freeing up space.
Before vs After

🗂️ What VACUUM Does to Your ADLS Files

❌ Before VACUUM (Day 45)

part-0001.parquet (version 0 — 45 days old)
part-0002.parquet (version 0 — 45 days old, logically removed at v2)
part-0003.parquet (version 1 — 44 days old)
part-0004.parquet (version 2 — 2 days old — CURRENT)
part-0005.parquet (version 3 — 1 day old — CURRENT)

Storage: 5 Parquet files (3 obsolete, wasting space)

✅ After VACUUM (30-day retention)

part-0001.parquet → 🗑️ DELETED (45 days old)
part-0002.parquet → 🗑️ DELETED (45 days old)
part-0003.parquet → 🗑️ DELETED (44 days old)
part-0004.parquet ✅ KEPT (2 days old)
part-0005.parquet ✅ KEPT (1 day old)

Storage: 2 Parquet files. Time travel to version 0/1 now IMPOSSIBLE.

Code

💻 VACUUM — Commands and Configuration

# ─────────────────────────────────────────────────────────────────
# STEP 1: DRY RUN first — see what would be deleted (safe preview)
# ─────────────────────────────────────────────────────────────────
spark.sql("""
VACUUM banking_prod.silver.transactions
RETAIN 720 HOURS   -- 30 days = 720 hours
DRY RUN            -- preview only, does NOT delete
""")

# ─────────────────────────────────────────────────────────────────
# STEP 2: Run VACUUM with 30-day retention (default)
# ─────────────────────────────────────────────────────────────────
spark.sql("VACUUM banking_prod.silver.transactions RETAIN 720 HOURS")

# PySpark API equivalent:
from delta.tables import DeltaTable
delta_txn = DeltaTable.forPath(spark, delta_path)
delta_txn.vacuum(720)  # hours


# ─────────────────────────────────────────────────────────────────
# ⚠️  DANGER: VACUUM 0 HOURS — deletes ALL old files immediately
# This DESTROYS time travel! Only do this if you're sure.
# ─────────────────────────────────────────────────────────────────
# First, disable the safety check:
spark.conf.set("spark.databricks.delta.retentionDurationCheck.enabled", "false")
spark.sql("VACUUM banking_prod.silver.transactions RETAIN 0 HOURS")  # ⚠️ IRREVERSIBLE
# Re-enable safety check:
spark.conf.set("spark.databricks.delta.retentionDurationCheck.enabled", "true")


# ─────────────────────────────────────────────────────────────────
# BEST PRACTICE: Set table-level retention properties
# ─────────────────────────────────────────────────────────────────
spark.sql("""
ALTER TABLE banking_prod.silver.transactions
SET TBLPROPERTIES (
  'delta.deletedFileRetentionDuration' = 'interval 30 days',
  'delta.logRetentionDuration'         = 'interval 60 days'
)
""")
# logRetentionDuration: how long to keep _delta_log JSON files
# deletedFileRetentionDuration: how long before VACUUM can delete Parquet files
SQL + PySpark
Aggressive VACUUM Risk

⚠️ What Happens if You VACUUM Too Aggressively?

🚨 VACUUM < 168 hours (7 days) breaks concurrent readers! Active queries reading a snapshot may be referencing Parquet files that VACUUM just deleted, causing FileNotFoundException.

VACUUM RETAIN 0 HOURS risks:

  • All time travel history destroyed
  • Active concurrent reads break
  • Streaming jobs reading Delta can fail
  • Cannot undo a bad MERGE/DELETE

Safe VACUUM practices:

  • Always use DRY RUN first
  • Minimum: RETAIN 168 HOURS (7 days)
  • Production recommendation: 720 HOURS (30 days)
  • Schedule during off-peak hours

🎤 Interview Insights — VACUUM

Q: What happens if you VACUUM too aggressively (e.g., RETAIN 0 HOURS)?
VACUUM with 0 hours retention immediately deletes ALL old Parquet files — including those referenced by active concurrent readers. This can cause FileNotFoundException on live queries. More critically, all time travel history is permanently destroyed — you can no longer restore to any previous version. In banking, this means losing the ability to recover from pipeline errors or regulatory audit queries on historical states. The safe minimum is 168 hours (7 days) to allow concurrent readers to finish, and 720 hours (30 days) is the production standard.
Q: Is VACUUM mandatory? What happens if you never run it?
VACUUM is not mandatory but important for cost management. Without it, every UPDATE/DELETE/MERGE creates new Parquet files while old ones accumulate on ADLS indefinitely. For a busy banking transaction table with millions of daily updates, this means your ADLS storage grows unboundedly and you pay for terabytes of stale files. VACUUM is typically scheduled as a weekly Databricks Workflow job.
🚨 Mistake: Confusing VACUUM with OPTIMIZE — VACUUM deletes old unused Parquet files for storage savings. OPTIMIZE compacts many small Parquet files into fewer large ones for query speed. They solve different problems.

The Problem

🔷 The Small Files Problem

Every Spark job — especially streaming and frequent appends — writes many small Parquet files (sometimes thousands per hour). Querying a table with 100,000 files of 1KB each is far slower than querying 100 files of 128MB each — because Spark creates one task per file, and the overhead of opening/closing thousands of files dominates execution time.

🏦 Bank Analogy: Imagine finding a customer's total balance from 10,000 separate slips of paper (small files) vs from 10 organized ledgers (large files). OPTIMIZE re-organizes the slips into ledgers.

CommandWhat It DoesSolvesWhen to Run

Auto Optimize (Databricks)

🤖 Auto Optimize — Let Databricks Handle It

Databricks offers Auto Optimize — two features that reduce manual maintenance:

Auto Compaction

After each write, Databricks automatically compacts small files produced during that write. No need to manually run OPTIMIZE after every streaming batch.

spark.sql("""
ALTER TABLE transactions
SET TBLPROPERTIES (
  'delta.autoOptimize.autoCompact' = 'true'
)
""")

Optimized Writes

Databricks automatically sizes Parquet files before writing — merges small data during the write itself rather than after. Great for streaming pipelines.

spark.sql("""
ALTER TABLE transactions
SET TBLPROPERTIES (
  'delta.autoOptimize.optimizeWrite' = 'true'
)
""")

🎤 Interview Insights — OPTIMIZE & Z-ORDER

Q: What is the small files problem and how does Delta Lake solve it?
The small files problem occurs when frequent small writes (streaming, micro-batch appends) create thousands of tiny Parquet files. Spark creates one task per file, so scanning 100,000 × 1KB files involves 100,000 tasks with massive scheduling overhead — often slower than scanning 100 × 100MB files. Delta Lake's OPTIMIZE command compacts small files into ~128MB target files by reading them all and rewriting into fewer, larger files. The old small files are marked as removed in _delta_log and can later be cleaned up by VACUUM.
Q: When would you use Z-ORDER vs table partitioning?
Partitioning (by txn_date for example) creates physical folder separation in ADLS — best for high-cardinality columns with coarse filtering (filter by month/year). Z-ORDER reorders data within each partition file — best for high-cardinality columns within a partition (customer_id within a month partition). Z-ORDER doesn't create extra folders, avoids the small partition problem, and can be applied to multiple columns. In practice: partition by date, Z-ORDER by customer_id for a banking transaction table.
🚨 Mistake: Over-partitioning by high-cardinality columns like customer_id — this creates millions of tiny partitions (worse than small files). Z-ORDER is the right tool for high-cardinality filtering.

🎤 Interview Insights — Managed vs External

Q: What happens when you DROP a managed vs external Delta table?
For a managed table, DROP TABLE deletes both the Unity Catalog metadata AND all the underlying Parquet files from ADLS — data is permanently gone. For an external table, DROP TABLE only removes the metadata entry from Unity Catalog — the Delta files on ADLS remain untouched. In production, we always use external tables with explicit ADLS paths to prevent accidental data loss from DROP commands.
🚨 Mistake: Creating production fact tables as managed tables — a single DROP TABLE by any authorized user permanently deletes all banking transaction history. Always use EXTERNAL tables with LOCATION in production.

Schema Enforcement & Evolution
Schema-on-Write · mergeSchema · overwriteSchema · ALTER TABLE ADD COLUMN
Two Behaviors

🔧 Schema Enforcement vs Schema Evolution

Schema Enforcement (Default ON)

Delta Lake rejects any write whose schema doesn't match the existing table schema. Prevents garbage data from corrupting the table. No option needed — it's the default.

# This FAILS if 'fraud_score' column doesn't exist in table
df_with_extra_col.write \
  .format("delta") \
  .mode("append") \
  .save(delta_path)
# → AnalysisException: A schema mismatch detected

Schema Evolution (Opt-in)

Allow the table schema to evolve safely — add new columns from the source, or overwrite the entire schema. Use mergeSchema or overwriteSchema options.

# ALLOW new columns to be added automatically
df_with_extra_col.write \
  .format("delta") \
  .mode("append") \
  .option("mergeSchema", "true") \
  .save(delta_path)
# New column 'fraud_score' added to table schema
# Old rows get NULL for the new column
# ─────────────────────────────────────────────────────────────────
# mergeSchema: ADD new columns (safe — old data gets NULL)
# ─────────────────────────────────────────────────────────────────
df_new_cols.write.format("delta").mode("append") \
  .option("mergeSchema", "true").save(delta_path)

# ─────────────────────────────────────────────────────────────────
# overwriteSchema: REPLACE entire schema (dangerous — use with caution)
# Only works with mode("overwrite")
# ─────────────────────────────────────────────────────────────────
df_totally_new.write.format("delta").mode("overwrite") \
  .option("overwriteSchema", "true").save(delta_path)

# ─────────────────────────────────────────────────────────────────
# ALTER TABLE — Add column explicitly (safest, explicit, auditable)
# ─────────────────────────────────────────────────────────────────
spark.sql("""
ALTER TABLE banking_prod.silver.transactions
ADD COLUMNS (
  fraud_score DOUBLE COMMENT 'ML fraud probability 0-1',
  processed_at TIMESTAMP
)
""")

# SQL auto-enable schema evolution for all writes:
spark.sql("SET spark.databricks.delta.schema.autoMerge.enabled = true")

Delta Lake + Streaming
Delta as a source and sink · Structured Streaming · Event Hubs → Delta
Concept

🔷 Delta as Both Streaming Source and Sink

Delta Lake's transaction log enables it to work natively with Structured Streaming. You can write streaming data directly into a Delta table (sink) OR read a Delta table as a stream — processing each new committed version as a new batch.

🏦 Bank Analogy: Azure Event Hubs receives real-time card transaction swipes. Spark Streaming reads them and writes to a Bronze Delta table. Another streaming job reads that Bronze Delta table and applies fraud scoring — writing results to Silver. The same Delta table serves as both a write target and a change stream source.
Code

💻 Event Hubs → Delta Streaming Pipeline

# ─────────────────────────────────────────────────────────────────
# STREAMING SINK: Write Event Hubs stream → Bronze Delta Table
# ─────────────────────────────────────────────────────────────────
from pyspark.sql.functions import from_json, col
from pyspark.sql.types import StructType, StringType, DoubleType, LongType

# Define schema for incoming transaction events
schema = StructType() \
  .add("txn_id", LongType()) \
  .add("account_no", StringType()) \
  .add("amount", DoubleType()) \
  .add("status", StringType())

# Read from Event Hubs (configured via connection string in Key Vault)
ehConf = {
  "eventhubs.connectionString": spark.conf.get("eventhubs.connectionString")
}
raw_stream = spark.readStream \
  .format("eventhubs") \
  .options(**ehConf) \
  .load() \
  .select(from_json(col("body").cast("string"), schema).alias("data")) \
  .select("data.*")

# Write stream to Bronze Delta table (continuous micro-batches)
query = raw_stream.writeStream \
  .format("delta") \
  .outputMode("append") \
  .option("checkpointLocation", "abfss://checkpoints@bankdatalake.dfs.core.windows.net/bronze-txn") \
  .trigger(processingTime="30 seconds") \  # process every 30s
  .table("banking_prod.bronze.transactions")

query.awaitTermination()


# ─────────────────────────────────────────────────────────────────
# STREAMING SOURCE: Read Delta table as a stream
# Processes each new commit version as a new micro-batch
# ─────────────────────────────────────────────────────────────────
bronze_stream = spark.readStream \
  .format("delta") \
  .table("banking_prod.bronze.transactions")

# Apply transformations and write to Silver
silver_query = bronze_stream \
  .filter(col("amount").isNotNull()) \
  .filter(col("amount") > 0) \
  .writeStream \
  .format("delta") \
  .outputMode("append") \
  .option("checkpointLocation", "abfss://checkpoints@bankdatalake.dfs.core.windows.net/silver-txn") \
  .table("banking_prod.silver.transactions")
PySpark Structured Streaming
💡 checkpointLocation is mandatory for streaming — it records the last processed version so the job can resume from where it left off after a restart. Without it, restarting the job reprocesses all data from the beginning.

Concept

🔷 What is Table Partitioning?

Partitioning physically splits your Delta table data into separate ADLS subfolders by a column's values. When you query with a filter on the partition column, Spark only reads the relevant subfolder — skipping all others entirely (partition pruning).

Code & Structure

💻 Partitioning — Code and ADLS Layout

# Write Delta table partitioned by txn_date
df.write \
  .format("delta") \
  .mode("overwrite") \
  .partitionBy("txn_date") \
  .save("abfss://silver@bankdatalake.dfs.core.windows.net/transactions")
PySpark
transactions/
  ├── txn_date=2024-01-01/ ← partition folder
  │   ├── part-00001.snappy.parquet
  │   └── part-00002.snappy.parquet
  ├── txn_date=2024-01-02/
  │   └── part-00001.snappy.parquet
  └── _delta_log/
Query: WHERE txn_date = '2024-01-01' → Spark reads ONLY the txn_date=2024-01-01/ folder. All other date folders are completely skipped. This is called partition pruning.
When NOT to Partition

⚠️ Partitioning Rules — Common Mistakes

✅ Good Partition Columns

  • Low cardinality (few distinct values)
  • Date, month, year, region, status
  • Columns you frequently filter on
  • Columns with balanced data distribution

Example: PARTITION BY txn_date — only 365 folders per year

❌ Bad Partition Columns

  • High cardinality (millions of distinct values)
  • customer_id, txn_id, email, timestamp
  • Creates millions of tiny partition folders
  • Worse than the small files problem!

Example: PARTITION BY customer_id — 10 million folders, each with 1 row

💡 Pro Tip 1 — VACUUM order: Always run OPTIMIZE before VACUUM. OPTIMIZE creates better-structured files; VACUUM then cleans up the old ones. Never run VACUUM RETAIN 0 HOURS on a production table without a DRY RUN first.

💡 Pro Tip 2 — Z-ORDER columns: Z-ORDER your most selective filter columns — the ones used in WHERE clauses that narrow down results significantly. customer_id, account_no, and txn_date together make great Z-ORDER candidates for a banking table.

💡 Pro Tip 3 — checkpoint location: In streaming jobs, always put the checkpoint in a different ADLS container from your Delta tables. If you accidentally delete the Delta data, you don't want the checkpoint to be gone too.
