🔧 Prerequisites — Foundation Concepts
File Systems · Distributed Systems · Storage basics · What is a schema?
File System (FS)

📁 What is a File System?

A File System is the way an Operating System organizes and stores files on a disk. Think of it like a library — it tracks where each book (file) is placed on the shelves (disk).

  • Examples: NTFS (Windows), ext4 (Linux), FAT32
  • Limitation: Stored on ONE machine — if that machine dies, data is lost.
  • Use case: Laptop files, local databases
Distributed File System (DFS)

🌐 What is a Distributed File System?

A DFS stores files across multiple machines as if they were one giant hard drive. Data is automatically split (chunked) and replicated for fault tolerance.

  • Examples: HDFS (Hadoop), ADLS Gen2 (Azure), S3
  • Key benefit: If one machine fails, data survives on others
  • Use case: Big Data processing with Spark
Schema

📐 What is a Schema?

A schema is the blueprint/structure of your data — it defines column names, data types, and how the data is organized.

Schema-on-Write
Used in Data Warehouses. Data MUST match the schema before being stored. Like a form you must fill correctly before submitting.
Schema-on-Read
Used in Data Lakes. Store raw data first, apply schema only when reading. Like a messy inbox — organize when needed.
HDFS

🐘 Hadoop Distributed File System (HDFS)

HDFS is the distributed storage layer of Hadoop. It splits large files into blocks (128MB each) and stores multiple copies across nodes.

NameNode
Tracks metadata

⚠️ HDFS vs Block vs Partition: HDFS stores blocks (128MB physical chunks). Spark creates partitions (logical units for parallel processing) — they are NOT the same thing.

💾 What is Data?
Structured · Semi-structured · Unstructured · Data Types
Concept Snapshot

🔷 Data — One-liner

Data is raw facts and figures without context. When processed and organized, data becomes information. When information is used for decisions, it becomes knowledge.

🏦 Analogy: A bank transaction record — "1000, 2024-01-01, ACC123" is raw data. When labeled as "₹1000 withdrawal on Jan 1 from Account 123," it becomes useful information.
Types of Data

📂 3 Types of Data

✅ Structured

Organized in rows & columns. Has a fixed schema. Easily queryable with SQL.

Examples: Bank transactions, Customer table, Excel sheets

⚠️ Semi-Structured

Has some organization but not fixed rows/columns. Self-describing.

Examples: JSON, XML, CSV, Parquet, Avro

❌ Unstructured

No fixed format. Cannot be queried directly with SQL.

Examples: PDFs, Images, Videos, Audio, Emails, Logs

💡 Why Parquet over CSV? Parquet stores data column-by-column. When you query only 2 columns out of 100, Parquet reads only those 2 columns from disk — CSV reads everything. This is called columnar storage and is critical for performance in analytics.

🎤 Interview Insights — Data

Q: What's the difference between structured, semi-structured, and unstructured data?
Structured data has a fixed schema (rows & columns) like a SQL table. Semi-structured like JSON has flexible keys but some organization. Unstructured like images or PDFs has no predefined format. In banking, customer account info is structured, API logs are semi-structured, and scanned cheque images are unstructured.
Q: Why would you choose Parquet over CSV for a data pipeline?
Parquet uses columnar storage, which means for analytical queries touching only a few columns, it reads far less data from disk. It also supports efficient compression (Snappy, Gzip). In a fraud detection pipeline with 100+ columns, Parquet can reduce I/O by 90% compared to CSV.
🚨 Mistake: Saying "Parquet is just a compressed CSV" — it's fundamentally different in storage layout.

Key Concepts

🔑 SQL Concepts Every DE Must Know

-- 1. Basic SELECT with Filter
SELECT customer_id, account_no, amount
FROM transactions
WHERE status = 'COMPLETED'
  AND amount > 50000
  AND txn_date >= '2024-01-01';

-- 2. GROUP BY with HAVING (Aggregation)
SELECT customer_id, SUM(amount) AS total_spent
FROM transactions
WHERE status = 'COMPLETED'
GROUP BY customer_id
HAVING SUM(amount) > 100000;

-- 3. JOIN (combining two tables)
SELECT c.customer_name, t.amount, t.txn_date
FROM transactions t
INNER JOIN customers c ON t.customer_id = c.customer_id
WHERE t.status = 'FRAUD';

-- 4. Window Function (rank customers by spend)
SELECT customer_id, amount,
  RANK() OVER (PARTITION BY region ORDER BY amount DESC) AS spend_rank
FROM transactions;

🎤 Interview Insights — SQL

Q: What is the difference between WHERE and HAVING?
WHERE filters rows BEFORE grouping. HAVING filters groups AFTER GROUP BY. Example: WHERE amount > 1000 filters individual transactions; HAVING SUM(amount) > 100000 filters customer groups whose total exceeds ₹1 lakh.
Q: Explain a Window Function with an example.
Window functions perform calculations across a set of rows related to the current row without collapsing them like GROUP BY does. RANK() OVER (PARTITION BY region ORDER BY sales DESC) ranks customers within each region by sales — each customer row is preserved.
🚨 Mistake: Confusing window functions with GROUP BY — GROUP BY reduces rows; window functions don't.
03
🗄️ Database
RDBMS · ACID Properties · Normalization · Indexes
Concept

🔷 What is a Database?

A Database is an organized collection of structured data stored electronically, managed by a DBMS (Database Management System). A Relational Database (RDBMS) organizes data in tables with relationships between them.

🏦 Analogy: A bank's core system stores customer accounts in a relational database — each customer row links to transaction rows via customer_id, just like a foreign key relationship.
ACID Properties

⚗️ ACID — The Pillars of Database Reliability

ACID properties ensure that database transactions are processed reliably. Critical in banking where partial transfers (e.g., money debited but not credited) must never happen.

A — Atomicity

ALL steps of a transaction succeed, or NONE do. If a fund transfer fails midway, the entire transaction is rolled back.

C — Consistency

The database moves from one valid state to another. Total balance before = total balance after a transfer.

I — Isolation

Concurrent transactions don't interfere. Two people transferring money simultaneously won't corrupt each other's data.

D — Durability

Once committed, data is permanently saved — even if the server crashes immediately after. Data is written to disk/log.

Normalization

📐 Normalization — Reducing Redundancy

Normalization is the process of organizing a database to reduce redundancy and improve integrity. Key normal forms:

FormRuleBanking Example
1NFNo repeating groups; each cell has one valueNo column "phone1, phone2" — use a separate phones table
2NF1NF + No partial dependenciesCustomer name should not be in transactions table (depends only on customer_id)
3NF2NF + No transitive dependenciesBranch city should not be in accounts table — put it in branches table
⚠️ Normalization vs Denormalization: OLTP databases are normalized (less redundancy, fast writes). OLAP/Data Warehouses are often denormalized (Star Schema) for fast read/analytical queries — we'll cover this in the Data Modelling chapter.
Indexes

⚡ Database Index

An index is a data structure that speeds up queries by creating a lookup pointer — like a book's index that tells you which page a topic is on, so you don't read every page.

-- Create an index on customer_id for faster transaction lookups
CREATE INDEX idx_customer_txn 
  ON transactions (customer_id);

-- Without index: Full table scan (slow for millions of rows)
-- With index: Direct lookup (O(log n) — much faster)
SELECT * FROM transactions WHERE customer_id = 'C12345';

⚠️ Trade-off: Indexes speed up reads but slow down writes (INSERT/UPDATE must update the index too). Use selectively on high-cardinality, frequently queried columns.

🎤 Interview Insights — Database

Q: Explain ACID properties with a banking example.
In a fund transfer: Atomicity ensures both debit and credit happen or neither does. Consistency ensures the total balance doesn't change. Isolation ensures another concurrent transfer doesn't see an intermediate state. Durability ensures the committed transfer survives a server restart.
Q: Why would you NOT normalize a data warehouse?
Data warehouses use denormalized Star Schemas because analytical queries join many tables. Normalization reduces redundancy but increases JOINs — in OLAP, fewer JOINs mean faster query performance. We denormalize intentionally to optimize for reads, not writes.
🚨 Mistake: Saying "normalization is always good" — it's optimal for OLTP but harmful for OLAP performance.

🏛️ Data Warehouse
Centralized, structured, SQL-optimized historical data store
Concept

🔷 What is a Data Warehouse?

A Data Warehouse (DWH) is a centralized repository that stores structured, cleaned, historical data from multiple sources — specifically designed for analytical queries and business intelligence.

🏦 Analogy: If OLTP is the bank's daily ledger, the Data Warehouse is the bank's annual report room — everything is organized, audited, and ready for deep analysis.
Architecture Layers

🏗️ Data Warehouse Architecture — 3 Layers

📥
Staging Layer
Raw data landing zone. No transformation. Temporary or persistent copy of source data.
⚙️
Integration Layer
Data is cleaned, transformed, standardized, and loaded into fact/dimension tables.
📊
Access Layer
Data Marts, Views, OLAP cubes — exposed to BI tools like Power BI, Tableau.
Limitations

⚠️ Why Data Warehouses Have Limitations

  • ❌ Only structured data: Can't store images, logs, JSON natively
  • ❌ Expensive to scale: Compute and storage are tightly coupled — you pay for both together even if you only need one
  • ❌ ETL complexity: Data must be cleaned BEFORE loading (Schema-on-Write) — slow to onboard new data sources
  • ❌ No ML support: Not designed for ML workloads or unstructured training data
💡 These limitations led to the invention of Data Lakes (flexible, cheap storage) and eventually Data Lakehouses (best of both worlds).
Banking Example

🏦 Real-World Example — Banking DWH

Scenario: A bank wants a monthly report on total transaction value per branch, per account type, filtered for completed transactions only.

-- Star Schema query in Azure Synapse Analytics
SELECT
    d.branch_name,
    d.account_type,
    SUM(f.amount) AS total_amount,
    COUNT(f.txn_id) AS txn_count
FROM fact_transactions f
JOIN dim_branch d ON f.branch_id = d.branch_id
JOIN dim_date dt ON f.date_id = dt.date_id
WHERE f.status = 'COMPLETED'
  AND dt.month = 1
  AND dt.year = 2024
GROUP BY d.branch_name, d.account_type
ORDER BY total_amount DESC;

🎤 Interview Insights — Data Warehouse

Q: What is the difference between a Database and a Data Warehouse?
A database (OLTP) is optimized for fast transactional reads/writes with current data. A data warehouse (OLAP) stores historical, aggregated data from multiple sources, optimized for analytical queries. A bank's core system is a database; the reporting system that shows last year's trends is a warehouse.
Q: What are the key layers in a Data Warehouse?
Staging layer (raw data landing), Integration/Core layer (cleaned, normalized data with Fact + Dimension tables), and Access/Presentation layer (Data Marts, OLAP cubes, views exposed to BI tools). Each layer improves data quality progressively.

Store everything raw — structured, semi-structured, unstructured
Concept

🔷 What is a Data Lake?

A Data Lake is a centralized repository that stores data in its native/raw format — any volume, any velocity, any variety — without imposing a schema upfront. It uses Schema-on-Read.

🏦 Analogy: If a Data Warehouse is an organized filing cabinet, a Data Lake is a large warehouse where you throw in everything — documents, photos, audio tapes — and organize them only when you need to find something.

Limitations

⚠️ Common Data Lake Problems

  • ❌ No ACID transactions: Concurrent writes can corrupt data
  • ❌ No Schema Enforcement: Anyone can write garbage data
  • ❌ No Data Catalog: Hard to discover what data exists
  • ❌ Ineffective Partitioning: Poor query performance without partitioning strategy
  • ❌ Too Many Small Files: Thousands of tiny Parquet files kill performance (the "small files problem")
  • ❌ Leads to Data Swamp when ungoverned

🎤 Interview Insights — Data Lake

Q: What is Schema-on-Read vs Schema-on-Write?
Schema-on-Write (used in Data Warehouses) enforces the schema when data is written — data must conform before being stored. Schema-on-Read (used in Data Lakes) stores raw data without validation; schema is applied only when querying. Lakes are more flexible but risk data quality issues.
Q: What are the limitations of a Data Lake?
No ACID transactions (concurrent writes corrupt data), no schema enforcement (garbage in, garbage out), no built-in data catalog (poor discoverability), and the small files problem (too many tiny files degrade Spark performance). These are the exact problems Delta Lake was built to solve.
🚨 Mistake: Saying "Data Lakes replace Data Warehouses" — they are complementary. Lakehouses combine both.

🦟 Data Swamp
A Data Lake gone wrong — ungoverned, unusable mess
Concept

🔷 What is a Data Swamp?

A Data Swamp is a Data Lake that has become ungoverned, undiscoverable, and unreliable — full of data no one understands or trusts. It's what happens when data is dumped without proper governance.

🏦 Analogy: Imagine a bank archive room where every document, scrap, and post-it is thrown in a pile with no labels or filing system. Even if the data you need is there, you can't find it or trust it.
Why It Happens

⚠️ How a Data Lake Becomes a Data Swamp

🚨 Warning Signs:
  • No data catalog (no one knows what exists)
  • No schema enforcement (mixed formats, broken files)
  • No data lineage (unknown data origin)
  • No access control (everyone writes anything)
  • No data quality checks (nulls, duplicates everywhere)
  • No retention policy (terabytes of stale data)
✅ Prevention:
  • Azure Purview / Unity Catalog for data catalog
  • Delta Lake for schema enforcement + ACID
  • Data Governance policies and ownership
  • RBAC (Role-Based Access Control)
  • Data quality frameworks (Great Expectations, dbt tests)
  • Medallion Architecture (Bronze→Silver→Gold)

Why Cost Saving?

💰 Why is Lakehouse Cheaper than Synapse?

In a traditional Data Warehouse (like Azure Synapse with dedicated pool):

  • Compute and storage are tightly coupled — when data grows, you pay for bigger compute too

In a Lakehouse (ADLS + Databricks):

  • Data grows → only cheap ADLS storage cost increases
  • Compute (Databricks clusters) can be scaled independently and shut down when idle
  • ADLS costs ~₹1.5 per GB/month vs Synapse dedicated pool at ~₹150+ per DWU/hour

🎤 Interview Insights — Lakehouse

Q: What problems does a Data Lakehouse solve that neither a DWH nor a Data Lake could alone?
A Data Lake lacks ACID transactions and schema enforcement — leading to unreliable data. A Data Warehouse can't handle unstructured data or ML workloads cost-effectively. A Lakehouse (ADLS + Delta Lake + Databricks) gives you ACID transactions on cheap object storage, supports all data types, enables ML and streaming, while keeping compute and storage independent for cost efficiency.
Q: Explain Medallion Architecture.
Medallion Architecture organizes data in three progressive quality layers: Bronze (raw ingestion, no transformation), Silver (cleaned, deduplicated, type-cast data), and Gold (aggregated, business-ready data consumed by BI and ML). Each layer improves data quality and is stored in Delta format on ADLS in Databricks.
🚨 Mistake: Saying Medallion is only for Data Lakes — it's the standard pattern for Lakehouse implementations in Databricks.

⚡ Delta Lake
Open-source storage framework — ACID on your Data Lake
Concept

🔷 What is Delta Lake?

Delta Lake is an open-source storage framework created by Databricks that brings ACID transactions, schema enforcement, time travel, and scalable metadata management to Data Lakes. It does NOT replace ADLS — it enhances it.

🏦 Analogy: ADLS is the bank vault (storage). Delta Lake is the ledger + audit trail system on top — tracking every change, enabling rollbacks, and ensuring consistency.
How It Works

🔧 Delta Lake Internals

Delta Lake works by storing data as Parquet files + a transaction log (_delta_log) folder:

📁 ADLS Container / Delta Table Directory
├── part-00001.snappy.parquet ← actual data
├── part-00002.snappy.parquet ← actual data
└── _delta_log/
    ├── 00000000000000000000.json ← txn log (version 0)
    ├── 00000000000000000001.json ← txn log (version 1)
    └── 00000000000000000010.checkpoint.parquet ← checkpoint

Each JSON file in _delta_log records exactly what changed — which files were added or removed. This enables time travel, ACID, and schema evolution.

Key Features

✅ Delta Lake Key Features

ACID Transactions

Multiple writers won't corrupt the table. Reads always see a consistent snapshot.

Time Travel

Query any previous version of the data. Restore accidentally deleted data.

Schema Enforcement

Rejects writes that don't match the schema — prevents garbage data from entering.

Schema Evolution

Safely add new columns without breaking existing queries — using mergeSchema.

Upserts (MERGE)

MERGE INTO for insert-or-update logic. Critical for SCD Type 2 patterns in banking.

Unified Batch + Streaming

Same Delta table works for both batch writes (ADF) and streaming writes (Event Hubs + Spark Streaming).

Code Examples

💻 Delta Lake in PySpark

# 1. Write data as Delta format
df.write.format("delta").mode("overwrite").save("abfss://container@account.dfs.core.windows.net/transactions")

# 2. Read Delta table
df = spark.read.format("delta").load("abfss://container@account.dfs.core.windows.net/transactions")

# 3. Time Travel — Query Version 3 of the table
df_v3 = spark.read.format("delta") \
    .option("versionAsOf", 3) \
    .load("abfss://container@account.dfs.core.windows.net/transactions")

# 4. MERGE (Upsert) — Update if exists, Insert if new
from delta.tables import DeltaTable

delta_table = DeltaTable.forPath(spark, "abfss://container@account.dfs.core.windows.net/transactions")

delta_table.alias("target").merge(
    source_df.alias("source"),
    "target.txn_id = source.txn_id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()

# 5. Schema Evolution — add new column safely
df_new.write.format("delta") \
    .mode("append") \
    .option("mergeSchema", "true") \
    .save("abfss://container@account.dfs.core.windows.net/transactions")

🎤 Interview Insights — Delta Lake

Q: How does Delta Lake achieve ACID transactions?
Delta Lake maintains a _delta_log directory containing JSON commit files. Each write operation appends a new JSON file atomically — listing which Parquet files were added or removed. Readers always see a consistent snapshot by reading the latest committed log entry. Conflicting concurrent writes are detected and rejected through optimistic concurrency control.
Q: What is Time Travel in Delta Lake and when would you use it in banking?
Time Travel lets you query any historical version of a Delta table using VERSION AS OF or TIMESTAMP AS OF. In banking, if a data pipeline accidentally overwrites a customer balance table, I can use time travel to query the previous version and restore it — without maintaining separate backup files. It's essentially built-in point-in-time recovery.
🚨 Mistake: Saying Delta Lake stores separate copies of all historical data — it only stores changed files, not full copies. Time travel reads the log to reconstruct the state.

🧠 Quiz

A Spark job writes to a Delta table. Simultaneously, another job reads the same table. What happens?

  • A. Both jobs fail due to a write conflict
  • B. The reader sees partial/corrupted data
  • C. The reader sees a consistent snapshot of the data before the write started (Snapshot Isolation)
  • D. The write job is blocked until the read completes

✅ C is correct. Delta Lake uses MVCC (Multi-Version Concurrency Control) — readers always see a consistent snapshot. The writer creates a new version atomically without blocking readers.

📚 Data Catalog
Metadata inventory — discover, understand, and govern your data assets
Concept

🔷 What is a Data Catalog?

A Data Catalog is a metadata management system that acts as an inventory of all data assets in your organization — telling you what data exists, where it lives, who owns it, and who can access it. It does NOT store actual data — only information ABOUT the data.

🏦 Analogy: A Data Catalog is like a bank's file registry — it doesn't contain the actual customer files, but it tells you "Customer file #12345 is in Cabinet 3, Row B, owned by the KYC team, and only compliance staff can access it."
Unity Catalog

🏗️ Databricks Unity Catalog — How It Works

Unity Catalog Metastore (1 per Azure Region per Company)
  └── Catalog (e.g., banking_prod, banking_dev)
      └── Schema / Database (e.g., transactions, customers)
          └── Table / View (e.g., fact_transactions, dim_customer)
              (metadata only — actual data in ADLS as Delta files)

Unity Catalog knows WHERE each Delta table lives in ADLS, controls WHO can query it, tracks DATA LINEAGE (which pipeline created it), and applies ROW/COLUMN security — all as tiny metadata files (KBs, not GBs). All real data always lives in ADLS.

Capabilities

✅ What Unity Catalog Provides

  • Data Discovery: Search tables, columns, descriptions across all catalogs
  • Access Control: GRANT/REVOKE at catalog, schema, table, column, row level
  • Data Lineage: Visual graph of where data came from and how it flows
  • Schema Registry: Track column names, types, and changes over time
  • Tagging: Tag sensitive columns (PII, PCI) for compliance
  • Audit Logs: Who accessed what data and when

🎤 Interview Insights — Data Catalog

Q: Why does a Data Catalog matter in a large banking data platform?
In a large bank with hundreds of Delta tables in ADLS across multiple teams, without a catalog, engineers waste hours finding the right table or unknowingly create duplicates. Unity Catalog solves this — it's a single source of truth for what data exists, where it lives, who owns it, and who can access it. It also enforces column-level security for PII data like Aadhaar numbers without duplicating the underlying data.

Concept

🔷 What is Data Governance?

Data Governance is the set of policies, standards, processes, and roles that ensure data is accurate, consistent, secure, and used responsibly across the organization.

Data Management is the day-to-day operational implementation of governance policies — the tools and processes that enforce those rules.

🏦 Analogy: Governance is the RBI regulation that says "banks must mask customer PAN numbers in reports." Data Management is the team that implements that masking in Databricks using column-level security in Unity Catalog.

1. Data Quality

Accuracy, completeness, consistency, timeliness. No duplicate transactions, no null account numbers.

3. Data Lineage

Track where data came from, how it was transformed. Critical for regulatory audits.

4. Data Ownership

Every dataset has a named owner/steward responsible for its quality and documentation.

5. Data Discoverability

Engineers can find and understand available data assets via catalog. Reduces duplication.

6. Compliance

GDPR, RBI guidelines, PCI-DSS — governance policies ensure regulatory compliance.

📐 Data Modelling
Star Schema · Snowflake Schema · Fact vs Dimension · SCD Types
Concept

🔷 What is Data Modelling?

Data Modelling is the process of designing how data is structured, organized, and related for efficient storage, retrieval, and analysis. In Data Engineering, this primarily means Dimensional Modelling (Kimball methodology) for analytical systems.

🏦 Analogy: Like designing the layout of a library — which shelves (tables) to have, how to organize books (rows), what labels to use (columns), and how to link sections (foreign keys).
Fact vs Dimension

📊 Fact Tables vs Dimension Tables

📈 Fact Table

Contains measurable, quantitative business events — the "what happened." Has foreign keys to dimension tables and numerical measures.

Banking Example: fact_transactions (txn_id, amount, date_id, branch_id, customer_id, status)

📋 Dimension Table

Contains descriptive attributes for context — the "who, what, where, when." Usually smaller, denormalized.

Banking Example: dim_customer (customer_id, name, city, segment), dim_date (date_id, day, month, year, quarter)

Star Schema

⭐ Star Schema — Most Used in DWH

In a Star Schema, one central Fact table is surrounded by denormalized Dimension tables — forming a star shape. Simple, fast for queries, great for Power BI.

dim_date

Key: Dimensions are denormalized (all attributes in one table). Simple JOINs, fast for BI tools.

Snowflake Schema

❄️ Snowflake Schema

In a Snowflake Schema, dimension tables are normalized — split into multiple related tables. More complex, saves storage, but requires more JOINs.

🕐 Slowly Changing Dimensions (SCD)

SCD handles how dimension data changes over time. In banking, a customer's address or credit tier can change — SCDs manage this history.

-- SCD Type 2 MERGE using Delta Lake
delta_customers.alias("target").merge(
    new_customers.alias("source"),
    "target.customer_id = source.customer_id AND target.is_current = true"
).whenMatchedUpdate(
    condition="target.city != source.city",
    set={"is_current": "false", "valid_to": "current_date()"}
).whenNotMatchedInsert(
    values={"customer_id": "source.customer_id", "city": "source.city",
            "is_current": "true", "valid_from": "current_date()", "valid_to": "null"}
).execute()

🎤 Interview Insights — Data Modelling

Q: When would you choose a Snowflake Schema over a Star Schema?
Star Schema is preferred when query performance is critical because it requires fewer JOINs. Snowflake Schema is better when dimension tables are very large and storage cost is a concern, or when the data has natural hierarchies (e.g., Country → State → City). In practice, most BI tools like Power BI perform better with Star Schema.
Q: Explain SCD Type 2 with a banking use case.
SCD Type 2 preserves full history by adding new rows instead of updating. When a bank customer moves from Chennai to Mumbai, instead of overwriting the city, we close the old row (set valid_to = today, is_current = false) and insert a new row with Mumbai (valid_from = today, is_current = true). This lets us accurately report transaction totals per city for any historical date.
🚨 Mistake: Confusing SCD Type 1 (overwrite, no history) with Type 2 (new row, full history). In any compliance-heavy banking system, Type 2 is the default for dimensional data.

🔄 ETL & ELT
Extract → Transform → Load · Modern data pipeline patterns
Concept

🔷 ETL vs ELT — The Fundamental Difference

🏭 ETL (Extract → Transform → Load)

Transform data BEFORE loading it into the target. Data is cleaned/transformed in a staging area or ETL tool, then loaded into a Data Warehouse.

Used with: Azure Data Factory, SSIS → Azure Synapse

⚠️ Slower ingestion, but target stays clean

🚀 ELT (Extract → Load → Transform)

Load RAW data first into the lake, THEN transform inside it using Spark/SQL. Modern approach for big data.

Used with: ADF → ADLS → Databricks (Bronze→Silver→Gold)

✅ Faster ingestion, full raw history preserved

Banking Pipeline Example

🏦 ELT Pipeline — Fraud Detection (Azure + Databricks)

Architecture: Core Banking → ADF → ADLS Bronze → Databricks Silver → Gold → Power BI

# Bronze Layer: Load raw transactions from ADLS (no transformation)
bronze_df = spark.read.format("csv") \
    .option("header", "true") \
    .option("inferSchema", "true") \
    .load("abfss://raw@bankdatalake.dfs.core.windows.net/transactions/*.csv")

bronze_df.write.format("delta").mode("append") \
    .save("abfss://bronze@bankdatalake.dfs.core.windows.net/transactions")

# Silver Layer: Clean, deduplicate, cast types
silver_df = bronze_df \
    .dropDuplicates(["txn_id"]) \
    .filter(col("amount").isNotNull()) \
    .withColumn("txn_date", to_date(col("txn_date"), "yyyy-MM-dd")) \
    .withColumn("amount", col("amount").cast("double"))

silver_df.write.format("delta").mode("overwrite") \
    .option("overwriteSchema", "true") \
    .save("abfss://silver@bankdatalake.dfs.core.windows.net/transactions")

# Gold Layer: Aggregate for fraud detection KPIs
gold_df = silver_df \
    .filter(col("status") == "COMPLETED") \
    .groupBy("customer_id", "txn_date") \
    .agg(
        sum("amount").alias("daily_total"),
        count("txn_id").alias("txn_count")
    ) \
    .filter(col("daily_total") > 500000)  # flag high-value days

gold_df.write.format("delta").mode("overwrite") \
    .save("abfss://gold@bankdatalake.dfs.core.windows.net/fraud_flags")

🎤 Interview Insights — ETL/ELT

Q: What is the difference between ETL and ELT? Which would you use for a modern Lakehouse?
In ETL, data is transformed before loading — good for Data Warehouses that need clean data immediately. In ELT, raw data is loaded first into cheap storage (ADLS Bronze), then transformed using distributed compute (Databricks). ELT is preferred for modern Lakehouses because ADLS storage is cheap (no cost to store raw data), Spark handles massive transformations efficiently, and you preserve the original data for debugging or reprocessing.
Q: What is Incremental Load vs Full Load?
Full Load overwrites all data — safe but expensive and slow for large tables. Incremental Load only processes new or changed records since the last run — done via watermarks (last_modified > last_run_timestamp) or CDC (Change Data Capture). In banking, a transactions table with millions of daily rows always uses incremental load in production.
🚨 Mistake: Using Full Load for production pipelines with large tables — it's acceptable for small dimension tables but never for fact tables.

🐘 Big Data
5Vs · Hadoop · Spark · Why traditional systems fail
Concept

🔷 What is Big Data?

Big Data refers to datasets that are so large, fast, or complex that traditional data processing tools (like a single-server RDBMS) cannot handle them efficiently. It's defined by the 5 Vs.

🏦 Analogy: A single bank branch processes 1,000 transactions/day — that's normal data. A national bank processing 50 million transactions per hour, with real-time fraud detection across 500 data sources — that's Big Data.
5Vs

📊 The 5 Vs of Big Data

Volume

Size of data — TBs, PBs, ZBs. Traditional DBs can't store or process this much. Banking: 10 years of all transactions = petabytes.

Velocity

Speed of data generation. Real-time transaction streams at millions/second. Needs streaming (Kafka, Event Hubs).

Variety

Different data types — structured (SQL), semi-structured (JSON logs), unstructured (cheque images, call recordings).

Veracity

Trustworthiness/quality of data. Big data often has noise, errors, inconsistencies. Data governance solves this.

Value

Business value extracted from data. The goal of all data engineering — fraud detection, customer insights, risk models.

Why Traditional Systems Fail

⚠️ Why RDBMS / MapReduce Fail for Big Data

RDBMS Limitations:
  • Vertical scaling only (bigger server = more $$)
  • Can't store unstructured data natively
  • Slow for analytical queries on PBs of data
  • No streaming data support
MapReduce Limitations:
  • Every intermediate result written to disk (slow I/O)
  • Rigid 2-step workflow (Map + Reduce only)
  • Multiple jobs needed for complex operations
  • Not suitable for iterative ML algorithms
💡 Apache Spark solves MapReduce limitations by keeping intermediate results in-memory (RAM), using a DAG (Directed Acyclic Graph) for multi-step operations, and supporting both batch and streaming in one framework.

🎤 Interview Insights — Big Data

Q: What are the 5Vs of Big Data and which is most important?
Volume (size), Velocity (speed), Variety (types), Veracity (quality), and Value (business outcome). Value is the ultimate measure — the other 4Vs are challenges that need to be overcome to extract value. A bank storing PBs of transactions (volume) at millions/second (velocity) in JSON + images (variety) is only useful if the data is trustworthy (veracity) and used for fraud detection or risk models (value).
Q: Why is Apache Spark faster than Hadoop MapReduce?
Spark stores intermediate computation results in memory (RAM of worker nodes) instead of writing to disk after every step like MapReduce. Spark also uses a DAG (Directed Acyclic Graph) to optimize the entire execution plan at once, while MapReduce executes one Map+Reduce pair per job — requiring separate disk reads/writes between pipeline steps. Spark is 10-100x faster for multi-step data transformations.

☁️ Object Storage

Concept

🔷 What is Object Storage?

Object Storage stores data as objects (not files or blocks) — each object contains the data itself + metadata + a unique identifier. It's the backbone of cloud data lakes. Unlike file systems, there's no folder hierarchy — only flat namespaces with path-like keys.

🏦 Analogy: Object storage is like a warehouse where each item (object) has a unique barcode (key), can be any size, and comes with a tag (metadata) describing it. You retrieve items by barcode, not by searching shelves.

Key Concept

💡 Block vs Partition — Important Distinction

📦 Block (HDFS)
  • Physical storage unit (128MB)
  • Exists before Spark runs
  • Managed by HDFS NameNode
  • Purpose: How data is stored
🔀 Partition (Spark)
  • Logical unit of parallelism
  • Created when Spark reads data
  • Subset of a DataFrame
  • Purpose: How data is processed
  • 1 partition = 1 task = 1 core
💡 Key Rule: spark.sql.files.maxPartitionBytes (default 128MB) controls partition size. spark.sql.shuffle.partitions (default 200) controls shuffle partitions after groupBy/join. Tune these for performance.

🎤 Interview Insights — Object Storage

Q: How does Spark read data from ADLS differently from HDFS?
In HDFS, Spark executors are co-located on data nodes — data is read locally from disk (data locality). In ADLS, Spark executors make remote HTTP byte-range requests to Azure's object storage API (via ABFS driver). Azure internally handles replication and distribution. This means ADLS reads are slightly slower per executor than HDFS local reads, but the unlimited scalability and managed durability (LRS/ZRS/GRS) outweigh this cost. Spark reads only the required byte ranges, not full files — critical for performance with Parquet columnar files.

🧠 Final Quiz

Your bank's data engineering team stores 500TB of transaction data in ADLS Gen2 in Parquet format. Analysts complain that queries are slow. You find there are 50 million small Parquet files (each ~10KB). What is the PRIMARY cause and fix?

  • A. Parquet is the wrong format — switch to CSV
  • B. ADLS is not suitable for analytics — migrate to HDFS
  • C. Small files problem — each tiny file creates a Spark task with high overhead. Fix by running OPTIMIZE (Delta Lake) to compact small files into larger ones.
  • D. Increase Databricks cluster size

✅ C is correct. 50 million tasks for 50 million files overwhelms the Spark driver and creates massive scheduling overhead. Delta Lake's OPTIMIZE command compacts small files into ~128MB Parquet files. Also use ZORDER for co-locating related data (e.g., ZORDER BY customer_id). This is called the "Small Files Problem" — a classic big data anti-pattern.
